🔷 WorkItem kya hai?
👉 WorkItem = Approval task jo kisi user (manager, owner) ko diya jata hai.
👉 WorkItem = workflow ka wo task jo user ke haath me diya jata hai 🔥
🧠 Simple samajh:-
Workflow me step aaya → ManagerApproval
        ↓
System ne manager ko ek task diya
        ↓
Ye task hi = WorkItem

🔥 Real Example:-
Employee ne role request kiya
   ↓
Manager ko approval dena hai
   ↓
Manager ke inbox me ek item aata hai
   ↓
Wo item = WorkItem

🧠 🔹 WorkItem me kya hota hai:-
✔️ id
✔️ workflowCase (kis workflow ka part hai)
✔️ comments (approve/reject note)
✔️ created date
✔️ completed date
✔️ owner (current assignee)
✔️ requester (kisne request kiya)
✔️ target (kispe action ho raha hai)
✔️ state (Pending / Finished)

🔥 🔹 Kya karta hai?
👉 Workflow me jab bhi human decision chahiye hota hai:-
Workflow
   ↓
Approval step aaya
   ↓
WorkItem create hua
   ↓
User ko assign (owner)
   ↓
User action (Approve / Reject)
   ↓
WorkItem complete
   ↓
Workflow resume

()manager ke approval ke baad workitem ke finished hote hi kya hota hai,koun pata lagata hai ki approve hua ya reject aur fir provissiningplan me add koun karta hai workitem ko?
-->step:1:-
Q1. WorkItem ke Finished hote hi kya hota hai?
Jaise hi manager UI par submit dabata hai, background me ye sequential operations trigger hote hain:-

spt_work_item Update: Jo temporary row database ki spt_work_item table me pending padi thi, 
uska status badal kar Finished ya Closed ho jata hai. Usme approver ka decision aur comments permanent save ho jate hain.

Workflow Awaken (Neend se jagna): Jo main Workflow execution thread abhi tak Suspended (Hold) state me database me lock tha 
(spt_workflow_case table me), use background process wapas active memory (Heap) me khinch kar laati hai aur unfreeze karti hai.

Step Context Evaluation: Engine workflow ke agle step par badhta hai, 
jo standard workflows me hamesha processApprovalDecisions hota hai.

--step:2:-
Q2. Koun pata lagata hai ki Approve hua ya Reject?
Yeh pata lagane ka kaam sailpoint.workflow.IdentityRequestLibrary class ke backend methods karte hain, 
aur ise workflow me action="processApprovalDecisions" step ke throw chalaya jata hai.

Manager ka actual decision ApprovalSet ke andar ke har ek ApprovalItem ke state field
(jaise State="Approved" ya State="Rejected") me band hota hai.

processApprovalDecisions action is ApprovalSet par loop chalata hai aur check karta hai ki manager ne kis item ke samne green 
tick (Approve) mara hai aur kiske samne red cross (Reject).

--step:3:-
Q3. ProvisioningPlan me add koun karta hai workitem ko?
Yahan par ek chhota sa technical correction hai bhai: WorkItem ko ProvisioningPlan me add nahi kiya jata, 
balki ProvisioningPlan pehle se hi bana hota hai,aur manager ke decision ke mutabik us plan me se chizen KAATI(Filter ki) jaati hain.

Aur ye kaam bhi processApprovalDecisions hi karta hai!

Iska complete backend modification mechanism samajho:-
Jab tumne cart submit kiya tha, tabhi ek Master ProvisioningPlan ban gaya tha (Maan lo: AD account + SAP account).

Agar manager ne SAP ko Approve kiya aur AD ko Reject kiya.

Toh processApprovalDecisions original ProvisioningPlan object ko memory me kholega.Woh dekhega ki AD request ke samne State="Rejected" laga hai.

toh Yeh step turant us ProvisioningPlan ke andar se AD waale AccountRequest block ko permanently delete (remove) kar dega.

Ab plan me sirf approved waala SAP account bacha.Is filtered plan ko lekar yeh aage badhega aur agle step compileProject aur provisioner ko de dega.

🔷 WorkItemConfig kya hai?
👉 WorkItemConfig = WorkItem ka UI + behavior control object
Approval screen ka design + rules + options
👉 Ye decide karta hai:-
Approver ko kya dikhega
Kaise approve/reject karega
Kaunse buttons honge
Comments mandatory hai ya nahi

🔷 Default (Non-Custom) WorkItemConfig me kya hota hai:-
👉 SailPoint already kuch default config deta hai (tum custom define nahi karte):-

🔹 1. Approval Buttons:-
Approve
Reject
👉 Ye config decide karta hai ki:-
Kaunse actions allowed hain

🔹 2. Comments Requirement:-
Reject → comment required
Approve → optional
👉 Config control karta hai:-
Comment mandatory hai ya nahi

🔹 3. WorkItem Type:-
Approval
ManualAction
👉 UI ka behavior change hota hai type ke hisaab se

🔹 4. Form / Fields:-
👉 Approval screen pe fields dikhti hain:-
Identity Name
Requested Role
Application
Comments box
👉 Ye sab config define karta hai

🔹 5. Auto Complete Behavior:-
👉 Config decide karta hai:-
Approve → next step pe jao
Reject → workflow stop ya rollback

🔹 6. Escalation Settings:-
👉 Agar approver action nahi leta:-
Reminder bhejo
Escalate karo
👉 Ye bhi config me hota hai

🔹 1️⃣ complete():-
Kya karta hai:-
Ye method WorkItem ko completed mark karta hai.
Matlab workflow engine ko signal milta hai ki ye step successfully finish ho gaya.
Completion ka matlab hota hai ki approver ne action le liya aur WorkItem ka status ab Completed ho gaya.

Code Example:-
// WorkItem initial status pending
System.out.println("Before: " + workItem.getStatus());

// Complete the work item
workItem.complete();

// After completion
System.out.println("After: " + workItem.getStatus());

🟢 Output:-
Before: Pending
After: Completed

Kaun complete karta hai:-
Original approver ya assigned user jo WorkItem ke liye responsible hai.
Workflow engine me jab user(manager/owner) approve button click karta hai, tab internally complete() call hota hai.

------------------------------------------------------
🔹 2️⃣ reject():-
Kya karta hai:-
Ye method WorkItem ko rejected mark karta hai.

Matlab approver ne request deny kar di aur workflow ko signal milta hai ki ye step fail hua.

Code Example:-
// WorkItem initial status pending
System.out.println("Before: " + workItem.getStatus());

// Reject the work item
workItem.reject();

// After rejection
System.out.println("After: " + workItem.getStatus());

🟢 Output:-
Before: Pending
After: Rejected

Kaun reject karta hai:-
Assigned approver / reviewer — usually manager ya workflow ke liye designated user.
Jab approver reject button click karta hai ya workflow me reject action trigger hota hai, internally ye reject() call hota hai.
-------------------------------------------------------------------------------------------
🔹 3️⃣ validate():-
Kya karta hai:-
WorkItem ke mandatory fields aur rules check karta hai.
Agar required data missing ya invalid ho, exception throw kar sakta hai.
Validation workflow engine ko ensure karta hai ki WorkItem properly configured hai aur execute ho sakta hai.

Fields jo validate karta hai (common):-
Field	               Check
Id	              WorkItem ID present hai?
Status	          Valid status hai (Pending / Completed / Rejected)
Target	          Target object exist karta hai (Identity / Role / Plan)
EscalationLevel	  Valid integer, >=0
EscalationOwner	  Valid Identity object hai?
Reminders	      Valid dates

Code Example:-
try {
    workItem.validate();
    System.out.println("WorkItem validated successfully");
} catch (Exception e) {
    System.out.println("Validation failed: " + e.getMessage());
}

🟢 Output (if valid):-
WorkItem validated successfully.

🟢 Output (if invalid, e.g., target missing):-
Validation failed: Target object is missing.
