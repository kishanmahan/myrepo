1.approvalSplitPoint variable ka Asli Kaam Kya Hai?
Maano ki aapne ek saath 3 applications ke liye access request daali: Active Directory (AD), SAP, aur Salesforce.
Ab in teeno applications ke approvals alag-alag logon ke paas jaane hain (jaise AD ka approval IT Team degi, SAP ka Finance Manager dega, aur Salesforce ka Sales Head dega).
approvalSplitPoint isi jagah par kaam aata hai:-
-->Plan ko Split Karna (Todna): Yeh chal rahe main Provisioning Plan ko dekhta hai aur use applications ya approvers ke hisab se alag-alag tukdon (sub-plans) mein tod deta hai.
-->Parallel Approvals Bhejna: Yeh teeno approvals ko ek saath (parallelly) unke respective managers ke paas bhej deta hai taaki kisi ek ke chakkar mein poora kaam na ruke.
-->Result ko Merge Karna: Jab teeno managers apna-apna approval de dete hain, toh approvalSplitPoint un saare approved tukdon ko wapas ek sath merge (jod) kar deta hai aur unhe aage final provisioning ke liye bhejta hai.

Plan Kaun Banata Hai? Workflow ka starting step (jaise Compile Project ya provisioning subprocess).
approvalSplitPoint Kya Karta Hai? Bane-banaye plan ko pakadta hai, use approval ke liye alag-alag hisson mein split (divide) karta hai, 
aur approvals milne ke baad wapas combine (merge) kar deta hai.

2.value koun deta hai approvalSplitPoint ko?
-->SailPoint ki Global Configuration (LCM Configuration) se
SailPoint ke UI mein ek jagah hoti hai jise LCM Configuration (Lifecycle Manager Configuration) kehte hain.
Wahan system administrator select karta hai ki approvals ka level kya hona chahiye:-
Approve by Application: Har application ke liye alag approval jaye.
Approve by Role: Har role ke liye alag approval jaye.
Jo bhi option UI mein select kiya jata hai, 
SailPoint background mein approvalSplitPoint variable ko wahi value (jaise "Application" ya "Role") de deta hai.
Bhai, aapne bilkul sahi pakda hai! approvalSplitPoint ke andar jo value hoti hai, 
woh asal mein ek list (ya scheme) hi hoti hai jo SailPoint ko batati hai ki pure plan ko kis basis par alag-alag tukdon mein todna hai.

()plan ka hissa hai approvalset?
Aapne jo kaha: "approvalSet toh plan ka tukda hai" — iska concept ekdum sahi hai, 
kyunki jo cheez approve honi hai, woh aakhirkaar plan se hi aayi hai.
Aaiye ab isko technical tarike se samajhte hain ki approvalSplitPoint dono ko kaise todta hai:-
-->Asliyat mein kya hota hai?
approvalSplitPoint sirf plan ko nahi todta, balki Plan aur ApprovalSet dono ko aamne-saamne rakh kar, 
ek saath(parallelly) unke jode (pairs) banakar todta hai.

()ProvisioningPlan (The "What" - Kya kaam hona hai): Identity: Amit ──> AccountRequest (Application: AD) ──> 
->AttributeRequest (Operation: Add, Name: memberOf, Value: Finance-Group).
(Yeh ekdum raw technical instruction hai engine ke liye).

()ApprovalSet (The "Who" - Kiski permission chahiye): ApprovalItem (Application: AD, Instance: Finance-Group, 
Approver: Rahul Manager, Status: Pending).
(Yeh sirf approvals ko track karne ke liye hai).

()1. Splitting se pehle (Bada Project):-
Bada ProvisioningPlan: [AD Request + SAP Request]
Bada ApprovalSet: [AD ApprovalItem (Manager) + SAP ApprovalItem (App Owner)]

2. approvalSplitPoint ka Action (The Breakdown):-
Yeh engine donon ko aapas mein match karta hai aur unke tukde kar deta hai:-
Tukda 1 (For AD): Isme se AD ka ProvisioningPlan aur AD ka ApprovalItem alag ho jata hai. 
Yeh dono milkar ek chota subprocess banate hain aur AD ke manager ke paas chale jaate hain.

Tukda 2 (For SAP): Isme se SAP ka ProvisioningPlan aur SAP ka ApprovalItem alag ho jata hai. 
Yeh dono milkar doosre subprocess mein SAP ke owner ke paas chale jaate hain.

flow:-
[User Request / Lifecycle Event] 
       │
       ▼
 1. Provisioning Plan (Pehle banta hai)
       │  (Sirf kacha data: "Amit ko Active Directory me Finance Group de do")
       ▼
 2. Compile Project (Workflow Step)
       │  (Is raw plan ko system compile karta hai)
       ▼
 3. Provisioning Project (Baad me banta hai)
       │  (Isme Plan + ApprovalSet dono taiyar hokar aate hain)
       ▼
 4. approvalSplitPoint (Plan/Project ko todta hai)

()manager ke approval ke baad workitem ke finished hote hi kya hota hai,koun pata lagata hai ki approve hua ya reject aur fir provissiningplan me add koun karta hai workitem ko?
-->step:1:-
Q1. WorkItem ke Finished hote hi kya hota hai?
Jaise hi manager UI par submit dabata hai, background me ye sequential operations trigger hote hain:-

spt_work_item Update: Jo temporary row database ki spt_work_item table me pending padi thi, 
uska status badal kar Finished ya Closed ho jata hai. Usme approver ka decision aur comments permanent save ho jate hain.

Workflow Awaken (Neend se jagna): Jo main Workflow execution thread abhi tak Suspended (Hold) state me database me lock tha 
(spt_workflow_case table me), use background process wapas active memory (Heap) me khinch kar laati hai aur unfreeze karti hai.

Step Context Evaluation: Engine workflow ke agle step par badhta hai, 
jo standard workflows me hamesha processApprovalDecisions hota hai.

--step:2:-
Q2. Koun pata lagata hai ki Approve hua ya Reject?
Yeh pata lagane ka kaam sailpoint.workflow.IdentityRequestLibrary class ke backend methods karte hain, 
aur ise workflow me action="processApprovalDecisions" step ke throw chalaya jata hai.

Manager ka actual decision ApprovalSet ke andar ke har ek ApprovalItem ke state field
(jaise State="Approved" ya State="Rejected") me band hota hai.

processApprovalDecisions action is ApprovalSet par loop chalata hai aur check karta hai ki manager ne kis item ke samne green 
tick (Approve) mara hai aur kiske samne red cross (Reject).

--step:3:-
Q3. ProvisioningPlan me add koun karta hai workitem ko?
Yahan par ek chhota sa technical correction hai bhai: WorkItem ko ProvisioningPlan me add nahi kiya jata, 
balki ProvisioningPlan pehle se hi bana hota hai,aur manager ke decision ke mutabik us plan me se chizen KAATI(Filter ki) jaati hain.

Aur ye kaam bhi processApprovalDecisions hi karta hai!

Iska complete backend modification mechanism samajho:-
Jab tumne cart submit kiya tha, tabhi ek Master ProvisioningPlan ban gaya tha (Maan lo: AD account + SAP account).

Agar manager ne SAP ko Approve kiya aur AD ko Reject kiya.

Toh processApprovalDecisions original ProvisioningPlan object ko memory me kholega.Woh dekhega ki AD request ke samne State="Rejected" laga hai.

toh Yeh step turant us ProvisioningPlan ke andar se AD waale AccountRequest block ko permanently delete (remove) kar dega.

Ab plan me sirf approved waala SAP account bacha.Is filtered plan ko lekar yeh aage badhega aur agle step compileProject aur provisioner ko de dega.
