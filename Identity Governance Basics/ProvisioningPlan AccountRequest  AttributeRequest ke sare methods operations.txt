1. ProvisioningPlan.Operation.Add:-(Add)
Yeh operation kisi multi-valued attribute (jaise AD mein memberOf ya kisi application ke roles/entitlements) mein naya data add karne ke liye use hota hai.
Kaise kaam karta hai: Yeh target system par user ke pehle se exist karne wale data ko touch nahi karta. 
Jo value aap bhej rahe hain, use purani list mein append (add) kar deta hai.
Example: Maan lo ek user ke paas AD mein pehle se do groups hain: Finance aur HR.
Agar aap Operation.Add ke saath "Vpn_Group" bhejenge, toh execution ke baad user ke paas teen groups ho jayenge: Finance, HR, aur Vpn_Group.
Use Case: Jab aapko user ko koi naya access, group, ya role assign karna ho bina uske purane access ko cheede.
2. .Operation.Add Kab Use Karte Hain? (Multi-Valued Attributes)
Jab kisi attribute ke andar ek se zyada values (List/Array) store ho sakti hain, 
toh wahan Add use hota hai. Add ka matlab hota hai: "Purani values ko mat chhedo, 
unke sath is nayi value ko bhi list mein jod do."
AD Examples: memberOf (Groups), proxyAddresses (Email aliases), description (kabhi-kabhi multi-valued hota hai).
Kyun? Ek user ek hi waqt par 10 alag-alag AD Groups ka member ho sakta hai. 
Agar aap user ko VPN_Group mein daalna chahte ho, 
toh aap .Operation.Add chalate ho taaki uske purane groups delete na hon aur naya group list mein add ho jaye.

2. ProvisioningPlan.Operation.set:-
Yeh operation kisi attribute ki value ko overwrite (re-write) karne ke liye use hota hai.
Kaise kaam karta hai: Yeh target system ko bolta hai, "User ke paas iss attribute ke liye pehle se jo kuch bhi hai use hatao, 
aur abhi jo main de raha hoon sirf wahi set karo."Single-valued attributes par: Jaise email, firstname, ya department. 
Yahan set use karna ekdum safe aur standard hai kyunki user ka ek hi email ya department ho sakta hai.
Example: Agar user ke paas Finance aur HR groups hain, aur aapne Operation.set ke saath "Vpn_Group" bheja, 
toh target system se Finance aur HR gayab ho jayenge, aur user ke paas sirf Vpn_Group reh jayega.
1. .Operation.Set Kab Use Karte Hain? (Single-Valued Attributes)
Jab kisi attribute ke andar sirf ek hi value reh sakti hai, 
toh wahan hamesha Set use hota hai. Set ka matlab hota hai: "Pehle se jo bhi value hai use saaf karo, 
aur meri naye wali value wahan daal do (Overwrite)."
AD Examples: givenName (firstName), sn (lastName), mail (email), telephoneNumber, title, department.
Kyun? Ek insaan ka legal first name ya email ek hi ho sakta hai. 
Agar aap AD mein pehle se majood firstName="Rahul" par .Operation.Add chala kar "Amit" bhejoge, 
toh AD gussa ho jayega kyunki ek hi bande ke do first name nahi ho sakte.

1. Revoke (Access Hata Dena):-
Jab aap kisi access (role, group, ya entitlement) ko Revoke karte hain, 
toh iska matlab hai ki aap us user se woh access wapas le rahe hain.
Kaise kaam karta hai: SailPoint background mein ek ProvisioningPlan generate karega jisme Operation.
Remove use hoga. Yeh plan target system (jaise Active Directory) ko command bhejega ki user ko us specific group se nikaal diya jaye.
Example: Ek employee HR department se Finance department mein transfer ho gaya. Manager HR group ko Revoke kar dega taaki user ka purana access band ho jaye.

2. Retain (Access Rehne Dena):-
Jab aap kisi access ko Retain karte hain, toh iska matlab hai ki aap us access ko approve kar rahe hain taaki user ke paas woh access jaisa hai waisa hi rahe.
Kaise kaam karta hai: SailPoint target system par koi badlav (change) nahi karta. 
Provisioning engine ko koi plan nahi bheja jata kyunki access ko touch hi nahi karna hai. 
Bas SailPoint ke database mein audit log entry ho jaati hai ki yeh access certified/approved hai.
Example: Manager ne dekha ki user ke paas "Vpn_Group" hai aur user abhi bhi work from home kar raha hai, 
toh manager use Retain kar dega taaki uski VPN service chalti rahe.
Use Case: Regular access reviews (Manager/Application Certification) mein valid access ko verify aur approve karne ke liye.

import sailpoint.api.SailPointContext;
import sailpoint.object.AccountRequest;
import sailpoint.object.AttributeRequest;
import sailpoint.object.Identity;
import sailpoint.object.ProvisioningPlan;

import sailpoint.tools.GeneralException;

import java.util.List;

public class BeforeProvisioningExample {

    public void execute(ProvisioningPlan provisioningPlan, SailPointContext context) throws GeneralException {

        Identity identity = provisioningPlan.getIdentity();

        if (identity == null) {
            throw new GeneralException("Identity is null");
        }

        List accountRequests = provisioningPlan.getAccountRequests();

        if (accountRequests != null) {

            for (Object obj : accountRequests) {

                AccountRequest accReq = (AccountRequest) obj;

                // 🔹 Get attribute request
                AttributeRequest sam = accReq.getAttributeRequest("sAMAccountName");

                if (sam != null) {

                    Object value = sam.getValue();

                    if (value != null) {
                        String username = value.toString();

                        System.out.println("Username before provisioning: " + username);

                        // 🔥 Modify value
                        sam.setValue(username.toLowerCase());
                    }
                }

                // Add custom attribute
                AttributeRequest customAttr = new AttributeRequest();
                customAttr.setName("source");
                customAttr.setValue("BeforeProvisioning");
                customAttr.setOperation(ProvisioningPlan.Operation.Add);

                accReq.add(customAttr);
            }
        }
    }
}
