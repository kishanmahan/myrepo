()<Variable name="operations" input="true"></Variable>

(jaise Enable, Disable, Modify) perform karna hai.

()Case A: Identity Lifecycle Event se (Trigger):-
Maan lijiye aapne is workflow ko ek Termination Lifecycle Event ke sath map kiya hai. 
Jab SailPoint kisi user ka termination detect karega, toh SailPoint ka Lifecycle Engine background mein is workflow ko trigger karega.
Agar aapne Lifecycle Event setting mein koi static ya dynamic value pass ki hai, toh SailPoint use is variable mein fill kar dega.

()Case B: QuickLink ya LCM Form se:-
Maan lijiye aapne SailPoint UI mein ek custom form banaya hai jahan Manager dropdown se select karta hai: "Disable User".
Jab Manager form submit karega, toh SailPoint UI us selected value ko uthayega aur aapke workflow ke operations variable mein pass kar dega.

1. Kya yeh koi bhi value (Create, Modify, etc.) store kar sakta hai?
Haan, bilkul kar sakta hai. SailPoint mein workflow variables default roop se ek generic java.lang.String ya java.lang.Object ki tarah treat hote hain. Aap isme koi bhi text value store kar sakte hain.

Case A: Ek baar mein sirf EK value store karni ho (String format)
Agar aapka workflow ek baar mein sirf ek hi kaam karta hai (jaise sirf Disable ya sirf Enable), toh aap isme single text store karenge:

operations = "Create"

operations = "Disable"

Case B: Ek baar mein MULTIPLE values store karni hon (List format)
Agar aap chahte hain ki yeh ek se zyada values store kare (jaise ek sath Disable bhi kare aur Delete bhi), 
toh aapko is variable ka type java.util.List define karna padega.

XML mein yeh aisa dikhega:-
XML
<Variable name="operations" type="java.util.List" input="true" />
Ab Parent Workflow ya SailPoint isme ek sath multiple values bhej sakta hai: ["Modify", "Enable"].

()2. Aggregation Task (Data ka SailPoint mein aana):-
Yeh kahan hota hai? Jab aapka Account Aggregation Task chalta hai, 
toh SailPoint ka Connector target system se saara raw data khinch kar lata hai.
Iska data kahan rehta hai? Yeh data temporary roop se ResourceObject class ke andar store hota hai.

import sailpoint.object.Link;
import sailpoint.object.ResourceObject;
import sailpoint.object.Identity;
import sailpoint.tools.Util;

// Maan lijiye yeh dono objects aapko rule ya task ke context se mil rahe hain
// Link link = ... (SailPoint ke database mein save purana account data)
// ResourceObject rObj = ... (AD se aggregation ke dauran aaya naya data)

if (link != null && rObj != null) {

    // 1. LINK OBJECT (Old Data from SailPoint DB)
    // Link ke andar se purana attribute nikalna
    String oldDepartment = (String) link.getAttribute("department");
    String oldTitle = (String) link.getAttribute("title");

    // 2. RESOURCE OBJECT (New Data coming from Active Directory)
    // ResourceObject ke andar se naya incoming attribute nikalna
    String newDepartment = (String) rObj.getAttribute("department");
    String newTitle = (String) rObj.getAttribute("title");

    System.out.println("--- Comparing Link (Old) vs ResourceObject (New AD Data) ---");

    // -------------------------------------------------------------------------
    // Scenario 1: Department Compare Karna
    // -------------------------------------------------------------------------
    // Util.nullSafeEquality use karna best practice hai taaki null hone par crash na ho
    if (!Util.nullSafeEquality(oldDepartment, newDepartment)) {
        
        System.out.println("ALERT: Department badal gaya hai!");
        System.out.println("Old Data (Link DB): " + oldDepartment);
        System.out.println("New Data (ResourceObject AD): " + newDepartment);
        
        // Aap yahan apna custom logic likh sakte hain
    } else {
        System.out.println("Department mein koi badlav nahi hai.");
    }

    // -------------------------------------------------------------------------
    // Scenario 2: Job Title Compare Karna
    // -------------------------------------------------------------------------
    if (!Util.nullSafeEquality(oldTitle, newTitle)) {

()semicolon(;):-
kyunki SailPoint me semicolon (;) lagana hai ya nahi, 
yeh is baat par depend karta hai ki aap script ko XML ke kis attribute ya tag ke andar likh rahe hain.

-->Rule 1: Agar aap <Transition> tag ke andar hain (Semicolon NAHI lagana hai)
Jab aap kisi step se dusre step par jaane ke liye single-line condition likhte hain (jaise when="Script:..."), toh wahan semicolon nahi lagaya jata.
Kyun? Kyunki SailPoint wahan ek "Expression" expect karta hai jo turant true ya false (ya koi value) return kare. Semicolon lagane se BeanShell use ek statement maan leta hai aur kabhi-kabhi kuch return nahi karta (void ho jata hai), jisse workflow ruk jata hai.
❌ Galat: when="Script: "Approved".equals(item.getApprovalState());"
Sahi: when="Script: "Approved".equals(item.getApprovalState())"

Rule 2: Agar aap <Source> ya <Script> tag ke andar hain (Semicolon ZAROORI hai).

Rule 3:-
Agar aap <Approval> tag ke attributes jaise ownerExpression, argIdentity, ya when ke andar script: prefix ke sath single-line expression likh rahe hain, toh wahan semicolon nahi lagana hai.

Kyun? Kyunki SailPoint yahan direct ek dynamic value ya object return chahta hai. Semicolon lagane se logic break ho sakta hai.

❌ Galat:

<Approval name="Manager Approval" ownerExpression="script:getManagerName(identityName, launcher, fallbackApprover);" renderer="nativeChangeApprovalRenderer.xhtml">
