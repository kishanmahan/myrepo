-> FTP vs SCP ->
FTP :- Purana File transfer protocol
(Star) Features :-
-> Port :- 21
-> unsecure (data plain text me jata hai)
-> username/password bhi encrypt nahi hota.
-> separate control & data connection use karta hai.
-> Firewall issues aa sakta hai.

(Star) use :-
-> Public File Sharing
-> Old systems

SCP:- (Secure Copy Protocol)
() Feature :-
-> port : 22 (SSH)
-> Fully secure (encrypted)
-> simple command based transfer
-> Ek hi connection use karta hai.

()use :-
-> server to server transfer
-> secure environments (production)

()log4j kya hai in iiq?
--log4j ek logging framework/system hai jo application ke andar hone wale events,errors,aur activity ko record karta hai.

()log4j ke main kaam:-
->errors ko log karna(exception)
->debug information dena(developer ko)
->system actvity track karna
->troubleshooting me help karna

()log levels:-
--debug:-developer ke liye detailed info.
--info:-normal system activity
--warn:-warning
--error:-exception me
--FATAL:-system crash type issue

()sailpoint me log4j ka use:-
🔹 1. Log Levels in SailPoint (Log4j)
👉 1. DEBUG (sabse detailed)

👉 Developer ke liye deep internal info

log.debug("Entering provisioning rule for user: " + identity.getName());

✔ Use kab:

Rule debugging
Workflow trace
Variable values check

👉 2. INFO (normal activity)

👉 System kya kar raha hai (normal flow)

log.info("User created successfully: " + identity.getName());

✔ Use kab:

Successful operations
Important steps (aggregation start/end)

👉 3. WARN (warning)

👉 Kuch galat ho sakta hai, but system chal raha hai

log.warn("Manager not found for user: " + identity.getName());

✔ Use kab:

Missing data
Optional field null

👉 4. ERROR (serious problem)

👉 Operation fail ho gaya

log.error("Provisioning failed for user: " + identity.getName());

✔ Use kab:

Exception
Provisioning failure
DB connection issue

👉 5. FATAL (system crash type)

👉 Bahut critical error (rare use)

log.fatal("System is down! Unable to connect to database");

✔ Use kab:

Server crash
Critical system failure

--------------------------------------------------life cycle event---------------------------------------------------------------------

🔥 Case: Previous = false → New = true:-
👉 Matlab kya?
Pehle attribute false tha
Ab change hoke true ho gaya
✅ Example:
Previous Value Filter:
isManager == false

New Value Filter:
isManager == true
🎯 Meaning:

👉 User pehle manager nahi tha
👉 Ab manager ban gaya

🔥 Case: Previous = true → New = false:-
👉 Matlab kya?
Pehle attribute true tha
Ab change hoke false ho gaya
✅ Example:
Previous Value Filter:
isActive == true

New Value Filter:
isActive == false
🎯 Meaning:

👉 User pehle active tha
👉 Ab inactive ho gaya.

⚡ Real-life examples aur clear ho jayega:-
🔸 1. Employee join karta hai
Previous: employeeStatus == false
New: employeeStatus == true

👉 Joiner event trigger

🔸 2. Employee resign karta hai
Previous: employeeStatus == true
New: employeeStatus == false

👉 Leaver event trigger

🔸 3. Privilege milta hai
Previous: hasVPNAccess == false
New: hasVPNAccess == true

🔸 4. Privilege hata diya
Previous: hasVPNAccess == true
New: hasVPNAccess == false

🔷 1. Identity Trigger in Life Cycle Event (LCE):-
👉 Identity Trigger = condition jo decide karta hai ki event chale ya nahi.
Pehle ye check karega condition match hui ya nahi 
Fir hi workflow / provisioning chalega.

🔥 Dono ko ek sath samjho:-
👉 Identity Trigger + Attribute:-
<IdentityTrigger>
   <Filter>
      status == "Active"
   </Filter>
</IdentityTrigger>

👉 Yaha:-
status = attribute
condition = trigger

🔥 Real Scenario:-
🧑‍💼 Joiner Event:-
Previous: status == "Inactive"
New: status == "Active"
👉 Attribute = status
👉 Trigger = inactive → active change

🔄 Mover Event:-
Previous: department == "HR"
New: department == "IT"
👉 Attribute = department
👉 Trigger = HR → IT change

🚫 Leaver Event:-
Previous: isActive == true
New: isActive == false
👉 Attribute = isActive
👉 Trigger = true → false
------------------------------
🔷 🔥 Event Type (IIQ me real options ka matlab):-
Ye decide karta hai ki kis type ki change pe event trigger hoga.

🧑‍💼 1. Create:-
👉 Jab nayi identity create hoti hai
🎯 Use:-
New employee onboard
Account provisioning
👉 Basically = Joiner type case

🔄 2. Manager Transfer:-
👉 Jab user ka manager change hota hai
🎯 Example:-
Old Manager → New Manager
👉 Use:-
Approval flow update
Access change (manager-based roles)

🔁 3. Attribute Change:-
👉 Jab koi bhi identity attribute change hota hai
🎯 Example:-
department change
location change
title change
👉 Ye sabse common hai (Mover type cases)

⚙️ 4. Rule:-
👉 Jab tum custom logic use karna chahte ho
👉 Ye decide kar sakta hai:-
Event trigger ho ya nahi
Workflow me kya action lena hai
Provisioning me kya change karna hai
🎯 Use:-
Complex condition
Multiple attributes check
👉 Ye tab trigger hoga jab rule true return karega
example:-
🧑‍💼 Manager Promotion detect
<Rule name="ManagerPromotionRule" type="IdentityTrigger">
   <Source>
      import sailpoint.object.Identity;

      public boolean execute(Identity identity) {

         Boolean isManager = identity.getBooleanAttribute("isManager");

         if (isManager != null && isManager == true) {
            return true;
         }

         return false;
      }
   </Source>
</Rule>
👉 Jab user manager banega → event trigger

🔌 5. Native Change:-
👉 Jab application side pe change detect hota hai
🎯 Example:-
AD (Active Directory) me direct change hua
SAP ya DB me update hua
👉 IIQ detect karega → event trigger karega

🚨 6. Alert:-
👉 Jab koi alert ya violation aaye

()Event type: Ye define karta hai ki event kis trigger par chalega. Image mein Rule select kiya gaya hai, 
jiska matlab hai ki ek specific custom logic/rule decide karega ki kab ye event fire hona chahiye.

()Rule: Kyunki upar 'Rule' select kiya gaya hai, isliye yahan aapko wo specific SailPoint Rule select karna hoga jo logic check karega.

()Disabled: Agar aap is checkbox ko tick karte hain, to ye event inactive ho jayega aur background mein run nahi hoga.

()Included Identities: Isse aap filters set kar sakte hain ki ye event kin Identities (users) par apply hona chahiye (jaise 'All' ya koi specific population).

()Threshold Type: Ye ek safety feature hai. Isme aap choose karte hain ki limit Fixed (ek fix number) hogi ya Percentage.

()Threshold: Agar ye event ek sath bahut saari identities par trigger ho jaye (jaise termination process), 
to ye system ko rok deta hai agar count is number se upar chala jaye. Ye accidental mass-processing se bachata hai.

()Business Process: Ye sabse important part hai. Jab event trigger hota hai, 
to ye option batata hai ki kaunsa Workflow (jaise Joiner, Mover, ya Leaver workflow) execute hona chahiye.
Business Process option ek dropdown menu hai jo ye decide karta hai ki jab Lifecycle Event trigger hoga, tab kaunsa background workflow execute hona chahiye.
Iska basic concept ye hai:-
--Workflow Execution: Jab Event Type (jaise Rule ya Identity Attribute Change) ki condition meet ho jati hai, tab SailPoint isi Business Process ko call karta hai.
--Automation: Isme aksar standard workflows select kiye jate hain jaise Leaver, Joiner, ya Mover jo automatic provisioning, account disabling, ya email notifications handle karte hain.
--Mandatory Field: Jaise ki image mein red asterisk (∗) se dikhaya gaya hai, ye ek required field hai, isliye bina workflow select kiye event save nahi ho sakta.

--Lifecycle Event - Leaver: Ye standard workflow hai jo user ke resign karne ya terminate hone par uske access ko hatane (revocation) ke liye use hota hai.

--RapidSetup - Leaver: Agar aap SailPoint ka RapidSetup framework use kar rahe hain, toh is option ko select karein.

--Agar aapka maqsad kuch aur hai, toh baaki options ye kaam karte hain:

--Lifecycle Event - Joiner / RapidSetup - Joiner: Jab koi naya user system mein enter hota hai (Onboarding), tab ise select karte hain.

--RapidSetup - Mover: Jab kisi user ka department, location, ya manager change hota hai (Internal transfer), tab ise use kiya jata hai.

--Lifecycle Event - Manager Transfer: Ye specific workflow tab kaam aata hai jab sirf manager change ki reporting karni ho.

--Lifecycle Event - Reinstate: Agar koi purana user wapas join kar raha hai, toh ye workflow uski purani identity ko reactivate karne ke liye hota hai.

--Email manager / Manager Approval: Ye workflows tab select karein jab aap chahte hain ki event trigger hone par sirf email notification jaye ya manager se approval manga jaye.

()Process Variables ko agar ek line mein samjhna ho toh: "Yeh tumhare workflow (Business Process) ka temporary memory card 
ya storage dabba hote hain."
Jab ek workflow chal raha hota hai, toh usko apna kaam karne ke liye data yaad rakhne ki zaroorat padti hai. 
Yeh saara data Process Variables mein hi save hota hai.

--Initializer / Default Value: Agar start me variable me kuch daalna hai.
Example: Agar variable trace hai, toh uski default value true rakh sakte ho debugging ke liye.

()Business Process Editor ka "Process Variables" tab me jo 4 main variables dikh rahe hain(trigger, event, identityName, flow), 
wo LCE (Lifecycle Event) aur Workflow ke beech mein data transfer karne ke liye use hote hain.



1. trigger (String Variable)
Deep Detail: SailPoint mein ek hi workflow alag-alag events se start ho sakta hai.
trigger variable ek simple text (String) hota hai jo batata hai ki "Bhai, mujhe kisne jagaya?".
--Backend Values: Iske andar usually aisi values aati hain:
"Joiner" (Jab naya user create hota hai)
"Leaver" (Jab user terminate hota hai)
"Rehire" (Jab purana user wapas aata hai)
"AttributeChange" (Jab department ya title change hota hai)

2. event (Object: IdentityChangeEvent):-
Deep Detail:-
Yeh sabse complex aur sabse important variable hai. Yeh koi simple text nahi hai,
balki ek pura Java Object hota hai jiske andar bahot saari information bhari hoti hai.
Jab kisi target app (AD, HR) me koi native change hota hai, toh SailPoint ye object banata hai.
--Iske andar kya-kya milta hai? (Methods you can call):-
event.getApplication() 👉 Batata hai kis application pe change hua (e.g., "Active Directory").
event.getNativeIdentity() 👉 Batata hai account ka ID kya hai (e.g., "CN=John.Doe,OU=Users...").
event.getAttributeName() 👉 Batata hai kaunsa attribute change hua (e.g., "department" ya "phone").
event.getOldValue() aur event.getNewValue() 👉 Pehle department "IT" tha (Old), ab "HR" ho gaya (New). Yeh dono values isme milti hain.

4. flow: Kaam kis tareeqe se hoga? (Jaise: Background mein chupchap, ya Manager se approval maang kar).

5.source:-
​Variable Name: source
​Data Type: String (Text)
​Main Kaam: Workflow ke shuru hone ki jagah (Origin) ko track karna.
()Settings ka matlab:-
​Name: source — Is variable ka naam.
​Input (Checked): Iska matlab hai ki jab ye workflow shuru hoga, toh ye bahar se value le sakta hai.
​Output (Unchecked): Iska matlab hai ki workflow khatam hone par ye variable koi value bahar nahi bhejega.
​Editable (Unchecked): Iska matlab hai ki workflow ke beech mein ise manual tarike se change karne ki permission nahi di gayi hai (ye background mein process hota hai).
​Required (Unchecked): Iska matlab hai ki agar iski value nahi bhi di gayi, toh workflow rukega nahi (kyunki ye default value utha lega).
Yeh source kya karta hai?
​Asal mein, yeh ek identity mark ya tag ki tarah hai. Iska kaam sirf itna hai ki system ko bataye ki yeh request (jaise naye employee ka account banana) shuru kahan se hui.
​Audit Tracking: Kal ko agar check karna ho ki "Joiner Birthright AD" workflow kisne chalaya, toh yeh variable uska proof deta hai.
​Default Value (LCM): Agar aap ise aise hi chhod dete hain, toh SailPoint ise "LCM" (Lifecycle Manager) maanta hai. Iska matlab hai ki SailPoint ke normal process ne ise trigger kiya hai.
​Alag Sources: Agar koi request Batch file se, API se, ya kisi doosre software se aati hai, toh is variable ki value badal di jati hai taaki report mein saaf dikhe ki kaam kahan se shuru hua.

()Business Process Editor mein kisi variable (jaise identityName) par click karke use expand karte ho, toh uske andar 6 main attributes hote hain.
Yahi 6 attributes decide karte hain ki wo dabba (variable) kya aur kaise kaam karega:-

--name:-"Name" sirf us attribute/property ka label (heading) hai.

--Initializer (Default Value / Backup Plan):-
Yeh variable ki "Shuruwaati Halat" ya fallback option hai.
Asli Kaam: Agar workflow start hote waqt is variable ko bahar (kisi event) se koi data nahi milta,
toh system kya kare? Wo is Initializer ke andar likhi hui value ko apna lega.
Developer Context: Iska use default settings lagane ke liye hota hai taaki variable khali (null) na rahe 
aur aage ka logic smoothly chal sake.
Example: Maan lo ek variable hai approvalType. Agar kisi ne request karte waqt nahi bataya ki approval kisse lena hai, 
toh hum Initializer me default value <String>LineManager</String> set kar dete hain. 
Isse workflow atakne ke bajaye automatically manager ke paas chala jayega. Ek aur common example hai trace = true (debugging chalu rakhne ke liye).

--Type: Yeh batata hai ki is dabe (variable) me kaisa data aayega (sirf Text aayega, True/False aayega, ya koi bada Object aayega).

--Input (Tick Box): Agar aap is box ko tick karte ho (XML me input="true"), iska matlab hai ki yeh variable bahar se data receive karega.
Example: Lifecycle event identityName bhejega. Agar input tick nahi hai, toh workflow wo data receive nahi kar payega.

--Output (Tick Box): Agar workflow complete hone ke baad aapko koi result bahar (doosre process ya audit me) bhejna hai, toh ise tick karte hain.
Example: Koi custom approval ID jo aap chahte ho ki workflow ke bahar log ho.

--Required (Safety Lock / Zaroori):-
Asli Kaam: Yeh ek strict validation check hai. Agar yeh tick (✅) hai,
toh iska matlab hai ki is variable ke bina workflow ek kadam bhi aage nahi badh sakta.
Developer Context: Yeh system ko andha (blind) hone se rokata hai.
Maan lo "Leaver" workflow chal raha hai aur galti se HR system se user ka naam nahi aaya.
Agar identityName 'Required' tick kiya hua hai, toh workflow turant fail ho jayega aur bolega:-"Stop! Kiska account udana hai,
naam toh batao!". Agar required nahi hota, toh null error aa jata.
Example: Password reset workflow me newPassword variable hamesha Required hona chahiye.

()SailPoint IdentityIQ (IIQ) mein "Compile Project" ek workflow step ya action hota hai 
jiska kaam user ke dwara kiye gaye access request ko ek technical plan mein badalna hota hai.
​Saral bhasha mein kahein toh, jab koi user access ke liye request karta hai, 
toh IIQ seedhe use grant nahi karta. Pehle woh us request ko analyze karta hai 
aur check karta hai ki isse pura karne ke liye kaun-kaun se tasks zaroori hain.
​Yahan iske baare mein vistaar se bataya gaya hai:-
​
1. Compile Project ka Mukhya Kaam
​Is step ka kaam Provisioning Plan ko Provisioning Project mein convert karna hai.
​Provisioning Plan: Yeh ek "kacchi" list hoti hai, jisme likha hota hai ki kis user ko kaun sa access chahiye (jaise: AD account create karna ya SAP access dena).
​Provisioning Project: Yeh ek Detailed Blueprint hai. Isme system yeh tay karta hai ki:
​Kaun se connectors ka use hoga.
​Kaun se rules chalenge.
​Kya koi approval ki zaroori hai?
​Kya koi Policy Violation (SoD) toh nahi ho raha?
​
2. Aapke Screenshot mein kya ho raha hai?
​Aapne Business Process Editor khola hua hai, jahan ek workflow step configure kiya ja raha hai:
​Result Variable (project): Jab yeh step khatam hoga, toh jo final output (blueprint) banega, 
use project naam ke variable mein save kar diya jayega. Agle steps (jaise approvals ya actual provisioning) isi variable ka use karenge.
​Call Method (compileProvisioningProject): Yeh wahi internal function hai jo background mein saari calculation karta hai.
​3. Yeh kyun zaroori hai?
​Bina project compile kiye, IdentityIQ ko yeh samajh nahi aayega ki actual "execution" kaise karni hai.
Yeh step process mein transparency lata hai aur provisioner ko batata hai ki use exactly kya karna hai.

1. Compile Project ke Arguments:-
​Yeh arguments system ko "Action Plan" (Project) banane mein madad karte hain:
​plan: Yeh input hai. Isme user ki request hoti hai. Iska value Reference set kiya gaya hai,
matlab yeh workflow ke plan variable se data uthayega.
​optimisticProvisioning (false): Agar ise true kar dein, toh IIQ bina wait kiye provisioning start kar deta hai.
false ka matlab hai ki system standard checks aur process follow karega.
​source (UI): Yeh batata hai ki request kahan se aayi hai. "UI" ka matlab hai ki kisi user ne portal par jakar request submit ki hai.
​noApplicationTemplates (true): Iska matlab hai ki compilation ke waqt application-specific templates ka use nahi kiya jayega,
jo process ko fast banata hai agar templates ki zaroorat na ho.
​identityName: Yeh us user ki Identity ka naam hai jiske liye yeh process chal raha hai.
​
2. Provision Step ke Arguments:-
​Jab project compile ho jata hai, tab Provision step actual mein access grant karne ka kaam karta hai.Iske arguments ye hain:
​project: Yeh wahi "blueprint" hai jo Compile Project step ne taiyaar kiya tha.
Iska value Reference (project) rakha gaya hai taaki pichle step ka output use ho sake.
​background (true): Yeh bahut important hai. Iska matlab hai ki provisioning ka kaam server ke background mein chalega.
User ko screen par wait nahi karna padega; request submit ho jayegi aur IIQ peeche se apna kaam pura kar lega.

()Type Settings Checkboxes:-
1. Multi-Valued (Image mein Tick ✅ hai)
Kya karta hai: Yeh batata hai ki is dabe (field) mein ek se zyada values daali ja sakti hain.
Use Case: Jaise ek user ke paas 2 email IDs (Personal aur Work) ho sakti hain. Isko tick karne par backend mein data ek List (array) ki tarah save hoga, simple String ki tarah nahi.

2. Required (Zaroori)
Kya karta hai: Yeh validation lagata hai. Agar ye tick hai, toh form submit karte time ye field khali (empty) nahi chhod sakte.
Use Case: Jaise "Email" ya "Password" create karte waqt mandatory hona chahiye. Agar user bina bhare submit karega toh error aa jayega.

3. Refresh on Change (Bahot Important ⚡):-
Kya karta hai: Agar user form par is field ki value type karke change karta hai, toh page automatically ek chota sa "refresh" (reload) maarta hai.
Use Case: Maan lo ek field hai 'Department'. Agar user usme 'IT' select karta hai,
toh aap chahte ho ki 'Manager' wala field apne aap update ho jaye.
Toh 'Department' wale field par aap isko tick karoge, taaki value badalte hi background mein Rule chale aur dusre fields ko update kar de.

4. Review Required:-
Kya karta hai: Yeh flag laga deta hai ki is field ki value change hone par extra dhyaan dene ki zaroorat hai.
Use Case: Jab access request kisi approver ke paas jati hai, toh ye field highlight ho jata hai taaki approver 
ko pata chale ki is critical field me koi change hua hai.

6. Display Only (Sirf Dikhane Ke Liye):-
Kya karta hai: Agar yeh tick hai, toh is field ka data UI par dikhega zaroor, lekin jab aap "Submit" karoge, 
toh yeh data backend (Provisioning Plan) mein target system (LDAP) ko nahi bheja jayega.
Use Case: Form par user ko koi 'Warning Message' ya 'Current Status' dikhana ho jo save nahi karna hai.

7. Read Only (Image mein 'False' hai):-
Kya karta hai: Agar isko 'True' kar diya, toh field form par greyed-out (lock) ho jayega. 
User value padh toh payega, par usme click karke kuch type (edit) nahi kar payega.

8. Hidden (Image mein 'False' hai)
Kya karta hai: Agar isko 'True' kiya, toh yeh field screen par kisi ko dikhega hi nahi (gayab ho jayega).
Use Case: Jab aapko background me koi ID ya calculated value LDAP ko bhejni ho, 
par aap nahi chahte ki user usko form par dekhe ya chhede.

5.Authoritative field:-
--Asli Kaam (Technical Impact):-
Force Overwrite (Zabardasti Update):-Agar ye tick hai, toh user is form par jo bhi value type karega, 
SailPoint usko 100% final aur sahi maan lega. Wo target system (jaise LDAP ya Active Directory) mein jaakar purani value ko 
bina soche-samjhe overwrite (replace) kar dega. Target system me pehle se kya tha, is baat ko puri tarah ignore kar diya jayega.
--Conflict Resolution (Priority Dena):-Kai baar background me alag-alag Rules ya policies ek hi time par same attribute ko update karne ki
koshish karte hain. Aise me agar form ka ye field 'Authoritative' mark hai, toh backend ke baaki saare rules haar jayenge, 
aur is form ki value hi jeetegi. Provisioning Plan mein yahi value "Final Value" ban kar jayegi.

()Authoritative Application: Yeh pura ka pura ek application (jaise HR System - Workday ya SAP) hota hai jisko SailPoint apna "Maai-Baap"
maanta hai. Yahan se company ke naye accounts (Identities) SailPoint ke andar aate hain (Joiner process ke liye).

()Owner (None):-Yeh batata hai ki is specific field ka 'maalik' kaun hai.
Iska use advanced level par tab hota hai jab field-level approvals chahiye hote hain.
Zyadatar projects mein yeh 'None' hi choda jata hai.

()Value Settings (Niche wala hissa - The Brain of the Field):-
Yeh section sabse zyada important hai kyunki yeh tay karta hai ki field ke andar data aayega kahan se aur kaisa hoga:-

--Dynamic (Checkbox - Unticked):-Yeh option tab kaam aata hai jab form par multiple fields hon.
Agar yeh tick (✅) hai, toh jab bhi form kisi aur field ki wajah se refresh hoga (Refresh on Change), 
SailPoint is field ki value ko dobara background se calculate karega. Agar untick hai,
toh sirf ek baar shuru mein calculate hoke fix ho jayegi.

--Value (None):-Jab form pehli baar khulega, toh is dabe mein default kya likha aayega,
wo yahan se set hota hai. Dropdown se aap select kar sakte ho ki value static (String) hogi, ya kisi Rule/Script se auto-fetch hogi.

--Allowed Values (None):-Agar aap chahte ho ki user apni marzi se text type na kare, balki use ek Dropdown Menu (List) mile,
toh wo yahan banti hai.Aap backend se rule chala kar options la sakte ho(jaise LDAP se saare OUs ya Departments ki list nikal kar dikhana).

--Validation (None):-Jab user form "Submit" ya "Save" karega,toh yahan lagaya gaya Rule/Script check karega ki data sahi format mein hai 
ya nahi. Jaise validation rule check kar sakta hai ki Email address mein '@' laga hai ya nahi,ya password kam se kam 8 characters ka hai 
ya nahi. Agar galat hua, toh form save nahi hoga aur error de dega.

---------------------------------------------------------------------------Business Process (Workflow) Editor----------------------------
()wo SailPoint IdentityIQ ka Business Process (Workflow) Editor hai:-
1. Input (इनपुट):-
Deep Detail: Yeh option system ko batata hai ki jab yeh workflow start hoga, tab ise bahar se data lene ki zaroorat hai. 
Yeh data us user se aa sakta hai jisne process trigger kiya hai, ya phir kisi dusre parent workflow se.
--Real-world Example: Maan lijiye aapka workflow naye employee ka email account banane ke liye hai. 
Is workflow ko apna kaam shuru karne ke liye employee ka First Name aur Last Name chahiye hi hoga.
Toh ye dono values as an Input variable workflow ko pass ki jayengi.

2. Output (आउटपुट):-
Deep Detail: Yeh option tab check kiya jata hai jab workflow khatam hone ke baad koi final nateeja (result) wapas kisi 
aur process ko dena chahta ho. Agar aapka workflow ek "Sub-process" (chota process jo kisi bade process ke andar chalta hai) hai,
toh ye bahut important hota hai.
--Real-world Example: Maan lijiye ek chota workflow hai jiska kaam sirf ye check karna hai ki kya employee ka naam pehle se 
database mein exist karta hai ya nahi. Jab yeh workflow finish hoga, toh yeh ek Output dega: True(haan, karta hai) ya False(nahi karta).
Ye output phir main workflow use karega aage ka decision lene ke liye.

3. Editable (एडिटेबल):-
Deep Detail: Yeh option seedhe taur par WorkItems (jaise approval forms ya user tasks) se juda hota hai.
Agar workflow ke beech mein kisi human (jaise manager) ka intervention chahiye,
toh ye decide karta hai ki kya wo human is variable ki value ko badal sakta hai ya nahi.
--Real-world Example: Ek user ne '16GB RAM' request ki hai apne system ke liye, aur approval manager ke paas gaya.
Agar RAM wala variable Editable set hai, toh manager us 16GB ko change karke 8GB kar sakta hai aur phir approve kar sakta hai.
Agar ye unchecked (read-only) hota, toh manager value modify nahi kar pata, sirf approve ya reject kar pata.

4. Required (रिक्वायर्ड):-
Deep Detail: Yeh ek safety aur validation check hai. Agar aap isko check karte hain,
toh aap system ko bol rahe hain ki "Is variable mein data hona hi chahiye,
warna error throw kar do". Yeh workflow ko galat data ke saath aage badhne se rokata hai.
Real-world Example: Agar aap account create kar rahe hain aur Username variable ko aapne Required mark kiya hai.
Agar kisi bug ya user mistake ki wajah se Username blank reh gaya,toh workflow wahi ruk jayega aur account creation fail ho jayega,
bajaye iske ki system aage jake koi blank account bana de.

()koun sha variable pe upar ke charo option chalega?
--wo SailPoint IdentityIQ ka Business Process (Workflow) Editor name:-plan
Aap dhyan se dekhenge toh Name: label ke theek aage wale text box mein chote aksharon mein plan likha hua hai.

()initializer:-
aap process variable ke "Initial Value" (शुरुआती मूल्य) configuration ka section dekh rahe hain.
Jab aap chahte hain ki workflow start hote hi aapke variable mein pehle se hi koi default value assign ho jaye,
tab aap is section ka use karte hain.

()String: Agar aapko koi simple, fixed (static) text dalna hai (jaise ki status ke liye "Pending" ya "Approved" likhna).

()Reference: Agar aap chahte hain ki is variable ki value kisi dusre maujooda variable ya system object se utha li jaye (jaise identityName ya launcher).

()Script: Yahan aap apna custom chhota sa code (Beanshell script) likh sakte hain.
Workflow start hote waqt wo script run hogi aur jo bhi result niklega,
wo is variable mein save ho jayega.

()Rule: Agar logic lamba ya complex hai, toh aap SailPoint mein pehle se bana hua koi 'Rule' yahan attach kar sakte hain.
Wo rule execute hoga aur initial value return karega.

()Call Method: Kisi external ya existing Java class ke method ko call karke wahan se calculation karwa kar value lane ke liye iska use hota hai.

()Negate (Checkbox): Yeh option boolean values (True/False) ke sath kaam aata hai.
Agar script ya rule se value True aati hai aur aapne Negate par tick kiya hua hai, toh system use ulta karke False maan lega.

Jab aap flowchart (designer) mein kisi box par double-click karte hain, toh aisi window khulti hai.
Is window mein do main tabs hain: Details aur Arguments. Aaiye in dono ko detail mein samajhte hain:-

()1. Details Tab:-

()Description: Yahan aap detail mein likh sakte hain ki yeh step aakhir kya kaam kar raha hai.
Yeh sirf documentation ke liye hota hai taaki aage chal kar aapko ya kisi aur developer ko samajhne mein aasaani ho.

()Result Variable: Yeh bahut important hai.Jab yeh step apna execution poora kar lega,toh jo final output ya nateeja aayega,
wo system kisi variable mein save karega. Yahan (jaise "project" likha hai) aap us variable ka naam dete hain jisme result store hoga. 
Is variable ko aage ke steps mein use kiya ja sakta hai.aur ye project variable aapne process variable me define as local variable kiya hai.

()Enable Monitoring: Agar aap is checkbox ko tick karte hain, toh SailPoint is specific step ka data track karega

()Action Section: Yeh hissa define karta hai ki yeh step exactly "kaam kaise karega". Aapke paas 4 options hote hain,
jinme se aap koi ek chun sakte hain:-

()Script: Agar aap is step ke andar hi seedhe apna custom code (Beanshell) likhna chahte hain.

()Rule: Agar aapne pehle se hi SailPoint mein koi standard 'Rule' bana kar save kiya hua hai,
toh aap use yahan link kar sakte hain taaki wo run ho.

()Subprocess: Agar is step par aakar aap kisi dusre, chote workflow ko call (start) karna chahte hain.

()Call Method: Kisi specific backend Java class ke function (method) ko direct execute karne ke liye.

2. Arguments Tab (Iska kya kaam hai?):-
()Jab aap SailPoint mein koi Workflow banate hain (jaise ki user onboarding, access request, ya password reset), 
toh wo workflow kai chhote-chhote "Steps" se milkar banta hai.Har step ek specific kaam karta hai(jaise approval lena, email bhejna, ya koi condition check karna).
Aapke screenshot mein jo step open hai uska naam "Check for Maunal Process" hai.
Is step ko apna logic chalane ke liye kuch data (input) ki zarurat hai. Is input ko "Argument" kehte hain.
Ye options (String, Reference, Script, etc.)tay karte hain ki us step ke variable (Argument) mein data aayega kahan se.
Aaiye inhe detail mein dekhte hain:-

1. String (Static Value):-
Kaam: Jab aapko koi aisi value pass karni ho jo fix ho aur workflow chalne par kabhi change na ho.
Example: Maan lijiye aap is step ko batana chahte hain ki ye process "Pending" state mein hai.
Aap String select karenge aur box mein likhenge: Pending.
Use Case: Fixed configuration values, hardcoded statuses, ya email templates ke naam pass karne ke liye.

2. Reference (Workflow Variable):-
Kaam: Workflow ke andar pehle se bahut saare global variables hote hain (jaise identityName, launcher, plan).
Jab aapko unme se kisi variable ka data is step mein use karna ho, tab Reference use hota hai.
Example: Agar aapko is step mein us user ka naam chahiye jiske liye ye request aayi hai,
toh aap Reference select karenge aur box mein likhenge: identityName.
SailPoint runtime par automatically wahan "Rahul.Kumar" (ya jo bhi user hoga) put kar dega.
Use Case: Puraane steps se aane wale data ko current step mein pass karne ke liye.

3. Script (Inline BeanShell Code):-
Kaam: Jab aapko data direct nahi mil raha aur thodi si calculation ya logic lagana hai. Isme aap seedha BeanShell (Java jaisi) scripting likh sakte hain.
Example: Maan lijiye aapko variable mein aaj ki date pass karni hai. Aap Script select karenge aur code likhenge:
ex:-return new java.util.Date();
Ya phir do reference variables ko jodna ho:-
ex:-return firstname + "." + lastname;
Use Case:-Chote-mote data manipulation ke liye jiske liye alag se bada Rule banane ki zarurat nahi hai.

4. Rule (Pre-defined SailPoint Rule):-
--Kaam:-Jab logic bahut bada aur complex ho (jaise database se kuch fetch karna, ya alag-alag conditions check karna).
SailPoint mein aap alag se "Rules" (jaise `Workflow Rule`) banate hain. Yahan aap bas us Rule ka naam select kar lete hain.
--Example:-Agar aapko pata karna hai ki user ka Manager chhutti (vacation) par hai ya nahi,
toh aapne iske liye ek Rule banaya hoga `CheckManagerStatusRule`. Yahan `Rule` select karke aap bas wo naam de denge.
Wo Rule background mein chalega aur jo result aayega (True/False ya naam), wo is argument mein save ho jayega.
--Use Case:-Complex calculations aur reusable logic (jo kai workflows mein use hota ho) ko call karne ke liye.

5. Call Method:-
Kaam:-SailPoint backend Java par chalta hai. Agar aapne koi custom Java class likhi hai,
ya aap SailPoint ki inbuilt Java classes/methods ko direct call karna chahte hain (bina script ya rule likhe), toh iska use hota hai.
Example:-Class name mein `sailpoint.api.Util` aur method mein `isManualProcess()` call karna. 
Use Case:-Highly technical or custom Java code ko workflow mein integrate karne ke liye.

6. Negate (Checkbox):-
Kaam:-Ye upar diye gaye kisi bhi option ke result ko "ulta" (flip) kar deta hai.
Ye tabhi kaam aata hai jab aapka result Boolean (True ya False) mein aa raha ho.
Example:-Maan lijiye aapne `Reference` mein ek variable pass kiya `isApproved`, jiski value aayi `True`.
Agar aapne `Negate` par tick kiya hua hai, toh is step ke andar wo value `False` bankar jayegi.

()SailPoint IIQ ke workflow (Business Process Editor) mein jab ek step se dusre step par 
arrow(rasta) jata hai toh kis condition mein workflow ko aage badhna hai,ye hum Edit Transitions box mein define karte hain->
->Maan lo hum Step A par hain aur wahan se 2 raste nikal rahe hain:
​Rasta 1: Transition to Step B (Condition laga di)
​Rasta 2: Transition to Step C (Khaali chhod diya - Default)
​Ab dekhte hain har option ke saath flow kaise badlega:
​1. String (Text Match)
​Fulfill Hone Pe (True): Agar pichle step ka result wahi hai jo aapne "String" box mein likha hai, toh workflow turant Step B par chala jayega.
​Fulfill Na Hone Pe (False): SailPoint is raste ko cancel kar dega aur niche wale raste (Step C) ki taraf badh jayega.
​2. Reference (Variable Check)
​Fulfill Hone Pe (True): Agar aapka variable (jaise isApproved) memory mein true hai, toh workflow seedha Step B par jump kar jayega.
​Fulfill Na Hone Pe (False): Woh is arrow ko chhod dega aur agle available transition (Step C) ko check karega.
​3. Script (Custom Code)
​Fulfill Hone Pe (True): Agar aapki likhi hui script ne aakhiri line mein return true; diya, toh workflow Step B ki taraf mud jayega.
​Fulfill Na Hone Pe (False): Agar script ne return false; diya, toh woh rasta band ho jayega aur SailPoint agle raste (Step C) par chala jayega.
​4. Rule (Pre-defined Logic)
​Fulfill Hone Pe (True): Agar woh Rule execute hone ke baad True result deta hai, toh workflow Step B par jayega.
​Fulfill Na Hone Pe (False): Agar Rule ne False diya, toh woh is arrow ko ignore karke next transition (Step C) par check karega.
​5. Call Method (Java Function)
​Fulfill Hone Pe (True): Agar aapka Java method backend se true bhejta hai, toh workflow Step B par move kar jayega.
​Fulfill Na Hone Pe (False): Agar method ne false return kiya, toh workflow is raste ko chhod kar niche wale raste (Step C) ki taraf badhega.
​6. Negate (Result Ulta Karna)
​Yeh option thoda alag hai kyunki yeh upar ke 5 options ke saath milkar kaam karta hai.
​Kaise Kaam karta hai: Maan lo aapne Reference mein isApproved daala aur Negate ko tick kar diya.
​Fulfill (Logic Ulta): Agar isApproved ki value False hai, toh Negate usko True bana dega aur workflow Step B par chala jayega.
​Not Fulfill: Agar isApproved ki value True hai, toh Negate usko False bana dega aur workflow agle raste (Step C) par chala jayega.
-------------
​1. Agar Condition Fulfill (True) Ho Jaye:-
​SailPoint hamesha transitions ko upar se niche (sequential order mein) check karna shuru karta hai.
​Maan lo aapne pehle wale box (Transition to: Account Approval) mein ek Script ya Rule laga diya.
​Agar wo shart puri ho gayi aur result True (Fulfill) aaya, toh workflow turant us teer (arrow) ko pakad kar Account Approval step par chala jayega.
​Jaise hi use ek True condition mil jayegi, wo baki bache hue transitions ko check hi nahi karega.
​2. Agar Condition Fulfill NA (False) Ho:
​Agar pehle raste ki shart match nahi hui aur result False (Not fulfill) aaya, toh SailPoint us raste ko chhod dega.
​Fir wo list mein niche aayega aur dusre raste (Transition to: Stop) ki condition check karega.
​Agar dusre raste ki condition Fulfill (True) ho gayi, toh workflow Stop step par chala jayega.
​
💡 Sabse Important Baat: "Default" Rasta (Kya hoga agar sab False ho jayein?)
​Agar kisi step se 2 raste nikal rahe hain aur aapne dono par conditions laga di, aur dono ki dono False ho gayin, toh kya hoga? 
Workflow wahin atak (stuck ho) jayega aur error dega!
​Isiliye SailPoint mein ek Best Practice follow ki jati hai:-
​Hum hamesha ek "Default" rasta banate hain jisme hum koi bhi condition nahi daalte (sare option blank chhod dete hain).
​SailPoint ka ek rule hai: Jis transition mein koi condition nahi hoti, engine usko hamesha True manta hai.
​Toh hum pehle kuch transitions mein specific conditions (Rule, Script, etc.) lagate hain. Agar wo saari shartein fail (False) ho jayein,
toh end mein ek bina condition wala rasta rakhte hain, taaki workflow successfully us default step par chala jaye aur atke nahi.

()SailPoint IIQ workflows(Business Processes) mein jab ek se zyada logon se approval chahiye hota hai(jaise Manager,App Owner,aur Security Team),
toh hum in Approval Modes ka use karte hain.Yeh modes system ko batate hain ki approvals kis sequence mein lene hain aur rejection aane par kya karna hai.

1. Serial (Ek ke baad ek):-
Kaise kaam karta hai: Isme request bilkul line se (sequence mein) chalti hai. Pehle request Approver A ke paas jayegi.
Jab A approve karega, tabhi request Approver B ke paas jayegi, aur fir C ke paas.
Rejection par kya hoga: Agar beech mein kisi ne bhi (maan lijiye A ne) reject kar diya,
toh process wahin turant ruk jayega. B aur C ko request kabhi jayegi hi nahi.
Fayda: Time aur effort bachta hai, kyunki pehla approver reject kar de toh baaki logon ko disturb nahi karna padta.

2. SerialPoll (Ek ke baad ek, par sabse poocho):-
Kaise kaam karta hai: Isme bhi request sequence mein hi aati hai (A -> B -> C).
Rejection par kya hoga: Isme bada farq yeh hai ki agar A ne request reject bhi kar di,
tab bhi request aage B aur C ke paas jayegi. System process ko rokta nahi hai.
Fayda (Use Case): Yeh wahan kaam aata hai jahan organization ko "Audit Trail" ke liye sabka decision (vote) record karna hota hai,
chahe request final mein reject hi kyun na ho rahi ho.

3. Parallel (Sabko ek saath):-
Kaise kaam karta hai: Isme approval request ek hi time par sabhi approvers (A, B, aur C) ko ek saath bhej di jati hai.
Sab apna-apna inbox check kar sakte hain.
Rejection par kya hoga: Agar koi ek bhi approver request ko reject kar deta hai,
toh SailPoint request ko turant reject maan leta hai. Jo baaki approvers hain,
unke inbox se bhi wo request automatically gayab (cancel) ho jati hai.
Fayda: Process bahut fast ho jata hai kyunki logon ko ek dusre ka wait nahi karna padta.

4. ParallelPoll (Sabko ek saath, par sabka vote chahiye):-
Kaise kaam karta hai: Isme bhi request sabko ek saath (Parallel) bheji jati hai.
Rejection par kya hoga: Lekin yahan system sabke response aane ka wait karta hai.
Agar A ne reject kar diya, toh B aur C ke inbox se request cancel nahi hogi.Woh dono bhi apna approval ya rejection de sakte hain.

Fayda (Use Case): Yeh ek tarah ke "Voting System" jaisa hai.
Jab kisi badi access ke liye panel ke sabhi members ka opinion system mein darj karna zaroori ho (irrespective of final result),
tab iska use hota hai.

Jab target system mein success ka message aa jata hai, toh yeh workflow UI mein dikh rahi Request ke status ko "Pending"

Yeh workflow 5 minute tak shanti se baith kar wait karega.

5 minute baad yeh dusri koshish karega.

Agar 3 baar koshish karne ke baad bhi server error de raha hai,tab jaa kar yeh haar maan lega aur request ko sach mein

"Rahul ko SAP Admin access rehna chahiye ya remove?"
👉 Ye jo task hai = Certification.

👉 Specific user ka full access review

🔹 5. Account Group Certification:-
👉 Group level access review

🔹 6. Entitlement Certification:-
👉 Specific permissions review

👉 Specific location review your access.

()Access Certification (Review) Campaign ke "Lifecycle" (Timeline) setup ka page hai:-

1. Active Period (Main Review Phase):-
Active Period Duration (1 Month): Yeh wo main time hai jisme reviewers (jaise Managers ya App Owners) 
ke SailPoint inbox mein review task rahega. Is 1 mahine ke andar unhe login karke decision lena hoga ki kis access ko
"Approve" (rakhna hai) ya "Revoke" (hatana hai) karna hai.
 
2. Revocation Period (Access Hatane Ka Phase):-
Enable Revocation Period (Aapne isko Check/Tick kiya hua hai): Jab review (Active) period khatam ho jata hai,
tab yeh phase shuru hota hai.
Revocation Period Duration (1 Month): Jin logo ka access managers ne "Revoke" (hatane) ke liye mark kiya tha,
yeh wo time hai jisme SailPoint unka access target system (jaise Active Directory ya koi app) se actually remove karega.
Agar kuch error aata hai, toh IT team ke paas 1 mahine ka time hoga isko theek karke manually access hatane ka,
isse pehle ki campaign finally close ho.

3. Enable Challenge Period: Agar manager ne kisi user ka access revoke kar diya hai, 
toh yeh period us user ko ek chance (time) deta hai challenge karne ka: "Bhai, mera yeh access mat hatao, 
mujhe iski abhi bhi zaroorat hai project ke liye!".

4. Process Revokes Immediately: Agar aap isko tick kar dete, toh manager jaise hi kisi access par 'Revoke' aur 'Save' 
click karta, SailPoint campaign khatam hone ka wait nahi karta; wo background mein turant access hatane ka process (Provisioning) 
shuru kar deta.

5. Enable Automatic Closing:-
Yeh kya hai: Time khatam hone par apne aap File Band karna.
Kaam: Agar aapne total 2 mahine (1 mahina Active + 1 mahina Revocation) ka time set kiya hai,
toh 2 mahine baad SailPoint is campaign ko zabardasti "Closed" (Khatam) mark kar dega.
Agar kuch managers ne apna task poora nahi bhi kiya hoga, tab bhi campaign band ho jayega.
Agar yeh tick nahi hai, toh Administrator ko manually jaakar campaign ko close karna padta hai.

6. Active Period Enter Rule:-
Yeh kya hai: Ek custom BeanShell (Java) Rule/Script.
Kaam: Jaise hi aapka campaign "Active" (yani review) phase mein ghusne wala hota hai, 
aur aap chahte hain ki background mein kuch extra kaam ho (jaise kisi ko special notification bhejna ya koi custom log likhna),
toh aap wo custom script yahan attach kar sakte hain.

7. Enable Revocation Period (Image mein Tick hai):-
Yeh kya hai: Access delete karne (hatane) ka phase.
Kaam: Jab Managers apna review (Active phase) khatam kar lete hain, tab yeh phase shuru hota hai.
Is phase mein SailPoint un saare access ko target systems (jaise Active Directory ya Database) se actually remove karne ka
process (Provisioning) shuru karta hai.

8. Revocation Period Configuration:-
Yeh options tabhi khulte hain jab upar wala "Enable Revocation Period" tick ho.

()Revocation Period Enter Rule: Same logic jaisa Active rule ka tha. Jaise hi system kisi ka access hatane (Revoke phase) 
ke phase mein ghuse, aur tab aapko koi script chalani ho (jaise IT team ko alert bhejna), toh wo Rule yahan select karte hain.

()Revocation Period Duration: (Photo mein 1 Month set hai). Yeh access hatane ke liye diya gaya maximum time hai.
Agar background mein auto-provisioning fail ho jati hai aur access nahi hat pata, 
toh Helpdesk/IT team ke paas manual ticket resolve karke hathon se access hatane ke liye 1 mahine ka time hoga.

9. End Period Enter Rule:-
Yeh kya hai: Sabse aakhiri Rule/Script.
Kaam: Jab poora ka poora campaign (certification) bilkul khatam hone wala ho,tab agar aapko 
koi final task karwana ho(jaise poore audit ki ek PDF report generate karke CISO ya Auditors ko automatically email kar dena),
toh wo script yahan lagayi jati hai.

10. Challenge Period Start Notices To Certifiers:-
Kisko Email Jayega: Certifier (Manager/Reviewer) ko.
Kab Jayega: Jaise hi Challenge Period ki date shuru hoti hai.
Kya Likha Hoga: "Hello Manager, Challenge period shuru ho gaya hai.
Ab aapke team members aapke revoke kiye gaye decisions par challenge (objection) raise kar sakte hain."

11. Challenge Period End Notices To Certifiers:-
Kisko Email Jayega: Certifier (Manager/Reviewer) ko.
Kab Jayega: Jaise hi Challenge Period ka time khatam hota hai.
Kya Likha Hoga: "Hello Manager, Challenge period ab khatam ho gaya hai. Ab aapke paas koi naye challenges nahi aayenge."

12. Challenge Creation Notices To Challengers:-
Kisko Email Jayega: Challenger (User / Employee) ko jiska access hataya ja raha hai.
Kab Jayega: Jab uska access revoke mark hota hai aur challenge period shuru hota hai.
Kya Likha Hoga: "Hello User, Aapke manager ne aapka 'XYZ' access hatane ka decision liya hai.
Agar aapko lagta hai ki aapko is access ki abhi zaroorat hai, toh aap portal par login karke is decision ko 'Challenge' kar sakte hain."

13. Challenged Decision Notices To Certifiers:-
Kisko Email Jayega: Certifier (Manager) ko.
Kab Jayega: Jab koi user actually mein portal par jakar "Challenge" button daba deta hai aur reason likh deta hai.
Kya Likha Hoga: "Alert! Aapke ek team member ne aapke access hatane ke decision par objection uthaya hai.
Kripya SailPoint mein login karein aur unke challenge ko Approve ya Reject karein."

14. Challenge Expiration Notices To Challengers
(Aapke system mein yeh abhi blank/Select Template par set hai)
Kisko Email Jayega: Challenger (User) ko.
Kab Jayega: Jab user ke paas challenge karne ka time nikal jata hai aur usne koi action nahi liya hota.
Kya Likha Hoga: "Aapke paas apne access revoke ko challenge karne ka jo time tha, wo ab expire ho gaya hai."

15. Challenge Decision Expiration Notices To Challengers And Certifiers
(Yeh bhi abhi blank set hai)
Kisko Email Jayega: Dono ko (User aur Manager).
Kab Jayega: Maan lijiye user ne challenge kiya, par Manager ne time limit ke andar us challenge ko padh kar Accept ya Reject nahi kiya.
Kya Likha Hoga: "Yeh challenge request time-out (expire) ho gayi hai kyunki is par koi decision nahi liya gaya."

16. Challenge Accepted Notices To Challengers:-
Kisko Email Jayega: Challenger (User) ko.
Kab Jayega: Jab Manager user ki baat maan leta hai aur challenge ko "Accept" kar leta hai.
Kya Likha Hoga: "Badhiya khabar! Aapke manager ne aapka challenge accept kar liya hai. Ab aapka access nahi hataya jayega."

17. Challenge Rejected Notices To Challengers:-
Kisko Email Jayega: Challenger (User) ko.
Kab Jayega: Jab Manager user ki appeal ko cancel kar deta hai aur challenge "Reject" kar deta hai.
Kya Likha Hoga: "Sorry, aapke manager ne aapka challenge reject kar diya hai. Aapka access system se hataya (revoke kiya) jayega."

18. Suppress Initial Notifications: 'Suppress' ka matlab hota hai daba dena ya rokna. Agar aap is box ko tick kar denge,
toh campaign shuru hone par kisi bhi Manager (Reviewer) ko koi email nahi jayega. (Yeh aam taur par testing ke time use hota hai).

25. recipients-rule->Iski Zaroorat Kyun Padti Hai? (The Problem)
Maan lijiye aapke workflow mein ek step hai jisme "Access Approved" ka email bhejna hai.
Agar aapko hamesha kisi ek hi insaan (jaise admin@company.com) ko mail bhejna hai, toh aap direct wo email ID likh sakte hain.
Isme koi logic nahi chahiye.
Lekin, agar aapko email hamesha us user ke "Manager", ya us application ke "Owner" ko bhejna hai, 
toh har user aur app ke hisaab se email ID alag-alag hogi. Aap email ID fix (hardcode) nahi kar sakte.
Yahin par Recipients Rule kaam aata hai. Yeh background mein dimag lagata hai aur sahi insaan ko dhundh kar lata hai.

ex->Ek Simple Example:-
Maan lijiye ek naya access manga gaya hai, aur aapko requester ke Manager aur Department Head ko mail bhejna hai.
Aapka Recipients Rule kuch aisa logic lagayega:-
--Kisne request ki? (User = Rahul)
--Rahul ka Manager kaun hai? (Manager = Rohan)
--Rohan ki email id nikalo (rohan@company.com).
--Rahul ka Department kya hai? (Dept = IT)
--IT ka Head kaun hai? (Head = Amit)
--Amit ki email id nikalo (amit@company.com).
--Dono emails ko ek List mein daal kar SailPoint ko wapas de do.
--Ab SailPoint us email template ko uthayega aur in dono IDs par bhej dega.

ex->Iska Sabse Bada Real-World Example (Escalation):-
Maan lijiye aapka rule (policy) kehta hai ki: "Agar koi manager 3 din tak apna review complete nahi karta,
toh reminder email us manager ko toh bhejo hi, par sath mein uske Senior Manager (Boss) ko bhi us mail mein Cc kar do,
taaki thoda pressure bane."

Yeh fix (hardcoded) rahega.

dikhayega. Uske andar ke chhote-chhote parts (Tier apps jaise AD ya Oracle) ke access ko screen se gayab (exclude) kar dega.
Kyun use hota hai: Taaki Manager confuse na ho. Ek Manager ko kya pata ki HR Portal chalane ke liye pichhe kaunsi database table chahiye!
Wo sirf "HR Portal" ka access approve karega, aur SailPoint background mein khud samajh jayega ki dono tier apps ka access bhi rakhna hai.
Isse duplicate review se bacha jata hai.

63. Filter Logical Application Entitlements:-
Yeh kya karta hai: Yeh lagbhag upar wale option ka ulta (opposite) hai.
Kayi baar Logical App par jo entitlement (access) hota hai, wo sirf ek 'dikhawe' (virtual representation) ke liye hota hai,
jabki asli access uske Tier apps par hota hai. Agar aap isko tick karte hain,
toh SailPoint us virtual "Logical" access ko list se filter karke hata dega,
taaki manager sirf asli hardware/software level (Tier apps) ka review kare.
Kyun use hota hai: Yeh tab use hota hai jab company chahti hai ki managers bilkul deep technical level(actual system level)
par review karein, na ki upar-upar se kisi virtual grouping ka. Isse double-entry (ek hi access ka 2 baar review) filter ho jati hai.

64. Pre-delegation Rule:-
Isko naam se tod kar samajhte hain: Pre (Pehle) + Delegation (Task saunpna) + Rule (Java/BeanShell Script).
Yeh kya karta hai: Jab SailPoint ek campaign (review) banata hai, toh wo system ke hisaab se tasks Managers ko assign kar deta hai.
Lekin Campaign Live hone aur Managers ko Email jane se theek pehle, background mein yeh Rule (script) chalti hai.
Real-World Example: Maan lijiye system ne Rahul (Manager) ko ek task diya. Par 'Pre-delegation rule' mein humne logic likha hai:
"Agar koi access 'SAP Finance' ka hai, toh wo kisi Manager ko nahi, balki sidha 'Finance Head' ko review ke liye jana chahiye."
Toh email aane se pehle hi,yeh Rule wo specific task Rahul ki list se nikalega aur automatically Finance Head ko delegate(forward) kar dega.
Fayda: Campaign start hone se pehle hi complex policies ke hisaab se tasks ko sahi (custom) logon tak pahuncha dena, bina kisi manual kaam ke.

()9️⃣ Policy
Policy policy:-
Kya hai: Ye rules ka set hai jo access aur entitlements ko govern karta hai.
Kya karta hai:-
Access reviews, SoD, Least Privilege enforce karta hai
Violations detect karta hai.

1️⃣ Identity/Access Policy:-
Ye define karti hai ki ek user ko kaunse system, application ya data ka access milega.
Example: "Finance team ke employees sirf finance application access kar sakte hain."

2️⃣ Entitlement Policy:-
Ye specific permissions ya roles ke liye rules define karti hai.
Example: "Manager role wale users ko payroll data read karne ki permission milegi, write nahi."

3️⃣ Password/Authentication Policy:-
Ye security rules define karti hai login ke liye.
Example: "Password kam se kam 8 characters ka ho, aur 90 din me change karna mandatory ho."

4️⃣ Policy Violation:-
Jab koi user defined rules ko violate karta hai, to system usko detect karta hai.
Example: Agar koi employee sensitive data ko unauthorized share karta hai, to Policy Violation trigger hoti hai.

1. Role SOD Policy (Segregation of Duties):-
​Kya hai: Ye check karta hai ki kisi user ke paas do aise conflicting Roles toh nahi hain jo ek sath nahi hone chahiye.
SOD ka main purpose fraud aur errors ko rokna hota hai.
​Example: Ek hi user ke paas "Invoice Creator" (Maker) aur "Invoice Approver" (Checker) dono roles nahi hone chahiye.
Agar kisi ko ye dono roles assign hote hain, toh ye policy violation generate kar degi. 😁

​2. Entitlement SOD Policy:-
​Kya hai: Ye Role SOD ke jaisa hi hai, lekin ye roles ke badle direct Entitlements (permissions/privileges) par kaam karta hai.
Ye check karta hai ki user ke paas do conflicting entitlements toh nahi hain,
 chahe wo use kisi bhi role ke through mili ho ya direct assign hui ho.
​Example: Kisi target application (jaise Active Directory ya SAP) me ek hi user ke paas "Read_Salary_Data" aur "Modify_Salary_Data" ki
entitlement ek sath nahi honi chahiye.

3. Activity Policy:-
​Kya hai: Ye policy user ki system Activity ya behavior ko monitor karne ke liye hoti hai.
Ye data usually SIEM tools ya event logs se SailPoint me aata hai.
​Example: Agar koi user non-business hours (jaise raat ke 2 baje) me sensitive files download kar raha hai,
ya fir lagatar kai failed login attempts kar raha hai, toh Activity Policy trigger hoke violation create karegi.

4. Account Policy:-
​Kya hai: Ye policy ensure karti hai ki kisi bhi user ke Accounts company ke standard rules (baseline) ko follow kar rahe hain ya nahi.
Ye account attributes aur states ko check karti hai.
​Example: Kisi bhi employee ke paas Active Directory me ek se zyada active account nahi hone chahiye.
Ya phir, agar kisi account me last 90 days se password change nahi hua hai, toh ye policy violation pakad legi.

5. Risk Policy:-
​Kya hai: SailPoint har user ko unke access, entitlements, aur behavior ke hisaab se ek Risk Score deta hai.
Ye policy us risk score par aadharit hoti hai.
​Example: Aap ek rule set kar sakte hain ki agar kisi bhi user ka Overall Risk Score 800 (high risk) cross kar jata hai,
toh automatic ek policy violation trigger ho jaye, jisse manager us user ka access turant review (certify) kar sake.

​6. Advanced Policy:-
​Kya hai: Jab aapki security requirement itni complex ho ki upar di gayi 5 policies se solve na ho sake,
tab Advanced Policy ka use hota hai. Isme aap Beanshell Rule (script) likh kar apna custom logic define kar sakte hain.
​Example: Ek complex condition: "Agar user ka Department 'HR' hai, aur uski Location 'India' hai, lekin uske paas 'UK_Payroll_Folder'
ka access hai, tab hi violation trigger karo."

()role remove ke baad refresh identity cube ke baad (jhonson)identity me role wapash aane ke karan?
-Ye SailPoint ka ek bohot hi classic aur common issue hai! Isko technical bhasha mein "Rubber-banding effect" ya "Birthright Conflict" kehte hain.
​Aaiye samajhte hain ki ye kyu hota hai aur isko kaise theek kiya jaye.
()​Ye Kyu Hota Hai? (The Root Cause)
​--Ye problem tab aati hai jab wo Role (jisko manager ne revoke kiya hai) kisi Assignment Rule ya Match List ke through assign kiya gaya ho.
​
--Process samajhiye: Maan lijiye us Role par ek rule laga hai ki “Agar user ka Department 'IT' hai, toh usko ye Role de do.”
​Manager ne task mein aakar Role ko Revoke kar diya. SailPoint ne ek baar ke liye wo Role hata diya.
​Lekin, us user ka Department toh abhi bhi system mein 'IT' hi hai!
​Isliye, jaise hi aap Identity Refresh (piche 'Refresh assigned, detected roles' par tick lagakar) run karte hain,
SailPoint fir se check karta hai. Wo dekhta hai ki "Arre, iska department toh IT hai, iske paas ye role hona chahiye",
aur wo wapas se us role ko assign kar deta hai.
​Ye ek loop ban jata hai: Policy usko hatane ko bolti hai -> Manager hata deta hai ->Identity Refresh rule check karke wapas de deta hai->
Policy fir se naya violation task bana deti hai.
​
()Isko Kaise Theek Karein? (Solutions)
​Apne project mein is problem ko solve karne ke liye aapke paas 3 main raaste hain:
​1. Identity ka Data (Attributes) Change Karein
Agar us bande ko sach mein wo Role nahi chahiye, toh us role ko manual revoke karne se kaam nahi chalega.

<Rule language="beanshell" name="Last Login Greater Than 180 Days" type="Policy">
  <Description>
    This rule evaluates if an active identity has not logged in for more than 180 days.
  </Description>

      <Argument name="log" type="org.apache.commons.logging.Log"/>
      <Argument name="context" type="sailpoint.api.SailPointContext"/>
      <Argument name="identity" type="sailpoint.object.Identity"/>
      <Argument name="policy" type="sailpoint.object.Policy"/>
      <Argument name="constraint" type="sailpoint.object.BaseConstraint"/>
    </Inputs>

      <Argument name="violation" type="sailpoint.object.PolicyViolation"/>
    </Returns>

    import sailpoint.object.Identity;
    import sailpoint.object.PolicyViolation;
    import java.util.Date;
    import java.util.Calendar;

    // Agar identity already inactive hai, toh policy check skip kar do (optional but recommended)
    if (identity.isInactive()) {
        return null;
    }

    PolicyViolation violation = null;

    // Apna Identity attribute ka sahi naam yahan daalein (e.g., 'lastLogin')
    Object lastLoginObj = identity.getAttribute("lastLogin");

    if (lastLoginObj != null) {
        Date lastLoginDate = null;

        // Ensure the attribute is stored/treated as a Date object
        if (lastLoginObj instanceof Date) {
            lastLoginDate = (Date) lastLoginObj;
        }

        if (lastLoginDate != null) {
            // Aaj ki date se 180 days minus karke threshold date nikalein
            Calendar cal = Calendar.getInstance();
            cal.add(Calendar.DAY_OF_YEAR, -180);
            Date thresholdDate = cal.getTime();

            // Check karo ki last login 180 days se purana (before) hai ya nahi
            if (lastLoginDate.before(thresholdDate)) {
                violation = new PolicyViolation();
                violation.setActive(true);
                violation.setIdentity(identity);
                violation.setPolicy(policy);
                violation.setConstraint(constraint);
                violation.setDescription("User violation: Account has not been logged into for over 180 days. Last Login: " + lastLoginDate.toString());
                violation.setStatus(PolicyViolation.Status.Open);

                // Violation ka owner set karein (Manager ko assign hoga, ya fail-safe me spadmin ko)
                Identity manager = identity.getManager();
                if (manager != null) {
                    violation.setOwner(manager);
                } else {
                    violation.setOwner(context.getObjectByName(Identity.class, "spadmin"));
                }
            }
        }
    }

    return violation;

1. Pehli Image: Baseline Access Risk (BAR):-

"Certification Age" isi risk ko measure karta hai. Jitna purana certification, utna zyada risk score.

()Composite risk kya hai?
--Composite Risk = Multiple individual risks ko combine karke ek single risk score nikalna.
Matlab ek project ya system me kai alag-alag risk ho sakte hain,
In sabko alag-alag dekhne ki bajaye, unka overall risk impact dekha jata hai.
Formula example:-
Composite Risk Score=Risk1 Score+Risk2 Score+⋯+RiskN Score

3️⃣ Example Table:-
Risk Name	       Probability	     Impact (₹)	         Risk Score
Server Crash----------0.2------------1,00,000--------------20,000
Data Leak-------------0.5------------2,00,000--------------1,00,000
Project Delay---------0.3------------50,000----------------15,000
Composite Risk Score = 20,000 + 1,00,000 + 15,000 = 1,35,000.
→ Matlab overall project ka risk ab 1,35,000 hai, jo High priority ka risk dikhata hai.

Simple Samjha:-
Probability – Risk hone ka chance (Low, Medium, High)
Impact – Agar risk ho gaya to nuksan kitna hai (Low, Medium, High)
Risk score/Weighting = Probability × Impact

Example Table:-
Risk Type	       Probability (1–5)	 Impact (1–5)	 Risk Weight = P×I
Data Breach-------------4----------------------5----------------20
Server Downtime---------3----------------------4----------------12
Human Error-------------2----------------------3-----------------6
High Risk → Weight 15–25
Medium Risk → Weight 6–14
Low Risk → Weight 1–5

()Previous Result Action ka matlab hota hai ki jab aap ek report ko dobara run (execute) karte hain,
toh SailPoint ko purane results ke saath kya karna chahiye. IdentityIQ mein report run hone ke baad ek "Task Result" banta hai,
aur yeh option usi result ki management ke liye hai.

Isme aapko teen main options milte hain:-

--Rename Old: Yeh sabse safe option hai.Isme system purane report result ka naam badal deta hai (aksar usme date aur time jod deta hai) 
aur naya result asli naam se save karta hai. Isse aapke paas purani reports ka record bana rehta hai.

--Delete: Agar aapko purane data ki zaroorat nahi hai aur aap system ki storage bachana chahte hain, toh yeh option select karein. 
Nayi report chalte hi purana result delete ho jayega.

--Override: Isme naya data purane result record ke upar hi likh diya jata hai.Isse result list mein hamesha sirf ek hi entry dikhegi jo sabse latest hogi.

()Description: Report kis baare mein hai, uska chhota sa vivaran.

()Email Recipients: Kin logo ko report email par bhejni hai, unhe yahan select karein.

()Email Attachment Format: Email mein report PDF chahiye ya CSV, woh yahan se select hota hai.

()Don’t email empty reports: Agar report mein koi data nahi hai, toh email nahi jayega.

()Host:-
Is option ka istemal report ki performance aur resource management ke liye kiya jata hai:-
Kya hai: Yeh batata hai ki report kis specific server (Task Server) par run honi chahiye.
Kaise kaam karta hai: Agar aapke environment mein kai servers (Nodes) hain, toh aap yahan kisi khaas machine ka naam likh sakte hain.
Isse woh report sirf usi machine par chalegi, taaki baaki nodes par load na bade.
Kyun select karein: Agar koi report bahut badi hai aur database par load daal sakti hai,
toh use kisi aise host par schedule kiya jata hai jo heavy processing ke liye dedicated ho.
1. Kiska Naam Denge?
Application Server/Task Server: Agar aapke environment mein multiple SailPoint nodes (servers) chal rahe hain,
 toh aap us specific server ka Host Name ya Network Name denge jis par IdentityIQ installed hai.
Localhost (Single Server): Agar aapka SailPoint sirf ek hi server par setup hai, 
toh zyadatar cases mein ise khali chhoda jata hai ya wahan system ka default name aata hai.

()Signoff Properties:-
Jab aap Require Signoff ko on karte hain, tab yeh extra options khul jaate hain:-
Initial Notification Email: Signer ko bheje jaane wala pehla email template.
Isme bataya jata hai ki "Aapke liye ek report approval ke liye pending hai".

--Escalation: Yeh dhyan rakhta hai ki agar koi signer report sign nahi kar raha hai toh kya ho:-
--None: Kuch nahi hoga.
--Send Reminders: User ko baar-baar reminder email jayenge.
--Reminders then Escalation: Pehle reminders jayenge, phir report unke manager ke paas approval ke liye chali jayegi.
--Escalation Only: Seedhe manager ke paas bhej di jayegi.
--Signers: Yahan aap un logo ya Workgroups ko select karte hain jinhe report approve karni hai (Jaise: IT Compliance Team).

()Allow Concurrency ek technical setting hai jo yeh decide karti hai ki ek hi report ke multiple instances allowed hain ya nahi.

1. Jab Checkbox UNCHECKED ho (Default Setting):-
Agar aapne ise select nahi kiya hai, toh system "One at a time" rule follow karta hai:-
Queue System: Agar "Identity Details Report" pehle se chal rahi hai aur aapne (ya kisi aur ne) wahi report dobara "Execute" kar di,
toh doosri report turant nahi chalegi.
Wait State: Doosri report "Pending" ya "Queued" status mein chali jayegi aur tabhi shuru hogi jab pehli wali khatam ho jaye.
Fayda: Yeh server ke database aur CPU par load kam rakhta hai.

2. Jab Checkbox CHECKED ho (Allow Concurrency = ON):-
Agar aap ise tick kar dete hain, toh system "Parallel Processing" ki ijazat deta hai:-
Multi-Tasking: Ek hi report ke 2, 3 ya usse zyada instances ek saath chal sakte hain.
Different Filters: Maan lijiye aap ek hi report ko do alag-alag departments (jaise HR aur IT) 
ke liye ek hi samay par run karna chahte hain, toh yeh option kaam aata hai.
Nateeja: Dono reports apna-apna data process karengi aur alag-alag Task Results generate karengi.

()Name Fields (First Name, Last Name, Display Name):-

Inhe khali chhodne ka matlab hai ki aap naam ke basis par koi filter nahi laga rahe.

()Inactive:-
Yeh ek Radio Button selection hai jo user ke status par filter lagata hai:-
No Selection (Default): Iska matlab hai filter off hai. Report mein Active aur Inactive dono tarah ke log dikhenge.
True:Sirf wahi log report mein aayenge jo Inactive hain(wo log jo company chhod chuke hain ya jinka account disable kar diya gaya hai).
False: Sirf wahi log aayenge jo Active hain (current employees).

()Mahatvapoorn Baat (Logic):-
Jab aap ek se zyada fields bharte hain (jaise Manager select kiya aur Inactive ko "False" kiya),
toh SailPoint "AND" logic lagata hai. Iska matlab report mein sirf wahi log aayenge jo:-
Us Manager ke under kaam karte hain AUR
Jo Active (False) hain.

()Email:-
Yeh field identity ke primary email address ke basis par filter lagati hai.
Kya karta hai: Yeh Identity Cube mein store kiye gaye email attribute ko scan karta hai.
Kaise kaam karta hai (Selection):-
Specific Email: Agar aap "john.doe@company.com" likhte hain, toh report mein sirf wahi ek person aayega.
Wildcard Search: Aap isme asterisk (*) ka use kar sakte hain. Jaise agar aap *@gmail.com likhte hain,
toh report un sabhi users ki list nikal degi jo company email ki jagah apna personal Gmail use kar rahe hain.
Kyun select karein: Iska use tab kiya jata hai jab aapko specific users ko target karna ho ya check karna ho ki kis-kis user ka email format galat hai.

()2. Manager:-
Yeh field "Reporting Hierarchy" (kaun kiske neeche kaam karta hai) par filter lagati hai.
Kya karta hai: SailPoint mein har identity ka ek manager attribute hota hai. Yeh field us attribute ke basis par data nikaalti hai.
Kaise kaam karta hai (Selection):-
Isme ek dropdown ya search box hota hai jahan se aap kisi bhi employee (manager) ka naam select kar sakte hain.
Jab aap kisi ko select karte hain, toh report sirf un logon ki list dikhayegi jo us manager ko Directly Report karte hain.
Kyun select karein:-
Team Audit: Agar kisi manager ko apni poori team ka access review karna hai, toh unke naam se yeh report nikali jati hai.
Access Control: Yeh check karne ke liye ki kya kisi manager ke under aise log toh nahi hain jinhe wahan nahi hona chahiye.

()Pro Tip:-

Is tarah se aap bahut hi precise (sateek) data nikaal sakte hain.

()Yeh Option Kya Karta Hai (Function):-

toh aap unka Employee ID yahan daal sakte hain.

Specific ID: Agar aap yahan ek number (jaise E12345) likhte hain, toh report mein sirf wahi ek insaan dikhayi dega jiska ID match karega.
Range ya Pattern: Kai baar SailPoint configurations mein aap 1* (jiski shuruat 1 se ho) 
jaisa pattern bhi use kar sakte hain (yeh aapke environment ki configuration par depend karta hai).

()Applications:-
Kya karta hai: Yeh identities ko unke linked accounts ke basis par filter karta hai.
Kyun select karein: Agar aapko dekhna hai ki kin-kin users ke paas Active Directory ya SAP ka account hai,

()Capabilities:-
Kya karta hai: Yeh SailPoint ke internal permissions (Admin rights) par filter lagata hai.
Kyun select karein: Audit ke waqt agar aapko yeh check karna hai ki poori company mein kin logon ke paas

()Roles:-

Kyun select karein: Agar aapko "Manager Role" wale saare employees ki list chahiye,

()Groups:-
Kya karta hai: Identities kisi na kisi group (jaise AD Groups ya Workgroups) ka hissa hoti hain.
Kyun select karein: Agar aapko kisi specific department group (jaise "Finance_Users") ke bando ka data chahiye, toh ise use karein.

()Last Refresh Date:-

Kyun select karein: Yeh check karne ke liye ki pichle 24 ghante ya ek hafte mein kin logon ka profile system mein update hua hai.

()Last Login Date:-

Selection: Isme bhi Start Date aur End Date ki range di jati hai.
Kyun select karein: Yeh security audit ke liye sabse important hai. 
Isse aap un "In-active" logon ko dhoondh sakte hain jinhone pichle 90 dinon se login nahi kiya hai, 
taaki unka access khatam kiya ja sake.

Yeh audit ke liye bahut kaam aata hai taaki pata chale report kaise nikali gayi thi.

()1. Sort By (Data ka kram tay karna):-

system report ke saare records ko usi ke hisaab se line mein laga dega.
Kyun select karein: Agar aapko report alphabetical order mein chahiye taaki kisi user ko dhoondhna asaan ho, 
toh aap 'Username' ya 'Last Name' yahan select karte hain.

()2. Sort Ascending (Kram ki disha):-
Kya hai: Yeh ek checkbox hai jo 'Sort By' ke theek bagal mein hota hai.
Selection:-

Unchecked: Data Z to A ya bade se chote (10 to 1) kram mein lagega.

()3. Group By (Data ke tukde karna):-
Kya hai: Yeh option data ko categories mein baantne ke liye use hota hai.
Kaise kaam karta hai: Agar aap yahan 'Manager' select karte hain, toh report mein pehle ek manager ke saare employees dikhenge,
phir doosre manager ke.
Nateeja: Har group ke khatam hone par system aksar ek sub-total ya summary dikhata hai,jo report ko aur bhi professional banata hai.
