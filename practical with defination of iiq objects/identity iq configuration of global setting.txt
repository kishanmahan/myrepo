1. Email Notification Type (Dropdown Options):-
Yeh decide karta hai ki IdentityIQ emails kaise bhejega:-

--SMTP/Basic: Sabse common option. Yeh normal email server (jaise Gmail ya Office 365) ka use karke mail bhejta hai.

--HTTP/OAuth: Agar aapka email server modern authentication (OAuth) mangta hai, toh iska use hota hai.

--Redirect to Email: Saare system emails kisi ek specific email address par redirect ho jaayenge (Testing ke liye best hai).

--Redirect to File: Emails bhejane ke bajaye system unhe ek file mein save kar dega.

2. Email Server Details:-
--Encryption: Aapka connection secure hai ya nahi (jaise SSL/TLS).
--Default SMTP Host: Aapke email server ka address (e.g., smtp.office365.com).
--Default SMTP Port: Server ka port number (jaise 25, 465, ya 587).
--Default From Address: Woh email id jo "Sender" ke naam se dikhegi (e.g., noreply@company.com).

3. Authentication (Login):-
--Username & Password: Email server mein login karne ke liye credentials.
--Confirm Password: Password verify karne ke liye dobara type karna hota hai.

4. Advanced Settings:-
--Maximum Email Retries: Agar mail send nahi hua, toh system kitni baar koshish (try) karega.
--Connection Timeout: Mail server se connect hone ke liye system kitni der wait kare.
--Read Timeout: Server se response aane ka maximum intezar kitna hoga.
--Suppress Duplicate Emails: Agar ek hi tarah ka email baar-baar ja raha hai, toh yeh use rok dega taaki user ka inbox bhar na jaaye.

()Server Root Path: Yeh aapke IdentityIQ ka URL hai (jaise http://localhost:8080/identityiq).
Jab email mein koi link jaata hai (jaise "Click here to approve"), toh woh link isi path ka use karke banta hai.

()Password Policy (Complexity Rules):-
Yahan aap password ke niyam set karte hain:-
--Minimum/Maximum number of characters: Password kitna chhota ya bada ho sakta hai.
--Minimum number of letters/digits: Password mein kam se kam kitne alphabets aur numbers hone chahiye.
--Minimum uppercase/lowercase letters: Bade aur chhote aksharon ki sankhya.
--Minimum special characters: @, #, $ jaise symbols kitne hone chahiye.
--Number of repeated characters allowed: Ek hi akshar kitni baar repeat ho sakta hai (e.g., 'aaaa' ko rokne ke liye).

()Password history length: User apne pichle kitne purane passwords dobara use nahi kar sakta (e.g., last 5 passwords).
()Triviality check against old password: Check karta hai ki naya password purane waale se bahut milta-julta toh nahi hai.
()Days until expiration (Manual/Generated): Password kitne dino baad expire ho jayega aur user ko badalna padega.
()Minimum Hours between password changes: Ek baar badalne ke baad user kitni der tak dobara password nahi badal sakta.
()Validate passwords against identity's list of attributes: User apna naam, employee ID, 
ya date of birth password mein use na kar sake, yeh usse rokta hai.
()Validate passwords against the password dictionary: Common passwords (jaise 'Password123') 
ko block karne ke liye dictionary ka use karna.

1. Other Object Expirations (Cleanup Settings):-
Iska kaam purane data ko delete karke database ki performance banaye rakhna hai.

--Days before snapshot deletion: Purane identity snapshots kitne din baad delete honge. (0 ka matlab hai kabhi nahi delete honge).

--Days before task result deletion: Task results (jaise aggregation ya refresh reports) kitne din baad hataye jayenge.

--Days before certifications are archived: Jab ek access review khatam hota hai, toh kitne din baad use archive (history mein save) kiya jaye.

--Days before certification archive deletion: Archive ki gayi certification history ko kitne din baad hamesha ke liye delete kiya jaye.

--Minutes before object locks are released: Agar koi user kisi identity ya role par kaam kar raha hai aur session disconnect ho jaye,
toh kitne minute baad woh object unlock ho jaye.

--Days before provisioning request logs expire: Provisioning requests (accounts banana/mitana) ki history kitne din tak save rahegi.

()Prohibit scripts from accessing plugin-loaded classes:-
Kaam: Yeh ek security feature hai. Agar ise check karte hain, toh IdentityIQ ke normal scripts (jaise Rules ya Workflows) 
un files aur codes (classes) ko nahi dekh payenge jo kisi plugin ne load ki hain.
Kyun use karein: Yeh plugins aur system ke baaki hisson ke beech ek diwar bana deta hai taaki koi plugin system ke core functions mein interfere na kar sake.

()Relax strict declaration enforcement:-
Kaam: Purane ya custom-built plugins kabhi-kabhi naye security standards ko follow nahi karte. 
Ise check karne se system thoda "liberal" (narm) ho jata hai aur un plugins ko chalne ki ijaazat de deta hai jo 
shayad strict rules par fail ho jate.
Kyun use karein: Agar aapka koi purana plugin error de raha hai, toh ise enable karke use chalaya ja sakta hai.
