()Monitoring Library:-
SailPoint IdentityIQ (IIQ) ke workflows mein Monitoring Library (sailpoint.workflow.MonitoringLibrary) 
ek behad mahatvapurna backend framework helper hai. Iska mukhya kaam workflow ke lifecycle ko monitor karna, 
system health checks track karna,background background processes(jaise background tasks ya background threads) ko update karna, 
aur failure scenarios mein automatic notifications trigger karna hai.
Jab aap workflow ke top element par libraries="Monitoring" define karte hain, 
toh is library ke ready-made control methods aapke execution pipeline ke liye active ho jaate hain.

()1. checkIdentity (User Consistency Checker):-
Asli Kaam (Simplest Definition): Yeh method check karta hai ki jis user profile par aapka workflow chal raha hai, 
kya woh user database (spt_identity table) mein abhi bhi sahi-salamat, active aur safe state mein hai ya nahi.
Backend Mechanism: Ek bade business workflow ko chalne mein kai baar ghante ya din lag jaate hain 
(jaise manager ke approval ka wait karna). Is dauran, background mein koi doosra nightly aggregate task chal sakta hai jo galti 
se us user ko delete ya correlate se hata de. checkIdentity step chalte hi database par query maarta hai aur verify karta hai ki
target identity abhi bhi intact hai ya nahi, taaki workflow aage crash na ho.
 
()2. sendMonitoringEmail (The Panic Button / Alert Machine):-
Asli Kaam (Simplest Definition): Agar workflow ke chalte-chalte koi bada technical crash, connector timeout, 
ya coding exception aa jaye, toh yeh method turant SailPoint Admin ko ek alert email bhej deta hai.
Backend Mechanism: Jaise hi workflow ke provisionProject step mein koi bad error aata hai 
(jaise Active Directory ka port lock milna), workflow engine control ko catch block mein bhejta hai. 
Wahan sendMonitoringEmail call hota hai, jo error stack-trace, user details, aur request ID ko uthakar ek predefined admin email
template mein pack karta hai aur automatic support desk (SMTP) par ticket generate karne ke liye mail fire kar deta hai.

()clearMonitoringForm:-
Jab koi workflow chal raha hota hai aur woh kisi technical state mein phas jata hai 
(jaise koi unexpected connector error ya system timeout), toh SailPoint Admin Central Dashboard 
(Setup -> Business Processes ya Monitor -> Workflows) par ek administrative screen ya control panel generate ho sakta hai. 
Is management dashboard par baithkar System Administrator manually kuch inputs barta hai—jaise kisi variable ko force-update 
karna, workflow thread ko aage push karna, ya workflow state ko modify karna.
Jaise hi administrator us monitoring control form ko submit karke thread ko forward ya fix kar deta hai, 
toh us interface ka poora raw metadata aur UI state workflow ke temporary cache memory (transient context) mein fasa reh jata hai.
clearMonitoringForm method ka kaam isi temporary dashboard form data aur structural metadata ko execution memory (RAM) se 
completely clear (wipe-out) karna hai.

example:-
<Workflow name="Enterprise-Monitoring-Core-Pipeline" libraries="Monitoring,Identity">

  <Variable name="identityName" input="true" required="true"/>
  <Variable name="project"/>

  <!-- STEP 1: Verify Identity State before heavy engine trigger -->
  <Step name="Validate Identity Integrity" action="call:checkIdentity">
    <Arg name="identityName" value="ref:identityName"/>
    <!-- Agar identity valid hai, toh target process par jao -->
    <Transition to="Execute Target Connector Provisioning"/>
  </Step>

  <!-- STEP 2: Actual Execution (Yahan exception aa sakta hai) -->
  <Step name="Execute Target Connector Provisioning" action="call:provisionProject">
    <Arg name="project" value="ref:project"/>
    
    <!-- Positive Path: Agar successful raha toh pipeline khatam -->
    <Transition to="End"/>
    <!-- Catch Exception Path: Agar connector crash hua toh handling block par dalo -->
    <Transition to="Handle System Exception Failure" condition="script:(size(errors) &gt; 0)"/>
  </Step>

  <!-- STEP 3: Alert Administration Team via Monitoring Email -->
  <Step name="Handle System Exception Failure" action="call:sendMonitoringEmail">
    <Arg name="template" value="LCM Workflow Connection Failure Template"/>
    <Arg name="to" value="iiq-support-admin@company.com"/>
    <Arg name="identityName" value="ref:identityName"/>
    <Transition to="Cleanup Monitoring State"/>
  </Step>

  <!-- STEP 4: Flush dynamic state memory garbage -->
  <Step name="Cleanup Monitoring State" action="call:clearMonitoringForm">
    <Transition to="Force Terminate Thread"/>
  </Step>

  <Step name="Force Terminate Thread" icon="Stop">
    <!-- Handle safe rollback or graceful stop logic -->
    <Transition to="End"/>
  </Step>

  <Step name="End"/>
</Workflow>

()Policy Violation Library:-
SailPoint IdentityIQ (IIQ) ke andar Policy Violation Library (sailpoint.workflow.PolicyViolationLibrary) ek behad specialized 
helper module hai. Iska mukhya kaam workflow execution ke dauran compliance checks run karna, 
toxic access combinations (Segregation of Duties - SoD) ko detect karna, aur unke violations ki tracking aur mitigation policies
ko back-end database (spt_policy_violation table) mein manage karna hai.
Jab aap kisi workflow mein <Workflow libraries="PolicyViolation"> define karte hain, 
toh is library ke built-in methods directly pipeline mein call karne ke liye available ho jaate hain.

()🔹 checkPolicyViolations:-
Real Project Example: Ek employee ne ek sath do roles request kiye: "Invoice Creator" aur "Invoice Approver". 
Jaise hi request submit hoti hai, checkPolicyViolations background mein SoD rules scan karta hai aur turant detect kar leta hai
ki yeh ek toxic combination hai. Yeh method request ko block karne ke bajay ek PolicyViolation object context generate karke
workflow mein attach kar deta hai.
 
()🔹 sendViolationNotifications:-
Detailed Definition: Jab checkPolicyViolations dwara koi violation confirm ho jaati hai, 
toh compliance team, security group, ya request manager ko alert karne ke liye yeh method call hota hai. 
Yeh method violation metadata (Violation Type, Severe Risk Score, Policy Name, Requester Name) ko target Email Template mein 
fill karke automatic notifications push karta hai.
Real Project Example: Jaise hi system mein "Invoice Creator vs Approver" ka violation pakda jata hai, 
yeh method Compliance Officer ko ek automatic email alert bhej deta hai: "Warning: High Risk Policy Violation detected for 
User Amit during Access Request." 

()🔹 isMitigated:-
Detailed Definition: Kai baar business urgency ke chalte manager ya auditor policy violation wale access ko temporary bypass ya 
accept kar dete hain, jise hum Mitigation kehte hain. Yeh method boolean check return karta hai (true/false) ki kya approver ne 
us violation ke sath ek valid validation comment aur end-date dalkar ushe 'Mitigate' kiya hai ya nahi.

()clearViolationForm ek bohot hi important cleanup aur resource management method hai.
Muti-stage validation ya multi-level approvals ke dauran jab ek bar policy violations check aur review ho jaati hain, 
toh uske baad is method ka chalate hai.
Jab kisi user ki request mein koi Policy Violation(jaise SoD-Ek hi bande ko Invoice banana aur approve karna dono ka access milna) 
pakda jata hai, toh SailPoint manager ke samne ek specialized screen/form kholta hai. 
Manager wahan reason likhta hai ki "Emergency hai, isliye bypass kar raha hoon" aur checkbox tick karke submit kar deta hai.
Jaise hi manager submit button dabata hai, us violation form ka poora data aur backend metadata ek temporary memory cache (RAM) 
mein fasa reh jata hai.
clearViolationForm ek vacuum cleaner ki tarah kaam karta hai. Iska sirf ek kaam hai: Us temporary policy form ke data aur 
garbage values ko memory se permanently sweep (saaf) kar dena, 
taaki agle steps ke liye workflow bilkul fresh aur light-weight ho jaye.

ex:-
<Workflow name="Enterprise-Policy-Compliance-Pipeline" libraries="PolicyViolation">

  <Variable name="identityName" input="true" required="true"/>
  <Variable name="plan" input="true" required="true"/>
  <Variable name="violationObject"/>

  <Step name="Scan Governance Policies" action="call:checkPolicyViolations">
    <Arg name="identityName" value="ref:identityName"/>
    <Arg name="plan" value="ref:plan"/>
    <Return name="violation" value="violationObject"/>
    
    <Transition to="Trigger Compliance Alerts" condition="script:(violationObject != null)"/>
    <Transition to="Proceed To Standard Provisioning"/>
  </Step>

  <Step name="Trigger Compliance Alerts" action="call:sendViolationNotifications">
    <Arg name="violation" value="ref:violationObject"/>
    <Transition to="Present Risk Mitigation Screen"/>
  </Step>

  <Step name="Present Risk Mitigation Screen">
    <Approval mode="serial" owner="script:IdentityLibrary.getManager(identityName)">
      <Arg name="handler" value="sailpoint.api.Workflower"/>
      <Arg name="method" value="processApproval"/>
    </Approval>
    <Transition to="Evaluate Mitigation Status"/>
  </Step>

  <Step name="Evaluate Mitigation Status">
    <Transition to="Clear Policy Form Cache" condition="script:PolicyViolationLibrary.isMitigated(violationObject)"/>
    <Transition to="Terminate Request Due To Risk"/>
  </Step>

  <Step name="Clear Policy Form Cache" action="call:clearViolationForm">
    <Transition to="Proceed To Standard Provisioning"/>
  </Step>

  <Step name="Proceed To Standard Provisioning" icon="Provision">
    <Transition to="End"/>
  </Step>

  <Step name="Terminate Request Due To Risk" icon="Stop">
    <Transition to="End"/>
  </Step>

  <Step name="End"/>
</Workflow>

()Approval Library:-
SailPoint IdentityIQ (IIQ) mein Approval Library (sailpoint.workflow.ApprovalLibrary) ek aur specialized built-in helper class 
hai.Iska mukhya kaam workflow ke andar complex aur multi-level approvals ko generate karna, 
unke notification emails handle karna.Jab aap kisi standard workflow mein <Workflow libraries="Approval"> define karte hain, 
toh is library ke saare methods directly use karne ke liye available ho jaate hain.

()🔹 buildApprovals:-
Bada Definition: Yeh is library ka sab सबसे heavy-duty method hai. 
Jab koi access request (ya koi lifecycle event) multiple managers ya application owners ke approval ke liye jaani hoti hai,
toh un sabhi workitems ka alag alag aprovers hote hai,jo workitems requests ko approve ya reject karenge.toh ye workitems 
ko banata hai.

Real Project Example: Ek user ne ek sath 3 roles maange—AD Admin, SAP Viewer, aur Salesforce Manager. 
buildApprovals chalte hi teenon roles ke owners ko scan karega aur unke liye alag-alag approval items taiyar kar dega taaki 
sahi task sahi person ke paas pahuche.

()🔹 sendApprovalNotifications:-
Bada Definition: Jab buildApprovals dwara approval structure taiyar ho jata hai, 
toh approvers ko inform karne ke liye yeh method call hota hai.Yeh method dynamic details(Requester name, item details, tracking URL) 
ko uthata hai, use target Email Template (jaise LCM Request Pending Approval) 
mein inject karta hai, aur automatic email triggers push kar deta hai.
Real Project Example: Jaise hi manager ke inbox mein approval request land hoti hai, 
unhe ek official email notification chala jata hai: "Hi John, you have a pending approval request REQ001 for user Amit."

()🔹 sendApprovalReminders:-
Bada Definition: Yeh escalation mechanism ka part hai.Agar koi approval item kisi approver ke dashboard par pichle 3-4 dino se 
pending pada hai, toh yeh method background timer ke hit hote hi trigger hota hai.Yeh automatic un sabhi users ko filter karta 
hai jinhone action nahi liya hai, aur unhe reminder notifications bhejta hai,sath hi database mein reminder count update karta hai.

()🔹 delegateApproval:-
Bada Definition: Agar koi approver chhutti (Leave) par ja raha hai ya woh us request par decision lene ke liye sahi candidate 
nahi hai, toh woh request ko kisi aur ko delegate karta hai. Yeh method us dynamic delegation request ko backend par process 
karta hai. Yeh original approver ke inbox se task ko secure tarike se detach karta hai aur naye delegate user ke dashboard par 
attach kar deta hai, sath hi database audit logs maintain karta hai.

()clearApprovalForm :-
Iska sabse bada use case Multi-Level Approval Routing mein hota hai.
Maan lijiye Level 1 par Manager ke samne ek form aaya jisme usne likha "Approved for Q2 project". 
Agar aapne iske baad clearApprovalForm nahi chalaya, aur Level 2 par Application Owner ke samne bhi ek custom form khulna hai, 
toh kai baar cache clear na hone ki vajah se system confuse ho jata hai ya Level 1 ka data Level 2 ke fields mein pre-populate 
hokar conflicts create kar deta hai.Isliye, jaise hi ek approval level poora hota hai, 
transition se theek pehle clearApprovalForm chalakar memory state ko bilkul fresh aur clean kiya jata hai.
example:-
<Workflow name="Custom-Enterprise-Approval-Pipeline" libraries="Approval">

  <Step name="Generate Approval Matrix" action="call:buildApprovals">
    <Arg name="plan" value="ref:plan"/>
    <Return name="approvals" value="approvals"/>
    <Transition to="Trigger Email Notifications" condition="script:(approvals != null)"/>
    <Transition to="End Pipeline"/>
  </Step>

  <Step name="Trigger Email Notifications" action="call:sendApprovalNotifications">
    <Arg name="approvals" value="ref:approvals"/>
    <Transition to="Wait For Human Action"/>
  </Step>

  <Step name="Wait For Human Action">
    <Approval mode="serial" owner="ref:approvers">
      <Arg name="handler" value="sailpoint.api.Workflower"/>
      <Arg name="method" value="processApproval"/>
    </Approval>
    <Transition to="Cleanup Form State"/>
  </Step>

  <Step name="Cleanup Form State" action="call:clearApprovalForm">
    <Transition to="End Pipeline"/>
  </Step>

  <Step name="End Pipeline" icon="Stop"/>

</Workflow>

()Role Library:-
SailPoint IdentityIQ (IIQ) mein Role Library (sailpoint.workflow.RoleLibrary) ek aur specialized built-in business helper class 
hai,jise mukhya roop se roles ke lifecycle, unke approvals, mining, aur management operations ko automatic tarike se handle karne
ke liye design kiya gaya hai.

()🔹 getRole:-yeh method role ka naam ya ID lekar database se pure Bundle (Role) object ko memory mein fetch karke laata hai.

()🔹 isNewRole:-Bada Definition: Yeh method ek boolean check (true ya false) return karta hai. 
Jab koi role editing workflow shuru hota hai, toh yeh check karta hai ki kya yeh role abhi naya-naya database mein insert hone 
ja raha hai (Creation Phase), ya phir pehle se bane hue kisi role ke attributes/entitlements ko modify kiya ja raha hai 
(Modification Phase). Iske aadhar par workflow approval routes badal jate hain.

()🔹 saveRole:-
Bada Definition: Jab role creation ya update ka workflow poora ho jata hai aur saare required managers/owners 
use approve kar dete hain, tab yeh method call hota hai. Yeh memory mein chal rahe temporary role metadata aur 
entitlements configuration ko uthata hai aur use permanent roop se IdentityIQ ke spt_bundle database table 
mein save (commit) kar deta hai, jisse woh role pooray system mein provision hone ke liye active ho jata hai.

()🔹 getRoleApprovers:-
Bada Definition: Jab koi role create ya modify hota hai, 
toh har role ke upar ek dedicated Role Owner ya governance group mapped hota hai. 
Yeh method us role ke internal properties ko scan karta hai aur un sabhi users/identities ki list return karta hai jinke paas 
is role ko approve karne ka right hai. Workflow isi list ka use karke WorkItems bhejta hai.

()🔹 determineRoleApprovalScheme:-
Bada Definition: SailPoint mein roles ke approval ke kai tarike ho sakte hain 
(jaise: sirf role owner approve kare, ya application owner bhi approve kare, ya explicit manager approval chahiye). 
Yeh method system settings aur role type (Business Role, IT Role) ko read karke yeh tay karta hai ki is specific request ke liye
kaunsa approval routing pattern (Scheme) apply kiya jayega.

()🔹 launchRoleMining:-
Bada Definition: Agar aapne kisi workflow mein automation lagayi hai jahan har mahine automatic user access patterns ko check karke 
naye roles suggest karne hain, toh yeh method background mein Role Mining Task ko trigger kar deta hai.

()🔹 getMiningStatus:-
Bada Definition: Role mining tasks bohot heavy hote hain aur chalne mein ghanton le sakte hain. 
Yeh method workflow ko background execution ka current progress status (Percent complete, items processed, pending status)
batata rehta hai taaki workflow dashboard par loading progress sahi se dikhai ja sake.
ex:-
<Workflow name="Standard Role Approvals" libraries="Role">

  <!-- Step 1: Check if new -->
  <Step name="Check Role State" action="call:isNewRole">
    <Transition to="Determine Scheme" condition="script:(isNew)"/>
  </Step>

  <!-- Step 2: Decide who should approve -->
  <Step name="Determine Scheme" action="call:determineRoleApprovalScheme">
    <Return name="approvalScheme" value="approvalScheme"/>
    <Transition to="Get Approvers"/>
  </Step>

  <!-- Step 3: Get exact approver identities -->
  <Step name="Get Approvers" action="call:getRoleApprovers">
    <Return name="approvers" value="approvers"/>
    <Transition to="Commit Role" condition="script:(approved == true)"/>
  </Step>

  <!-- Step 4: Save to Database permanently -->
  <Step name="Commit Role" action="call:saveRole">
    <Transition to="End"/>
  </Step>

</Workflow>

()LCM Library:-SailPoint IdentityIQ (IIQ) mein LCM Library (sailpoint.workflow.LCMLibrary) ek aur bohot hi mahatvapurna built-in 
helper class hai, jo mukhya roop se Lifecycle Manager (LCM) workflows ke business logic ko chalane ke liye design ki gayi hai.

🔹 getIdentityRequest:-
Bada Definition: Jab koi access request submit hoti hai, toh IIQ backend mein ek object banata hai 
jise IdentityRequest kehte hain (jo database ke spt_identity_request table mein jata hai). 
Yeh method workflow ke execution ke dauran us tracking object ko fetch ya retrieve karta hai. 
Iska kaam workflow ko yeh batana hai ki current request ki global tracking ID kya hai taaki use audit ya report kiya ja sake.

🔹 updateIdentityRequest:-
Bada Definition: Yeh ek dynamic tracking status updater hai. 
Jab workflow aage badhta hai (jaise approval pending se approval complete hona, 
ya provisioning start hona), toh yeh method IdentityRequest table ke andar user ki request ka status badalta rehta hai 
(e.g., In Progress, Verifying, Completed, Terminated). Isse end-user ko unke "My Requests" dashboard par sahi status dikhta hai.

🔹 checkPolicyViolations:-
Jab ek manager cart mein 5 naye roles add karke "Submit" dabata hai, toh yeh method target identity ke maujuda access aur 
un naye 5 roles ko aapas mein mix karke scan karta hai. Agar unme koi SoD (Segregation of Duties) violation milti hai, 
toh yeh workflow ko rok deta hai aur ek violation object return karta hai jise approval screen par dikhaya jata hai.

🔹 optimizePlan:-
Bada Definition: Yeh ek "Smart Filtering" method hai. Maan lijiye kisi manager ne galti se ek aise role ke liye request submit
kar di jo us employee ke paas pehle se hi moojud hai. optimizePlan us pure ProvisioningPlan ko scan karega, 
duplicate items ya unnecessary accesses ko filter karke drop kar dega,aur sirf vahi access aage bhejega jo user ke paas sach mein
nahi hai. Isse target system par fuzool ka load nahi padta.

🔹 isPasswordRequest:-
Bada Definition: LCM single workflow handle karta hai access requests ke liye bhi aur password reset ke liye bhi. 
Yeh method boolean check return karta hai (true ya false) ki kya current request ek access request hai 
ya fir user apna password change/reset karne ki koshish kar raha hai. Iske decision par workflow ka rasta badal jata hai.

🔹 getAccounts:-
Bada Definition: Jab koi user account unlock ya password reset ki request karta hai, 
toh yeh method us identity se Jude sabhi target application links/accounts ki ek comprehensive list nikaal kar laata hai, 
taaki user UI par select kar sake ki use kis specific account ka password reset karna hai.

🔹 sendLCMNotification:-
Bada Definition: LCM workflow mein communications handle karne ka sabse bada method yahi hai. 
Yeh method dynamic variables (jaise requester name, approver name, item requested) ko uthata hai, 
use kisi predefined Email Template (jaise LCM Request Approved Notification) mein map karta hai, 
aur automatic email trigger kar deta hai.manager ko remind ya escalate karne ke email bhejne ke liye.

🔹 checkIdentityRequestStatus :-
Jab koi user ya manager LCM dashboard se koi Access Request ya Lifecycle event submit karta hai, 
toh IIQ backend mein ek central tracking object banata hai jise IdentityRequest (spt_identity_request table mein) kehte hain.
Aapki single request ke andar multiple applications ke alag-alag roles ya entitlements ho sakte hain. 
Is wajah se, ek main IdentityRequest ke andar kai saare Child Items (IdentityRequestItem) hote hain—jaise ek 
item Active Directory ke liye, dusra SAP ke liye, aur teesra Salesforce ke liye.

checkIdentityRequestStatus method ka kaam pooray workflow execution ke dauran is main IdentityRequest aur uske saare child items
ke statuses ko continuously scan, read aur evaluate karna hai.
Yeh method check karta hai ki:-
Kya saare child items ke approvals poore ho chuke hain?
Kya target system par unka actual provisioning task (ProvisioningProject ke through) khatam ho chuka hai?
Kya koi item abhi bhi pending approval ya background retry loop mein fasa hua hai?

🔹 Yeh kab aur kyun use hota hai?
Iska sabse bada use case Asynchronous Provisioning (ya Integration Tasks) mein hota hai.
Maan lijiye aapne kisi aisi application ka access maanga jahan automatic account nahi banta, 
balki IIQ background mein ek Service Desk Ticket (jaise ServiceNow ticket) create kar deta hai. 
Ab workflow ko tab tak ruka rehna padega jab tak ServiceNow par ticket close nahi ho jati.
Aise samay par, workflow loop mein checkIdentityRequestStatus ko call kiya jata hai. 
Yeh baar-baar (wait step ke baad) check karta hai ki kya external ticket complete hui ya nahi. 
Jaise hi ise dikhta hai ki saare child items ka status Approved aur Provisioned ho chuka hai, 
yeh pooray request status ko Completed mark karne ka signal de deta hai aur workflow gracefully end ho jata hai.
ex:-

XML

(1). Standard Workflow Handler kya hai kitne methods hai?
-->Standard Business Logic:-👉 Application ka normal rules + kaam karne ka tareeka.
Ye decide karta hai ki software ko kya kaam karna chahiye aur kis rule ke according karna chahiye.

Example 1: Banking App:-
Business Logic:-
Balance check karo
Agar account me ₹5000 hai aur user ₹2000 withdraw kare
→ Allow karo
Agar account me ₹1000 hai aur user ₹2000 withdraw kare
→ Reject karo (Insufficient Balance)
Ye jo decision rules hain, ye business logic hai.

Example 2: E-commerce Website:-
Business Logic:-
Order amount ₹1000+ ho → 10% discount
Stock available ho → order place karo
Stock nahi ho → Out of Stock dikhao
Ye sab rules = Business Logic.

(2)Standard Business Logic ka matlab kya?
Standard = Normal/Default/Common rules
Example:-
Company ke common rules:-
Employee active ho tabhi login
Salary > 50000 ho toh tax calculate karo
Manager hi approval de sakta hai
Ye company ka standard business logic ho sakta hai.

()Standard Business Logic SailPoint IIQ me Example:-
IIQ me Standard Business Logic:-
Joiner → account create
Mover → entitlement update
Leaver → account disable/delete
Approval required ho toh workflow approval bhejo

(3)Custom Business Logic:-
Agar company apna custom rule add kare:-
If department = IT
then assign VPN access
Ye custom business logic ho gaya.

Short me:-
Business Logic = Software ke kaam karne ke rules
Standard Business Logic = Default/common predefined rules
Custom Business Logic = Company ke extra/custom rules.

()Standard Workflow Handler kya hai?
Standard Workflow Handler (sailpoint.api.Workflower):-
Jab aap kisi workflow mein standard business logic run karna chahte hain (jaise approvals bhejna, 
identity attributes check karna, ya provisioning steps handle karna),
toh aap BeanShell code likhne ke bajay is handler ke predefined methods ko call karte hain. 
Yeh kaafi efficient aur fast hota hai.

Example Syntax:-
XML
<Step name="Process Approval Step">
  <Approval mode="serial" owner="ref:manager">
    <Arg name="handler" value="sailpoint.api.Workflower"/>
    <Arg name="method" value="processApproval"/>
  </Approval>
  <Transition to="Provisioning Step"/>
</Step>

()Standard Workflow Handler ke Key Methods:-
Is handler mein kai saare useful methods hote hain jo workflow ke alag-alag stages par kaam aate hain.

-->
()checkApproval:-
Kaam: 🔴 checkApproval
Bada Definition: Jab aapka workflow kisi Approval step par pahunchta hai, toh yeh method sabse pehle background mein chalta hai. 
Yeh check karta hai ki kya us request ke liye sach mein kisi ka approval chahiye, ya phir use auto-approve kiya ja sakta hai. 
Yeh workflow ke variables, rule conditions, aur configuration settings (jaise LCM Electronic Signatures) ko read karta hai. 
Agar isko lagta hai ki approver list khali hai ya condition aisi hai jisme approval bypassed hai, 
toh yeh bina kisi ko notification bheje step ko Approved mark karke aage badha deta hai.
“Bypass kar deta hai” matlab:-
👉 Normal process / rule ko skip kar dena ya uske bina direct nikal jana.

()🔴 processApproval:-
processApproval SailPoint IIQ Workflow me bahut important utility/helper method hai.
Iska kaam:-
Jab workflow kisi <Approval> step par rukta hai, toh IdentityIQ database ke spt_work_item table mein ek naya record generate karta hai jise WorkItem (Approval Task) kehte hain. Yeh task approver ke dashboard (Inbox) par dikhai deta hai.
Jab approver apne dashboard par jaakar "Approve", "Reject", ya "Forward" button par click karta hai, toh UI se aane wala yeh raw action seedhe processApproval method ke paas land hota hai.
Backend par yeh method yeh 5 bade kaam karta hai:

-->Decision Capture: Yeh check karta hai ki approver ne kis button par click kiya hai (Approve ya Reject) 
aur us decision ko memory mein map karta hai.

-->Comment Logging: Agar approver ne form par koi comment likha hai (e.g., "Approved for Q2 Auditing"), 
toh yeh us text ko WorkItem ke transaction history log mein permanently write kar deta hai.

-->Digital Signatures (E-Sign): Agar aapke project mein roles ke liye Electronic Signatures enabled hain, 
toh yeh method user ke credentials aur signature token ko verify karta hai.

-->WorkItem Closure: Jaise hi action valid ho jata hai, yeh database mein jaakar us unique WorkItem ka status Pending se badalkar 
Closed (ya Finished) mark kar deta hai taaki approver ke inbox se task gayab ho jaye.

-->State Advancement: Yeh workflow engine ko signal deta hai ki ab is approval par action liya ja chuka hai, 
isliye ab conditional transitions (isApproved) ko evaluate karke aage badha jaye.

()🔴 isApproved:-
Bada Definition: Yeh ek logical decision-making utility method hai jo workflow ke transitions mein kaam aata hai. 
Yeh current approval step ki final state ko inspect karta hai. Agar approval step poora ho chuka hai aur sabhi required approvers
ne use 'Approve' kar diya hai, toh yeh true return karta hai. Agar kisi ek ne bhi reject kiya ho ya step pending ho, 
toh yeh false return karta hai. Iska use standard workflows mein conditional checking ke liye kiya jata hai 
(e.g., <Transition expr="script:isApproved" to="Provisioning"/>).

Agar false

ya script se:-

return isApproved;

3. to="Provisioning":-
Condition true hui to workflow Provisioning step me chala jayega.
Current Step

approve:-
Kaam: Kisi WorkItem (Approval item) ko programmatically 'Approved' state mein mark karne ke liye.
reject:-
Kaam: Kisi WorkItem ko programmatically 'Rejected' state mein mark karne ke liye.
🔴 approve aur reject:-
Yeh dono methods programmatically (yani bina kisi human user ke button dabaye) approval state ko change karne ke kaam aate hain.
Agar aapne workflow mein koi aisa logic lagaya hai jahan specific conditions ke under 
(jaise agar requester khud manager hai, ya request ek safe trusted software se aa rahi hai) 
approval ko automatically system level par hi accept ya decline karna ho, 
toh yeh methods direct pooray WorkItem ka status Approved ya Rejected set kar dete hain.

🔴 remind:-
Bada Definition: Yeh escalation ka hissa hai. Agar koi approval item kisi manager ke dashboard par kaafi dino se pending pada hai
aur uski deadline cross ho rahi hai, toh remind method trigger hota hai. Yeh automatic email notifications dobara bhejta hai, 
pending state ko highlight karta hai, aur workflow logs mein reminder ka count entry karta hai taaki reporting mein pata chal sake
ki kis approver ne kitna delay kiya.

🔴 forward:-
Bada Definition: Jab koi approver chhutti par ja raha ho ya woh us request par decision lene ke liye sahi vyakti na ho, 
toh woh us request ko kisi aur ko delegate/forward karta hai. Yeh method us dynamic delegation ko handle karta hai. 
Yeh purane approver ke inbox se task ko hatata hai, naye approver ke inbox mein assign karta hai, 
aur database ke IdentityRequest table mein ownership updates karta hai, 
jisse traceability bani rahe ki task originially kiske paas tha aur ab kiske paas hai.

2. Policy aur Validation Methods:-
Yeh methods compliance, security aur governance rules ko enforce karne ke liye hote hain.
🔵 checkPolicyViolations:-
Bada Definition: Provisioning (access dene) se theek pehle yeh method IdentityIQ ke internal Policy Engine ko call karta hai. 
Yeh un sabhi active policies (jaise Separation of Duties - SoD, Account Policies) ko check karta hai jo us request se trigger ho
sakti hain. Udaharan ke liye, agar koi user 'Invoice Creator' aur 'Invoice Approver' dono roles ek saath maang raha hai, 
toh yeh method us violation ko pakad lega, aur workflow variables mein violation ki poori details 
(Policy Name, Violating Roles) ko save kar dega taaki manager ko screen par alert dikhaya ja sake.
 
🔵 checkPreDelegation:-
Bada Definition: Kisi approval item ko kisi aur ke naam par auto-delegate karne se pehle yeh method security validation run karta
hai. Yeh check karta hai ki kya IdentityIQ mein active delegation rules set hain, kya delegate hone wala banda active status mein
hai, aur kya requester ke paas delegation configure karne ke sahi rights hain. 
Yeh workflow ko unauthorized ya galat delegation loops mein phasne se rokta hai.

3. Data aur State Management Methods:-
🟢 assimilateOutputs:-
Bada Definition: Jab ek workflow kisi sub-process (child workflow) ko chalata hai ya kisi dynamic form se data collect karta hai,
toh woh saara data ek isolated temporary space mein hota hai.assimilateOutputs ka kaam us scattered data ko collect karna aur 
main parent workflow ke global variables ke saath merge karna hai. Agar child process mein koi naya variable generate hua hai ya
provisioning ka result status aaya hai, toh yeh method use main workflow map mein sahi key-value pair ke roop mein inject kar 
deta hai taaki aage ke steps use kar sakein.

🟢 assimilateForm:-
Bada Definition: Jab ek workflow kisi user ke saamne ek custom UI Form display karta hai 
(jaise naye employee ki details bharne ke liye form), toh user ke submit karne ke baad yeh method chalte hain. 
Yeh form ke har ek field (TextBox, Dropdown, Checkbox) se user ki entered values ko nikaalta hai, 
unhe validate karta hai, aur workflow ke decision context ke andar variable bana kar store kar deta hai, 
taaki un values ko aage provisioning plan mein use kiya ja sake.

🟢 archive:-
Bada Definition: Jab workflow ka poora kaam khatam hone wala hota hai, 
toh saara transaction data memory se clear hone lagta hai. 
archive method us khatam hone wale workflow ka ek historical record (audit trail) bana kar IdentityIQ Archive Tables mein
transfer karta hai. Isme workflow kab shuru hua, kisne approve kiya, kya attributes badle, aur kab khatam hua, 
is sabki permanent history safe ho jaati hai jise auditors baad mein Access History Reports mein dekh sakte hain.

()4. Execution aur Sub-process Control Methods:-
🟡 launchSubprocess:-
Bada Definition:Bade workflows (jaise Joiner or Lifecycle Lifecycle workflows) bohot complex hote hain. 
Unhe saaf rakhne ke liye chote chote sub-workflows banaye jaate hain (e.g., Active Directory provisioning ka alag workflow, 
SAP ka alag).launchSubprocess method main workflow se un chote child workflows ko trigger karta hai. 
Yeh unhe main workflow ka kuch data pass karta hai(input parameters) aur engine ko bolta hai ki is child process ko alag se execute karo.

🟡 wait:-
Bada Definition: Kai baar workflow ko kisi external event ka intezar karna padta hai—jaise kisi dusre system se flat file aane ka
,ya target application par account generate hone ka (Asynchronous Provisioning).wait method workflow ke pure execution thread ko
pause (sleep mode) mein dal deta hai aur background mein ek listener task open rakhta hai. 
Jab tak woh external dependency ya condition poori nahi hoti, tab tak yeh step agle step par transition nahi hone deta, 
jisse resources waste nahi hote.

🟡 end:-
Bada Definition: Yeh workflow ka final execution handler hai.Jab workflow ke saare tasks, approvals, 
aur provisioning steps safaltapoorvak khatam ho jaate hain,toh yeh method call hota hai. 
Yeh pooray workflow instance ko Completed status deta hai, locks ko release karta hai, 
temporary memory allocations ko wipe out karta hai,aur task scheduler ko inform karta hai ki yeh task ab thread pool se hataya ja sakta hai.

example:-
<Workflow name="Standard Workflower Execution" Sequential="true">

  <Step name="Process Decision">

  </Step>

    <Action name="mergeData" handler="sailpoint.api.Workflower" method="assimilateOutputs">
      <Arg name="src" value="ref:approvalFormOutputs"/>
    </Action>
    <Transition to="Trigger Provisioning"/>
  </Step>

  <Step name="Trigger Provisioning">
    <Transition to="Wrap Up Execution"/>
  </Step>

  <Step name="Handle Rejection Layer">
    <Transition to="Wrap Up Execution"/>
  </Step>

  <Step name="Wrap Up Execution" handler="sailpoint.api.Workflower" method="end">
    <Transition to="End"/>
  </Step>

</Workflow>

()Sequential:-Iska kaam workflow engine ko yeh batana hai ki "Is workflow ke andar jitne bhi <Step> hain, 
unhe ek ke baad ek (line by line) hi execute karna hai." Agar aap Sequential="true" nahi lagate hain, 
toh SailPoint default roop se explicit transitions (<Transition to="..."/>) par depend karta hai. 
Lekin jab yeh true hota hai, toh agar aap kisi step ke aakhir mein <Transition> likhna bhool bhi jayein, 
toh engine automatic uske theek neeche wale agle step par jump kar jata hai.

()<Utility>:-
Aamtaur par, jab aap workflow mein koi kaam karna chahte hain, toh aap <Action> tag ka use karke kisi Java class ya Rule ko 
call karte hain. Lekin <Utility> tag kisi Java class ko call nahi karta; yeh seedhe SailPoint ke Core Workflow Engine 
(Orchestration Engine) ko direct instructions ya commands deta hai.
Iska mukhya kaam workflow ke execution flow ko control karna, threading manage karna, 
debugging karna aur background tasks ko handle karna hai.

--<Utility> tag ke andar mukhyata 4 sabse mahatvapurna actions/arguments chalte hain:-
-->1. wait:-
Bada Definition: Yeh current workflow execution thread ke pipeline ko ek specific duration ke liye rok (pause/sleep) deta hai. 
Yeh memory background scheduler mein request register karke thread ko block hone se bachata hai.
Real Project Example: Jab aap kisi external ticketing system (jaise ServiceNow) se account creation ka confirmation aane ka wait
kar rahe hote hain, toh aap <Utility action="wait" argument="60"/> likhte hain. 
Isse workflow agle 60 seconds tak shaant baith jata hai aur server resources bache rehte hain.

-->2. audit:-
Real Project Example: Kisi high-risk production application (jaise Core Banking Application) ka access kisi user ko provision ho
 gaya. Aap chahte hain ki yeh event permanently system mein capture ho jaye, 
 toh aap <Utility action="audit" argument="High Risk Access Granted"/> chalate hain.

-->3.1. clearErrors;-
Deep Definition: Yeh method workflow instance ke transient execution context se saare accumulated error messages ko permanently 
wipe-out (mita) deta hai. Yeh actual database audit log ko nahi chhedta, balki current workflow runtime memory ko clean karta hai.
Real Project Example (Retry Loop):Maan lijiye Active Directory provisioning ke waqt network down hone se Target system unreachable 
ka error aaya. Workflow ne use detect kiya, 10 minute wait kiya, aur ab dobara try karne ja raha hai. 
Dobara try karne se theek pehle, developer <Utility action="clearErrors"/> chalate hain taaki workflow ka slate bilkul saaf ho jaye 
aur purana failure naye execution ko contaminate na kare.

-->3.2. clearWarnings:-
Deep Definition: Bilkul clearErrors ki tarah, yeh method workflow memory se saari purani warnings 
(jaise non-fatal connection dropouts ya policy evaluation alerts) ko remove kar deta hai.
Real Project Example: Ek user ko access dete waqt ek mild policy violation warning aayi thi. 
Manager ne use 'Mitigate' (bypass) kar diya. 
Ab provisioning phase mein jaane se pehle,workflow <Utility action="clearWarnings"/> chala deta hai taaki step validation bilkul
clear pass ho sake.

-->4. pp (Pretty Print):-
Bada Definition: Yeh ek debugging utility hai. Yeh complex Java Objects, Maps, 
ya Context lists ko readable JSON/String format mein internal logs (iiq.log ya stdout) mein print kar deta hai.
Real Project Example: Rule ya Workflow testing ke dauran agar aapko dekhna hai ki ProvisioningPlan object ke andar kya exact 
keys aur values pass ho rahi hain, toh aap pp utility ka use karte hain.

()<Action> vs <Utility> vs <Script>:-

()<Script>:-
Custom business logic chalata hai BeanShell compiler ke zariye. 
Fast development ke liye sahi hai par upgraded maintenance heavy hoti hai.	
BeanShell Scripting me code likhata hai.

()<Action>:-Predefined Java Class code ya ready-made workflow libraries (Identity, LCM) ke methods ko trigger karta hai.

()<Utility>:-
Aamtaur par, jab aap workflow mein koi kaam karna chahte hain, toh aap <Action> tag ka use karke kisi Java class ya Rule ko 
call karte hain. Lekin <Utility> tag kisi Java class ko call nahi karta; yeh seedhe SailPoint ke Core Workflow Engine 
(Orchestration Engine) ko direct instructions ya commands deta hai.
Iska mukhya kaam workflow ke execution flow ko control karna, threading manage karna, 
debugging karna aur background tasks ko handle karna hai.

ex;-

  <Transition to="Recheck Provisioning Status"/>
</Step>

(2). Identity Library:-
SailPoint IdentityIQ (IIQ) mein Identity Library ek ready-made Java helper class (sailpoint.workflow.IdentityLibrary) hai.
Yeh class workflow ke andar user profiles (Identities), unke accounts, automatic provisioning forms, policies aur lifecycle actions 
ko manage karne ke liye dher saare built-in methods deti hai.
Jab aap workflow ke <Workflow> tag mein libraries="Identity" likhte hain (ya kuch na likhein toh bhi yeh default enabled hoti hai), 
toh iske saare methods aapke workflow ke liye available ho jaate hain.

🔹 getManager:-
Definition: Yeh method kisi specific identity ke manager ka naam pata karne ke liye use hota hai. 
Jab aap ise target identityName pass karte hain,toh yeh database se us user ke profile ko read karta hai 
aur uske mapped manager ka string name return kar deta hai.

🔹 refreshIdentity aur refreshIdentities:-
Definition: refreshIdentity ek single user par aur refreshIdentities ek poore group/set 
(jo CSV list, filter string, ya roles ke aadhar par ho sakte hain) par Identity Refresh Task run karta hai. 
Yeh user ke links aggregate karta hai, dynamic attributes recalculate karta hai, aur entitlements sync karta hai.
Kab use karein: Jab workflow ke dauran kisi user ka department ya role badle aur aap chahte hain ki bina subah tak ka wait kiye 
uski profile turant update ho jaye.

🔹 compileProvisioningProject:-
Definition: Jab workflow ko ek ProvisioningPlan (jisme likha hota hai ki kis account ko kya access dena hai) milta hai, 
toh yeh method us plan ko ek ProvisioningProject mein badal (compile) deta hai. 
Compilation ke dauran yeh system ke saare Provisioning Policies, Joiner/Mover rules aur Policy violations ko check karke plan ko 
concrete action steps mein convert karta hai.

🔹 provisionProject:-
Definition: Compile hone ke baad execution ka asli kaam isi ka hai. 
Yeh method ProvisioningProject ke actions ko directly IIQ Connector Architecture ke paas bhejta hai taaki actual target applications 
(jaise Active Directory, SAP, Database) mein account create, modify ya delete ho sakein.

🔹 checkProvisioningStatus:-
Definition: Asynchronous provisioning ke mamle mein (jahan target system turant access nahi deta balki ticket create karta hai), 
yeh method target connector se baar-baar status poochta (poll karta) hai ki kaam poora hua ya nahi.

🔹 getIdentityModel:-
Definition: Yeh method pure Identity object ke attributes (first name, last name, assigned roles, accounts) 
ko ek asaan Map data structure mein badal deta hai jise Form Model kehte hain.
Kab use karein: Jab workflow mein user ke samne ek bada form dikhana ho jisme uski purani details pehle se bhari hui 
(pre-populated) hon.

🔹 assimilatePlanApprovalForm:-
Definition: Jab koi user ya manager buildProvisioningForm dwara banaye gaye form ko fill karke 'Submit' karta hai, 
toh yeh method us form ke data ko vapas nikal kar ProvisioningPlan ke andar inject kar deta hai taaki galat data provision na ho.

🔹 buildEventPlan:-
Definition: Lifecycle events ke dauran (jaise Leaver/Termination event),
yeh method user ke paas jitne bhi connected accounts (links) hain, un sabhi ko scan karta hai aur ek master plan taiyar karta hai

🔹 processApprovalDecisions:-
Deep Definition: Jab ek complex request mein multiple applications ke access shamil hote hain 
(e.g., User wants access to AD, SAP, and Salesforce) aur approvals alag-alag managers ke paas jaate hain, 
tab yeh method picture mein aata hai. Maan lijiye, Manager A ne AD ka access approve kiya,

rejected items ko main ProvisioningPlan se hamesha ke liye strip-out (hata) deta hai, 
aur approved items ko filter karke ek naya clean plan taiyar karta hai taaki actual engine sirf vahi provision kare jo officially
approved hai.

🔹 getProvisioningRetryThreshold:-
Deep Definition: Yeh ek configuration reader method hai. Jab koi provisioning request "Network Connection Timed Out" jaisa error
deti hai, toh yeh method system setting se check karta hai ki do retries ke beech mein IIQ engine ko kitna wait karna chahiye 
(e.g., Default configuration ke mutabik 60 minutes). Yeh workflow variable ko interval time supply karta hai taaki thread fauran 
retry karke target system par load na badhaye.

🔹 getProvisioningMaxRetries:-
Deep Definition: Yeh method yeh check karta hai ki kisi failed provisioning plan ko maximum kitni baar retry karne ki permission 
di gayi hai (jaise max 3 ya 5 baar). Agar system continuous fail ho raha hai aur threshold limits exhaust ho chuki hain, 
toh yeh workflow ko anant kaal (infinite loop) mein phasne se rokne ke liye ek hard stop limit return karta hai.

🔹 getProvisioningMaxStatusChecks:-
Deep Definition: Kuch target engines asynchronous kaam karte hain (yani woh request le lete hain aur ek ticket ID de dete hain).
Yeh method check karta hai ki IIQ ticket status ko kitni baar maximum refresh/poll karega. Agar di gayi limit 
(e.g., 10 status checks) cross ho jati hai, toh yeh use 'Timeout' mark karne ka signal deta hai.

1. assembleRetryProject (Isolating the Failures):-
Backend Mechanism:Jab provisioning fail hoti hai,toh original ProvisioningProject object ke andar kuch accounts/items/subSet
Success hote hain aur kuch Retry state mein chale jate hain failed hone ke karan.
Yeh method pure project ko scan karta hai aur safe/successful accounts/items/subSet ko vahin chhod deta hai.
Yeh sirf un application/items/subSet requests ka ek alag slice ya subset nikaalta hai jo pending ya failed hain, 
aur unhe ek naye isolated temporary project object mein wrap kar deta hai.
Workflow Variable: Standard LCM workflows mein is naye isolated project ko retryProject variable ke andar store kiya jata hai.

2. retryProvisionProject (The Actual Execution):-
Backend Mechanism: Yeh method assembleRetryProject dwara banaye gaye retryProject ko input ki tarah leta hai. 
Yeh workflow thread ko wahin par block karne ke bajay backend mein ek Request object(SailPoint background task) register kar deta hai. 
Yeh method getProvisioningRetryThreshold (e.g., 60 minutes) ka wait karta hai aur system ko bolta hai:"Thodi der baad is target 
application par dobara request hit karo, shayad tab tak unka server up ho jaye.
aur bacha hua item/subset ko retryProject ke through failed provissioning item/subset provision ho jaye.

3. mergeRetryProjectResults (Syncing Back to Reality):-
Backend Mechanism: Jab background mein retry attempt khatam ho jata hai

aur workflow gracefully end step par ja sakta hai.

      <![CDATA[
        import sailpoint.object.ApprovalSet;
        import sailpoint.object.ApprovalItem;
        import java.util.Map;
        import java.util.HashMap;

        ApprovalSet masterSet = (ApprovalSet) workflow.get("approvalSet");
        Map splitMap = new HashMap();

        if (masterSet != null && !masterSet.isEmpty()) {
            for (ApprovalItem item : masterSet.getItems()) {
                // Maan lijiye humne har application ka ek dedicated owner define kiya hua hai
                String ownerName = "spadmin"; 
                if ("Active Directory".equals(item.getApplication())) {
                    ownerName = "adManager";
                } else if ("SAP".equals(item.getApplication())) {
                    ownerName = "sapOwner";
                }

                // Agar is owner ka pehle se koi bucket nahi bana toh naya ApprovalSet banao
                ApprovalSet subSet = (ApprovalSet) splitMap.get(ownerName);
                if (subSet == null) {
                    subSet = new ApprovalSet();
                    splitMap.put(ownerName, subSet);
                }
                
                // Item ko uske sahi owner ke bucket me daal do
                subSet.add(new ApprovalItem(item));
            }
        }
        return splitMap;
      ]]>
    </Source>
  </Script>

()action="checkProvisioningStatus":-
-->
Yeh step kyu chahiye hota hai? (The Real Reason)
SailPoint me jab provisioning hoti hai, toh do tarike ke systems hote hain:-
Synchronous (Turant kaam hone wale): Jaise hi SailPoint ne Active Directory ko request bheji, 
usne 1 second me account bana diya aur bol diya "Success".

Asynchronous (Time lene wale):-
Jaise koi system down hai, toh SailPoint us request ko Retry Queue (spt_provisioning_transaction) me daal deta hai
ki "bhai 1 ghante baad fir koshish karenge".Ya fir SailPoint ne ServiceNow me ek ticket open kar diya ki 
"jab wahan ka admin physically account banayega, tab kaam poora hoga".

<Step name="Check Provisioning Status" action="checkProvisioningStatus" resultVariable="provisioningStatus">
  <Description>

  </Description>

  <Arg name="project" value="ref:project"/> <!-- Current dynamic ProvisioningProject reference -->
  
  <!-- TRANSITIONS: Status ke basis par decision lena -->
  <Transition to="Wait Step" when="script:\"pending\".equals(provisioningStatus);"/>
  <Transition to="Refresh Identity" when="script:\"success\".equals(provisioningStatus) || \"failure\".equals(provisioningStatus);"/>
</Step>

<Step name="Wait Step" wait="60">
  <Description>Agar transaction pending hai, toh 60 seconds tak thread ko pause rakho aur fir wapas check karo.</Description>
  <Transition to="Check Provisioning Status"/>
</Step>

ex:-
import sailpoint.api.Provisioner;
import sailpoint.object.ProvisioningProject;

    Provisioner provisioner = new Provisioner(context);
    
    // Engine se current execution status string nikalna
    // Expected Outputs: "success", "failure", "pending"
    String status = provisioner.checkProvisioningStatus(myProject);
    
    log.info("Current Background Provisioning Status: " + status);
    
    if ("pending".equals(status)) {
        // Project object ke andar ke dynamic provisioning transactions ko inspect karna
        boolean isAsynchronous = myProject.hasAsynchronousAssembling();
        log.warn("Some items are stuck in Retry Queue or Service Desk. Async Status: " + isAsynchronous);
    }
    
    workflow.setVariable("provisioningStatus", status);
}

Q1. WorkItem ke Finished hote hi kya hota hai?
Jaise hi manager UI par submit dabata hai, background me ye sequential operations trigger hote hain:-

spt_work_item Update: Jo temporary row database ki spt_work_item table me pending padi thi, 
uska status badal kar Finished ya Closed ho jata hai. Usme approver ka decision aur comments permanent save ho jate hain.

Workflow Awaken (Neend se jagna): Jo main Workflow execution thread abhi tak Suspended (Hold) state me database me lock tha

--step:2:-

Yeh pata lagane ka kaam sailpoint.workflow.IdentityRequestLibrary class ke backend methods karte hain, 
aur ise workflow me action="processApprovalDecisions" step ke throw chalaya jata hai.

Manager ka actual decision ApprovalSet ke andar ke har ek ApprovalItem ke state field
(jaise State="Approved" ya State="Rejected") me band hota hai.

processApprovalDecisions action is ApprovalSet par loop chalata hai aur check karta hai ki manager ne kis item ke samne green 
tick (Approve) mara hai aur kiske samne red cross (Reject).

--step:3:-

Yahan par ek chhota sa technical correction hai bhai: WorkItem ko ProvisioningPlan me add nahi kiya jata, 
balki ProvisioningPlan pehle se hi bana hota hai,aur manager ke decision ke mutabik us plan me se chizen KAATI(Filter ki) jaati hain.

Aur ye kaam bhi processApprovalDecisions hi karta hai!

Jab tumne cart submit kiya tha, tabhi ek Master ProvisioningPlan ban gaya tha (Maan lo: AD account + SAP account).

toh Yeh step turant us ProvisioningPlan ke andar se AD waale AccountRequest block ko permanently delete (remove) kar dega.

()action="assembleRetryProject":-
1. Fasi hui request ko database se nikalta hai
Jab kisi system (jaise AD) ke down hone par SailPoint request ko beech me hi rok kar database ki ek temporary table 
(spt_provisioning_transaction) me save kar deta hai, toh yeh step us fasi hui request ke pure data ko database se wapas khinch 
kar nikalta hai.

2. Request ko dubara chalne layak banata hai
Purani request par jo "Failed" ya "Retry" ka thappa (flag) laga hota hai, 
yeh step use saaf kar deta hai. Uske status ko badal kar wapas "Ready" kar deta hai aur ek fresh ProvisioningProject object 
taiyar karta hai, taaki agla step use bina kisi rukawat ke dubara target system par bhej sake.
assembleRetryProject ka kaam execution karna bilkul nahi hai. 
Iska kaam sirf aur sirf project ko execute hone ke liye taiyar (ready) karna hai.
ex:-
Yeh step kyu chahiye hota hai? (The Core Problem)
Maan lo tumne kisi user ko Active Directory (AD) ka access diya. provisionProject ne API hit maari, 
lekin us waqt tumhara AD domain controller temporary down (offline) tha.

SailPoint engine ne kya kiya? Usne workflow ko crash nahi kiya. 
Usne us request ko Retry Queue (spt_provisioning_transaction table) mein save kar diya ki "bhai isko baad mein chalayenge."

Ab maan lo 1 ghante baad AD server wapas online aa gaya, 
aur background timer task ne workflow ko wapas neend se jagaya (resume kiya). 
Ab engine ke samne do badi chunautiyan (challenges) hain:-
Jo request 1 ghante pehle fail hui thi, uske purane details (transaction state) ko database se nikalna.
Us 1 ghante ke andar agar identity ke andar koi aur naya badlav hua hai, 
toh use purani request ke sath sahi tarike se merge karke project banana.

ex:-
<!-- Agar checkProvisioningStatus ne bola ki "pending" ya retry chahiye -->
<Step name="Wait for Retry" wait="3600">
  <Description>1 ghante tak wait karo taaki target system online aa jaye</Description>
  <Transition to="Assemble Retry Project"/>
</Step>

<!-- TRIGER POINT: Purane project aur queue data ko re-assemble karna -->
<Step name="Assemble Retry Project" action="assembleRetryProject" resultVariable="project">
  <Description>
    Retry queue records ko parse karke execution project ko fir se live banana.
  </Description>
  <Arg name="project" value="ref:project"/>
  <Transition to="Retry Provisioning Execution"/>
</Step>

<!-- Dobara connector ko hit maarna -->
<Step name="Retry Provisioning Execution" action="provision" resultVariable="project">
  <Arg name="project" value="ref:project"/>
  <Transition to="Check Provisioning Status"/>
</Step>

()action="updatePasswordHistory":-

1. project.isSuccess() Asal me Kya Decision Leta hai?

()action=buildApprovalSetFromNativeChanges:-
iska kaam tab shuru hota hai jab SailPoint kisi target system (jaise Active Directory, SAP, ya koi HR system) me bina
SailPoint ke permission ke direct (native) badlav dekhta hai, aur un badlavon ko approve karwane ke liye ek set taiyar karta hai.
Native Changes Track Karna: Yeh check karta hai ki aapke sailpoint ad/sap application account me native taur par kya-kya metadata 
changes (jaise fields, validation rules, apex classes) hue hain.
Approval Set Banana: Un sabhi track kiye gaye changes ko ek jagah ik इकट्ठा (aggregate) karta hai aur ek Approval Set 
ya change bundle taiyar karta hai.
Compliance & Review: Is set ko aage manager ya QA ke paas approval ke liye bheja jaata hai taaki bina permission ke koi bhi 
change live production me na chala jaaye.
Automated DevOps Pipelines: Jab aap chahte hain ki jaise hi koi developer sandbox me kaam khatam kare, 
system apne aap changes ko detect karke pipeline me approval ke liye daal de.
Audit Trail: Company ke audit aur compliance ke liye yeh record rakhna ki native changes ko kisne approve kiya tha.

1. Yeh Kab Trigger Hota Hai?
Is action ka main kaam tab hota hai jab "Native Changes" (jise hum Out-of-Band changes bhi kehte hain) detect hote hain.
Example: Maan lijiye kisi Administrator ne direct Active Directory (AD) me jaakar kisi user ko ek sensitive Admin Group me daal diya (bina SailPoint ke through request kiye).
Jab SailPoint agla Aggregation Run (sync) karega, toh use pata chalega ki AD me yeh change natively hua hai.

2. Yeh Action Karta Kya Hai?
Jab SailPoint ko in native changes ka pata chalta hai, toh background me buildApprovalSetFromNativeChanges action call hota hai. Yeh niche diye gaye kaam karta hai:
Changes ko Identify Karna: Yeh check karta hai ki native change kya tha (e.g., "User X ko Group Y me add kiya gaya").
Approval Set Banana: Yeh un saare changes ko ek Approval Set Object me convert karta hai.
Approval Workflow Shuru Karna: SailPoint is approval set ko directly manager ya security team ke paas bhejta hai yeh poochne ke liye: "Yeh change bahar se kiya gaya hai, kya aap isse approve karte hain?"

3. Iske Baad Kya Hota hai? (The Outcome)
Is action ke approval set banane ke baad do hi raste hote hain:-
Approved: Agar manager ne approve kar diya, toh SailPoint apne database me us change ko accept kar leta hai (Identity Cube update ho jata hai).
Rejected (Remediation): Agar manager ne reject kar diya, toh SailPoint ek Reversion/Remediation event trigger karta hai aur us user ko us group se wapas nikal deta hai taaki security bani rahe.

()action="processNativeChangesApprovalDecisions":-
processNativeChangesApprovalDecisions SailPoint IdentityIQ (IIQ) ke Lifecycle aur Provisioning engine ka ek behad mahatvapurna internal method/action hai.

1. Yeh Action Karta Kya Hai? (The Final Execution)
Jaise hi manager apna decision submit karta hai, yeh method pooray ApprovalSet ko read karta hai aur har ek item par niche diye gaye actions leta hai:

Case 1: Agar Manager ne "Approve" kiya (Acceptance):
Agar manager ne kaha ki "Theek hai, mujhe manzoor hai," toh yeh action SailPoint ke database (Identity Cube) ko update kar deta 
hai. Yeh us change ko official maan leta hai taaki agli baar jab aggregation chale,
toh SailPoint isse "Violation" ya "Unidentified Change" na maane.
Approved milne par yeh target system me koi naya provision nahi karta.
Identity Cube Sync: Yeh SailPoint ke andar us user ke Identity Cube (database) ko update kar deta hai aur bolta hai: "Theek hai, yeh access ab official hai."
-->provision Kyun nahi karega? Kyunki woh access toh pehle se hi target system me maujood hai! Native change ka matlab hi yahi hota hai ki kisi ne SailPoint ke bahar se (natively) woh access user ko pehle hi de diya tha.

Case 2: Agar Manager ne "Reject" kiya (Remediation/Rollback):
Yeh iska sabse powerful kaam hai.Agar manager ne change ko reject kar diya(e.g., "Rahul ko AD ke Admin group me nahi hona chahiye tha"),
toh yeh action background me ek Provisioning Plan taiyar karta hai. 
Yeh target system Jab target system se access hat jata hai, toh SailPoint apne database (Identity Cube) se bhi us access ko remove kar deta hai taaki dono jagah data sahi ho jaye.(Active Directory) ko ek command bhejta hai aur us user ko us group se wapas nikal deta hai (deprovision kar deta hai) taaki security bani rahe.
