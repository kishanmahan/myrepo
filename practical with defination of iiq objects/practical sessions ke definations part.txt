()IT Roles kya hai?
--1️⃣ IT Roles kya hai?
Definition: Ye wo roles hote hain jo IT systems, networks, aur applications ke management ke liye diye jaate hain.
Purpose:-
System aur network ko manage karna
Security aur compliance maintain karna
Access aur permissions control karna
Ek line me:-
“IT Role = system aur technology access + responsibilities ka role”

2️⃣ Examples of IT Roles:-
IT Role	                          Responsibility	                   Permissions / Access
IT Admin------------------Servers, Network, System Config------Full admin access to servers & network devices
System Operator------------Daily operations of IT systems------Start/Stop servers, monitor performance
Database Admin--------------Manage databases-------------------Read/Write DB, Backup/Restore DB
Helpdesk------------------Support & troubleshooting----------Limited access to user accounts, reset passwords
Security Analyst----------Monitor & enforce security----------View logs, configure firewall, alert monitoring

3️⃣ Relation with Technical Role & Business Role:-
Technical Role: IT Roles ke under aate hain aur system/application specific permissions assign karte hain.
Business Role: IT Roles Business Role ke through employees ke job functions ke liye assign hote hain.

4️⃣ Code Conceptual Example (SailPoint Style):-
Identity ravi = getIdentity("EMP67890");

// Define Technical Role for IT
TechnicalRole itAdmin = new TechnicalRole("IT Admin");
itAdmin.setApplication("Server");
itAdmin.addPermission("Server Admin");
itAdmin.addPermission("Network Config");

// Define Business Role
BusinessRole itManager = new BusinessRole("IT Manager");
itManager.addTechnicalRole(itAdmin);

// Assign Business Role to user
ravi.assignBusinessRole(itManager);

// Ravi automatically gets IT Admin permissions
System.out.println("Ravi ke permissions: " + ravi.getPermissions());

()Optimistic Provisioning ka matlab hota hai: "Bina final confirmation ka wait kiye, pehle hi maan lena ki kaam ho gaya hai."
Ise ek simple real-life example se samajhte hain:-
Maan lijiye aapne Zomato/Swiggy se khana order kiya. Jaise hi aap payment karte hain,
app aapko turant dikha deta hai ki "Order Confirmed!" bhale hi restaurant ne abhi tak order accept karke banana shuru na kiya ho.
App ek optimistic (sakaaratmak) approach leta hai ki order ho hi jayega. Agar baad mein restaurant order cancel karta hai,
tab jake app aapko update karta hai.
Bas yahi same logic SailPoint mein lagta hai!

()SailPoint mein yeh kaise kaam karta hai?
Jab SailPoint kisi target system (jaise Active Directory, Workday, ya Salesforce) mein koi naya account banane ya access dene ki
request bhejta hai, toh normal process mein usko wait karna padta hai ki target system se "Success" ka message aaye. 
Lekin Optimistic Provisioning on hone par:-

--Request Bhejna: SailPoint target application ko provisioning ki request bhejta hai.

--Turant Update (Optimistic step): Target system se confirmation aane ka wait kiye bina hi,
SailPoint apne database (Identity Cube) ko turant update kar deta hai.

--UI Display: SailPoint ke UI mein dikhne lagta hai ki user ko access/account mil gaya hai.

--Reality Check (Aggregation): Kuch time baad jab SailPoint ka Aggregation task run hota hai,
tab SailPoint actual mein check karta hai ki us target system mein account sach mein bana tha ya nahi.

Agar ban gaya tha: Toh sab badhiya hai, status waisa hi rehta hai.

Agar fail ho gaya tha: (Jaise target server down tha ya koi error aa gayi thi),
toh SailPoint us access ko apne UI se hata deta hai (revert kar deta hai) kyunki actual mein user ko wo nahi mila tha.

Iska Fayda (Advantage) Kya Hai?
Faster User Experience: Users ko UI bahut fast aur responsive lagta hai.
Unhe turant dikh jata hai ki unka request process ho gaya hai aur dashboard jaldi update ho jata hai.

()SailPoint IIQ workflows(Business Processes) mein jab ek se zyada logon se approval chahiye hota hai(jaise Manager,App Owner,aur Security Team),
toh hum in Approval Modes ka use karte hain.Yeh modes system ko batate hain ki approvals kis sequence mein lene hain aur rejection aane par kya karna hai.

1. Serial (Ek ke baad ek):-
Kaise kaam karta hai: Isme request bilkul line se (sequence mein) chalti hai. Pehle request Approver A ke paas jayegi.
Jab A approve karega, tabhi request Approver B ke paas jayegi, aur fir C ke paas.
Rejection par kya hoga: Agar beech mein kisi ne bhi (maan lijiye A ne) reject kar diya,
toh process wahin turant ruk jayega. B aur C ko request kabhi jayegi hi nahi.
Fayda: Time aur effort bachta hai, kyunki pehla approver reject kar de toh baaki logon ko disturb nahi karna padta.

2. SerialPoll (Ek ke baad ek, par sabse poocho):-
Kaise kaam karta hai: Isme bhi request sequence mein hi aati hai (A -> B -> C).
Rejection par kya hoga: Isme bada farq yeh hai ki agar A ne request reject bhi kar di,
tab bhi request aage B aur C ke paas jayegi. System process ko rokta nahi hai.
Fayda (Use Case): Yeh wahan kaam aata hai jahan organization ko "Audit Trail" ke liye sabka decision (vote) record karna hota hai,
chahe request final mein reject hi kyun na ho rahi ho.

3. Parallel (Sabko ek saath):-
Kaise kaam karta hai: Isme approval request ek hi time par sabhi approvers (A, B, aur C) ko ek saath bhej di jati hai.
Sab apna-apna inbox check kar sakte hain.
Rejection par kya hoga: Agar koi ek bhi approver request ko reject kar deta hai,
toh SailPoint request ko turant reject maan leta hai. Jo baaki approvers hain,
unke inbox se bhi wo request automatically gayab (cancel) ho jati hai.
Fayda: Process bahut fast ho jata hai kyunki logon ko ek dusre ka wait nahi karna padta.

4. ParallelPoll (Sabko ek saath, par sabka vote chahiye):-
Kaise kaam karta hai: Isme bhi request sabko ek saath (Parallel) bheji jati hai.
Rejection par kya hoga: Lekin yahan system sabke response aane ka wait karta hai.
Agar A ne reject kar diya, toh B aur C ke inbox se request cancel nahi hogi.Woh dono bhi apna approval ya rejection de sakte hain.

Fayda (Use Case): Yeh ek tarah ke "Voting System" jaisa hai.
Jab kisi badi access ke liye panel ke sabhi members ka opinion system mein darj karna zaroori ho (irrespective of final result),
tab iska use hota hai.

()approvalSchema kya hai?
--SailPoint IIQ workflows mein ApprovalSchema ek tarah ka map ya blueprint hota hai,
jo yeh define karta hai ki approvals kis raaste se aur kis order mein jayenge.
Pichle sawal mein jo humne modes (Serial, Parallel, etc.) discuss kiye,
un sabko mila kar ek complex aur nested (ek ke andar ek) approval process banane ke liye ApprovalSchema ka istemaal kiya jata hai.
Iska sabse zyada use LCM (Lifecycle Manager) workflows, jaise ki 'LCM Provisioning' workflow mein hota hai.

--ApprovalSchema ki Zaroorat Kyun Padti Hai?
Agar approval simple hai (jaise sirf Manager se approval lena hai),
toh hum workflow mein sidha ek step bana kar approver ka naam daal sakte hain.
Lekin real-world scenarios bahut complex hote hain. Maan lijiye aapki requirement aisi hai:-
Sabse pehle Manager approve kare.
Agar Manager approve kar de, toh request Security Team aur Application Owner dono ke paas ek saath (Parallel) jaye.
Jab dono approve kar dein, tab jaake kaam pura ho.
Is tarah ke complex logic ko hardcode karne ke bajaye,SailPoint mein 
ek ApprovalSchema(jo basically ek structured data ya XML object hota hai)banaya jata hai.
Yeh schema dynamically workflow engine ko pass kiya jata hai.

--Yeh Kaise Kaam Karta Hai?
ApprovalSchema ek 'Tree' jaisa structure banata hai jisme blocks hote hain. In blocks ko ApprovalSet kaha jata hai.
Aap isme rules define kar sakte hain:-
Root Level (Serial): Pehle 'Block 1' chale, fir 'Block 2' chale.
Block 1 (Single Approver): Isme sirf Manager hai.
Block 2 (Parallel): Iske andar Security Team aur App Owner dono hain, aur iska mode 'Parallel' set hai.

()approvingIdentities kya hai?

तरीका 1: Method Overriding के ज़रिए (Polymorphism)
अगर कोई method Parent class में पहले से मौजूद (Declare) है, और Child class ने उसे Override (अपने तरीके से बदलाव) किया है, तो Parent class का reference अपने आप Child के method को call कर लेता है।

Java Example:
Java
class Parent {
    void show() {
        System.out.println("Parent ka method");
    }
    
    void callChild() {
        // यहाँ ये 'show' को call कर रहा है
        show(); 
    }
}

class Child extends Parent {
    @Override
    void show() {
        System.out.println("Child ka overridden method");
    }
}

public class Main {
    public static void main(String[] args) {
        Parent obj = new Child(); // Parent reference, Child object
        obj.callChild(); 
    }
}
Output:

Plaintext
Child ka overridden method

तरीका 2: Downcasting के ज़रिए (Explicitly)
अगर Child class में कोई बिल्कुल नया method है (जो Parent class में है ही नहीं), और Parent class को उसे call करना ही है, तो आपको Parent के reference को जबरदस्ती Child में बदलना पड़ेगा। इसे Downcasting कहते हैं।

Java Example:
Java
class Parent {
    void parentMethod() {
        System.out.println("I am Parent");
    }
}

class Child extends Parent {
    void childSpecialMethod() {
        System.out.println("I am Child's Special Method!");
    }
}

public class Main {
    public static void main(String[] args) {
        Parent p = new Child(); // Upcasting
        
        // p.childSpecialMethod(); // ERROR! Parent सीधे इसे नहीं देख सकता।
        
        // Downcasting: Parent को बताया कि तुम असल में Child हो
        if (p instanceof Child) {
            Child c = (Child) p; 
            c.childSpecialMethod(); // अब Call हो जाएगा!
        }
    }
}

Method Hiding Java ka ek aisa concept hai jo tab hota hai jab ek Child Class apne Parent Class ke bilkul same (same name aur same parameters waale) static method ko phir se define karti hai.
class Parent {
    static void display() {
        System.out.println("Parent ka static method");
    }
}

class Child extends Parent {
    // Parent ke static method ko hide kar raha hai
    static void display() {
        System.out.println("Child ka static method");
    }
}
public class Main {
    public static void main(String[] args) {
        Parent obj1 = new Parent();
        Parent obj2 = new Child(); // Reference Parent ka, Object Child ka
        Child obj3 = new Child();

        obj1.display(); // Output: Parent ka static method
        obj2.display(); // Output: Parent ka static method (Yahan twist hai!)
        obj3.display(); // Output: Child ka static method
    }
}

1. Left Side vs Right Side (Dono alag hain)
​Parent obj2 (Reference Type): Ye compiler ko batata hai ki obj2 naam ka ek remote control hai, 
jo sirf Parent class ke buttons (methods) ko jaanta hai.
​new Child() (Actual Object): Ye memory (Heap) mein banna waala asli insaan (object) hai.
​2. Runtime vs Compile-time ka Asar
​Agar display() ek normal (non-static) method hota, toh Java ka rule kehta hai: "Remote bhale hi Parent ka ho, 
chalega wahi jo Child ke andar define hai (Runtime Polymorphism)".
​Lekin yahan display() ek static method hai! Aur static methods par runtime ka koi rule lagu nahi hota.

Static variables aur static methods kahan store hote hain?
➡️ JVM ke Method Area (Class Area) me store hote hain, aur inki ek hi copy hoti hai jo sab objects share karte hain.
Instance variables kahan store hote hain?
➡️ Object ke andar Heap Memory me store hote hain.
Local variables kahan store hote hain?
➡️ Stack Memory me store hote hain.

🔍 Is Loophole Ka Matlab Kya Hai? (The Problem)
Aapne jo code likha tha, usme do try-catch blocks the—ek bada wala aur uske andar ek chota (nested) wala.

Java mein code top to bottom (upar se neeche) line by line chalta hai. Jab aapka purana code chala, 
toh background mein yeh hua:

Step 1: Code chota try block ke andar gaya aur Class.forName("com.mysql.cj.jdbc.Driver"); ko chalaya.

Step 2: Maan lijiye JAR file nahi mili, toh yahan ClassNotFoundException aa gayi.

Step 3: Java ne turant chote wale catch(ClassNotFoundException cx) block ko pakda aur print kar diya: 
"classnotfound exception...".

Step 4 (Yahan Gadbad Hui): Chote wale catch block ka kaam wahin khatam ho gaya! Java ko laga ki galti handle ho gayi hai. 
Isliye woh ruka nahi, aur chote catch block se baahar nikal kar seedhe agli line par chala gaya, jo ki thi:
