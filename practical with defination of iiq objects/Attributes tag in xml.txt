()2. <Attributes> Kya Hai?
Attributes basically ek Metadata Repository (Settings Box) hai. IdentityIQ me kai objects (jaise Form, Field, Section, Workflow) 
ke paas apni kuch extra properties hoti hain jinhe control karne ke liye hum <Attributes> tag ka use karte hain. 
Iske andar data hamesha <Map> aur <entry> ke form me hota hai.
Kaam: Yeh user ko screen par koi input box nahi dikhata, balki us element ke behavior ya settings ko backend me store karta hai.
Kiske andar hota hai: Yeh <Form> ke direct niche ho sakta hai, ya kisi <Field> ke andar bhi ho sakta hai.

one line:-matlab jitne bhi object hai jo spt_table me save hai un sabka setting configure karne ka tag hai ye <Attributes>.

Example 1: Form Level Par Attributes:-
Agar aapko poore Form ka title badalna hai ya buttons ke naam badalone hain:
XML
<Form name="KKR_Form_Example">
  <Attributes>
    <Map>
      <entry key="pageTitle" value="Special Onboarding Form"/>
      <entry key="submitLabel" value="Proceed Next"/>
      <entry key="cancelLabel" value="Abort"/>
    </Map>
  </Attributes>
</Form>
Yahan koi naya input field nahi bana, bas Form ke pure behavior aur look ko configure kiya gaya hai.

Example 2: Field Level Par Attributes:-
Kabhi-kabhi kisi <Field> ko special behavior dene ke liye uske andar bhi <Attributes> lagaya jata hai:
XML
<Field name="status" type="string" displayName="Account Status">
  <Attributes>
    <Map>
      <entry key="readOnly" value="true"/> <!-- Field ko read-only banane ke liye --
    </Map>
  </Attributes>
</Field>

()IdentityIQ me Application Object ke andar jab hum kisi target system (jaise Active Directory ya koi LDAP/Direct system) 
se connect karte hain, toh <Attributes> tag ke andar ye saari settings ek <Map> ke roop me save hoti hain.
ex:-
<?xml version='1.0' encoding='UTF-8'?>
<!DOCTYPE Application PUBLIC "sailpoint.dtd" "sailpoint.dtd">
<Application name="KKR_Active_Directory" type="Active Directory">
  
  <Description>Financial Planning users ke liye Active Directory Application Object</Description>
  
  <!-- Actual Schema aur baaki mappings database columns me hoti hain -->
  <!-- Lekin saari behavioral settings is <Attributes> tag ke andar hoti hain -->
  <Attributes>
    <Map>
      
      <!-- 1. Connection URL / Host / Port settings -->
      <entry key="host" value="192.168.1.50"/>
      <entry key="port" value="389"/> <!-- Standard LDAP Port, Secure ke liye 636 hoga -->
      <entry key="useSSL" value="false"/>
      <entry key="connectionTimeout" value="30000"/> <!-- Milliseconds me (30 seconds) -->

      <!-- 2. Authentication credentials (Username / Encrypted Password) -->
      <entry key="user" value="CN=iiq_service_acc,OU=ServiceAccounts,DC=kkr,DC=com"/>
      <!-- Password hamesha IIQ ke backend se encrypt hokar save hota hai -->
      <entry key="encryptedPassword" value="1:v84Nksj+Hj3lPqAsDwXqw==:A9jKm7xQ1LzP"/> 

      <!-- 3. Search DN / Search Scope / Timeout / Page Size -->
      <!-- Search DN: Kahan se users ko read karna shuru karna hai -->
      <entry key="searchDn" value="OU=FinancialPlanning,OU=CorporateUsers,DC=kkr,DC=com"/>
      <!-- Search Scope: SUBTREE (Yaani is OU ke andar ke saare child OUs bhi search honge) -->
      <entry key="searchScope" value="SUBTREE"/>
      <entry key="pageSize" value="100"/> <!-- Ek baar me 100 records ka batch fetch karega -->
      <entry key="searchTimeout" value="60"/> <!-- Seconds me timeout -->

      <!-- 4. Delta Aggregation tracking parameters -->
      <!-- Delta Aggregation ka matlab hai sirf wahi accounts uthao jo kal se aaj tak badle hain -->
      <entry key="deltaAggregation" value="true"/>
      <!-- Underlying system (AD) ka tracking attribute (jaise uSNChanged ya modifyTimestamp) -->
      <entry key="deltaAttribute" value="uSNChanged"/>
      <!-- Last successful aggregation ka sequence/timestamp jo IIQ automatic track karta hai -->
      <entry key="lastHighestUSN" value="1548293"/>

    </Map>
  </Attributes>

</Application>

()Host aur Port (host, port): SailPoint jab background me Connector chalayega, toh wo 192.168.1.50 IP par 389 port ke zariye hit karega.
()User aur Encrypted Password (user, encryptedPassword): Target AD me login karne ke liye system manual admin login nahi use karega, 
balki is Service Account (iiq_service_acc) aur uske encrypted password ko use karke secure handshake karega.
()Search DN aur Page Size (searchDn, pageSize): Poore AD me hazaron users ho sakte hain, par hume sirf Financial Planning ke bande chahiye.
Toh searchDn ne scope freeze kar diya. Aur pageSize="100" ka matlab hai ki agar 1000 users hain, 
toh system background me 100-100 ke 10 chakkar (batches) lagakar data layega taaki server par load na pade (Out of Memory Error na aaye).

()Delta Aggregation:-
1. Initial State (Pehli Baar Ka Aggregation):-
Active Directory me har account ke paas ek Number hota hai jise USN (Update Sequence Number) kehte hain. 
Jab bhi kisi user ka password, department, ya status badalta hai, AD uske USN ko badha deta hai.
Jab SailPoint pehli baar Full Aggregation chalata hai, 
toh wo sabhi users ko le aata hai aur dekhta hai ki AD me sabse bada (highest) USN kya chal raha hai. Maan lo wo number hai 1548293.
SailPoint is number ko apne Application Object ke andar save kar leta hai: <entry key="lastHighestUSN" value="1548293"/>.

2. Live Changes (Runtime par kya hua):-
Agle din, pooray department me sirf 2 employees ka data badla (ek ka Manager change hua, ek ka Last Name change hua).
AD ne un do users ka uSNChanged value badhakar 1548310 aur 1548325 kar diya. Baaki sabhi 99,998 users ka number purana hi raha.

3. Delta Aggregation (Agli subah jab task chala):-
Jab agle din Delta Task chalta hai, toh SailPoint AD se kehta hai:-
"Mujhe saare users mat do. Mujhe sirf wo users do jinka uSNChanged number 1548293 (pichla saved number) se bada hai."
AD query chala kar turant sirf un 2 users ka data SailPoint ko de deta hai.
Task khatam hote hi SailPoint naya highest number (1548325) phir se apne pass save kar leta hai agle din ke liye.

()Task Definition Object (Aggregation, Refresh, Run Rule):-
Target Filters (OU, Group, ya Sub-scoped identities)
Execution checkboxes (e.g., refreshIdentityEntitlements, correlateEntitlements)
Threshold alert limits aur error handling rules.

1. Account Aggregation Task Definition (Example Code):-
Yeh ek standard Aggregation Task ka XML hai jo Active Directory se naye accounts uthane aur purano ko refresh karne ke liye use hota hai:-
<?xml version='1.0' encoding='UTF-8'?>
<!DOCTYPE TaskDefinition PUBLIC "sailpoint.dtd" "sailpoint.dtd">
<TaskDefinition name="KKR_AD_Account_Aggregation" type="AccountAggregation">
  <Description>Active Directory accounts ko sync karne aur entitlements refresh karne ka task</Description>
  
  <Attributes>
    <Map>
      <!-- Target Application specify karna mandatory hai -->
      <entry key="applications" value="KKR_Active_Directory"/>

      <!-- 1. Target Filters (OU, Group, ya Sub-scoped identities) -->
      <!-- Sirf isi specific HR aur Finance OU ke accounts aggregate honge -->
      <entry key="accountFilterScope" value="OU=HR,OU=Users,DC=kkr,DC=com;OU=Finance,OU=Users,DC=kkr,DC=com"/>
      <!-- Group Filter: Sirf wahi accounts uthao jo 'VIP-Users' group ke member hain -->
      <entry key="groupFilter" value="CN=VIP-Users,OU=Groups,DC=kkr,DC=com"/>

      <!-- 2. Execution Checkboxes (Backend Processing Toggles) -->
      <!-- Correlate Entitlements: Aggregation ke waqt hi groups/roles ko map kar do -->
      <entry key="correlateEntitlements" value="true"/>
      <!-- Refresh Identity Entitlements: Aggregation ke baad identity ki profile immediately refresh karo -->
      <entry key="refreshIdentityEntitlements" value="true"/>
      <!-- Check Sibling Accounts: Agar user ke multiple accounts hain toh correlation verify karo -->
      <entry key="checkSiblingAccounts" value="false"/>

      <!-- 3. Threshold alert limits aur error handling rules -->
      <!-- Task Tabhi stop hoga jab errors limit se bahar jayengi -->
      <entry key="haltOnError" value="true"/>
      <!-- Max Error Threshold: Agar 50 se zyada accounts fail huye, toh poora task terminate kar do -->
      <entry key="maxErrors" value="50"/>
      <!-- Threshold Percentage Alert: Agar pichli baar se 10% zyada accounts delete ho rahe hain, toh process rok do -->
      <entry key="terminationThreshold" value="10"/>

    </Map>
  </Attributes>
</TaskDefinition>
/////////////////*
()applications:-
Kya karta hai: Yeh engine ko batata hai ki kis target application se data khinchkar laana hai.
Backend Asar: SailPoint direct us application (e.g., KKR_Active_Directory) ke configuration settings(Host, IP, Credentials) 
par jaata hai taaki connection bana sake.

()accountFilterScope:-
Kya karta hai: Yeh target system par ek boundary (scope) set karta hai.
Backend Asar: Agar aap poore AD forest ka data nahi chahte, toh yahan defined OUs(jaise OU=HR aur OU=Finance) 
ke andar maujood accounts ko hi SailPoint dhoondta hai, baaki OUs ko chhod deta hai.

()groupFilter:-
Kya karta hai: Yeh accounts ko filter karne ke liye ek extra condition lagata hai base on AD Groups.
Backend Asar: Target system se sirf wahi accounts aggregate honge jo is specific group(e.g., VIP-Users) 
ke andar mapped hain. Baaki users ignore ho jayenge.

()correlateEntitlements:-
Kya karta hai: Yeh check karta hai ki user ke paas target system par kaun-kaun se groups ya permissions hain 
aur unhe SailPoint me map karta hai.Backend Asar: Agar ye true hai, 
toh account ke sath-sath uske saare groups bhi fetch hote hain aur unhe SailPoint ke identity profile ke sath link kar diya jata hai.

()refreshIdentityEntitlements:-
Kya karta hai: Naye accounts aur permissions aane ke baad, 
pooray Identity Cube ki state ko background me immediate refresh karne ka signal deta hai.
Backend Asar: Jaise hi account aggregate hota hai, 
ye backend engine ko bolta hai ki is user par saare Joiner/Mover rules aur dynamic roles dobara calculate karo 
taaki data immediately accurate dikhe.

()checkSiblingAccounts:-
Kya karta hai:Yeh check karta hai ki kya ek hi insaan ke paas us application par ek se zyada accounts(Secondary/Admin accounts) toh nahi hain.
Backend Asar: Agar yeh false hai, toh system ek identity ke liye sirf ek main account link karke aage badh jata hai, 
sibling checking par time waste nahi karta.

()haltOnError:-
Kya karta hai: Yeh ek safety switch hai jo bataata hai ki agar task me koi error aaye toh kya karna hai.
Backend Asar: Agar ye true hai aur task me koi badi dikkat aati hai (jaise network disconnect hona), 
toh task turant ruk (halt) jata hai,जबरदस्ती aage nahi chalta.

()maxErrors:-
Kya karta hai: Yeh ek error counting limit (threshold) set karta hai.
Backend Asar: Agar aggregate karte waqt single accounts fail ho rahe hain(jaise data corruption ki wajah se), 
toh jaise hi failure count 50 tak pahuchega, SailPoint poore task ko terminate (band) kar dega taaki system par faaltu load na pade.

()terminationThreshold:-
Kya karta hai: Yeh Data Loss Prevention (DLP) ka kaam karta hai, jo percentage base par kaam karta hai.
Backend Asar: Maan lo target system se pichli baar ke muqable is baar 10% se zyada accounts gayab (delete) milte hain. 
SailPoint ko lagega ki kahin AD me bulk deletion toh nahi ho gayi? 
Toh safety ke liye ye task ko wahin freeze kar deta hai taaki galati se SailPoint me bulk accounts delete na ho jayein.

2. Identity Refresh Task Definition (Example Code):-
Yeh dusra sabse important task hai jo identity ka risk score compute karne, manager hierarchy check karne,
aur policy violations (jaise SOD) dhoondne me kaam aata hai. Isme checkboxes ka khel alag hota hai:-
ex:-
<?xml version='1.0' encoding='UTF-8'?>
<!DOCTYPE TaskDefinition PUBLIC "sailpoint.dtd" "sailpoint.dtd">
<TaskDefinition name="KKR_Identity_Refresh_Task" type="IdentityRefresh">
  <Description>Identities ka Risk Score, Manager Mapping aur Policy checking refresh karne ka task</Description>
  
  <Attributes>
    <Map>
      <!-- 1. Target Filters (Sub-scoped identities) -->
      <!-- Pure organisation ko refresh karne ke bajaye sirf 'Mumbai' location ke users filter honge -->
      <entry key="filter" value="location == &quot;Mumbai&quot;"/>

      <!-- 2. Execution Checkboxes (Kya-Kya action perform karna hai) -->
      <entry key="refreshManagerStatus" value="true"/> <!-- Manager changes update karo -->
      <entry key="correlateEntitlements" value="true"/> <!-- Roles aur permissions link karo -->
      <entry key="checkPolicies" value="true"/> <!-- Segregation of Duties (SOD) violations scan karo -->
      <entry key="provisionCorrelations" value="true"/> <!-- Agar koi naya automatic role banta hai toh provision karo -->

      <!-- 3. Error Handling and Optimization -->
      <!-- Multithreading support taaki task fast chale (8 parallel threads ek sath kaam karenge) -->
      <entry key="threads" value="8"/>
      <entry key="haltOnBadPartition" value="true"/> <!-- Agar kisi ek partition me error aayi toh pure task ko hold par daal do -->
    </Map>
  </Attributes>
</TaskDefinition> 
///////////////////*
()2. Identity Refresh Task ke Attributes
Yeh task SailPoint ke andar pehle se maujood Identity Cubes par calculation chalata hai. 
Iske attributes ye hain:-
--filter:-Kya karta hai: Yeh is baat ka filter hai ki pooray system me se kis-kis employee ki profile ko refresh karna hai.
Backend Asar: spt_identity table par query chalti hai. Jaise location == "Mumbai", toh engine baaki saare desh ke users ko chhod dega, 
sirf Mumbai wale identities ko pick karke unka data process karega.

--refreshManagerStatus:-
Kya karta hai: Yeh employees ke manager ki hierarchy ko calculate aur update karta hai.
Backend Asar: Agar backend HR data me kisi ka manager badla hai, 
toh ye attribute true hone par SailPoint us user ki identity par naye manager ki ID mapping aur approvals routes ko automatic thik kar deta hai.

--checkPolicies:-
Kya karta hai: Yeh system ke andar security rules aur compliance check karne ka engine trigger karta hai.
Backend Asar: true hone par, yeh har identity par jaakar check karta hai ki kahin is bande par koi Segregation of Duties (SOD) 
policy violation toh hit nahi ho raha (jaise ek hi bande ke paas Requester aur Approver dono rights hona).

--provisionCorrelations:-
Kya karta hai: Agar refresh hone ke baad user par koi naya automatic ya business role assign hota hai, 
toh usko target system par bhejne (provision karne) ka trigger hai.
Backend Asar: Agar refresh task ne dekha ki user ka department badal gaya hai aur ab use naya access milna chahiye, 
toh ye true hone par backend engine turant provisioning plan banakar target system par account update karne ka process start kar deta hai.

--threads:-
Kya karta hai: Performance badhane ke liye multithreading (parallel processing) configure karta hai.
Backend Asar: Agar value 8 hai, toh SailPoint ka engine server ke CPU se 8 parallel threads allocate karega. 
Isse 1-1 karke user refresh nahi hote, balki 8 users ka data ek sath parallel refresh hota hai, 
jisse task 8 guna fast khatam ho jata hai.

--haltOnBadPartition:-
Kya karta hai: Multithreading ke waqt agar kisi ek hisse (partition) me error aaye, toh pure task ka behavior set karta hai.
Backend Asar: Jab bade data ko alag-alag chunks me divide kiya jata hai, aur kisi ek chunk me koi severe database error aati hai, 
toh ye true hone par poore task ki execution ko block kar diya jata hai taaki data corrupt na ho.

Atomic Execution: AD connector ko jo provisioning plan milega, usme saaf likha hoga: "Create Account with attributes: groups = [Base-Group, Manager-Group]".
Zero Conflict & Errors: Target system par koi duplicate identity entity creation ka clash nahi hoga.

  <Signature returnType="sailpoint.object.Identity">

        <Description>The application object being aggregated.</Description>
      </Argument>
      <!-- Target system se aaya hua raw account data (Link/ResourceObject) -->

      </Argument>

      import sailpoint.object.Identity;
      import sailpoint.object.ResourceObject;
      import org.apache.commons.logging.Log;
      import org.apache.commons.logging.LogFactory;

      // Custom Logger initialize karna exact rules ke mutabik
      Log logger = LogFactory.getLog("sailpoint.custom.kkr.correlation");

      if (account == null) {
          logger.error("Bhai, rule me account object hi NULL aaya hai! Aggregation check karo.");
          return null;
      }

      // Context utilization: Target account se data nikalna
      String targetEmpId = account.getString("employeeID");
      String targetEmail = account.getString("mail");

      logger.info("Processing correlation context for Account ID: " + account.getIdentity() + " | EmpID: " + targetEmpId);

      Identity matchedIdentity = null;

      // Asli Core Logic
      if (targetEmpId != null) {
          // Internal context ka use karke database me search chalana
          matchedIdentity = context.getObjectByName(Identity.class, targetEmpId);
      }

      if (matchedIdentity != null) {
          logger.info("SUCCESS: Account mapped directly to Identity: " + matchedIdentity.getName());
          return matchedIdentity; // Engine account ko is user se link kar dega
      } else {
          logger.warn("WARNING: No direct match found for EmpID: " + targetEmpId + ". Sending to orphan accounts bucket.");
      }

      return null;
    ]]>
  </Source>
</Rule>

()Correlation Config Object:-
-->Account-to-Identity mapping expressions (Attribute priority lists)

  </Attributes>

  <!-- 2. FALLBACK/SECONDARY CORRELATION RULE (OR Condition)   -->
  <!-- ======================================================= -->

name="Financial_SAP_App_Correlation_Config": Jab tum Application XML likhte ho, toh wahan is config object ka reference de dete ho:

()Identity Trigger Object:-

<?xml version='1.0' encoding='UTF-8'?>
<!DOCTYPE IdentityTrigger PUBLIC "sailpoint.dtd" "sailpoint.dtd">

<IdentityTrigger name="KKR_Financial_Planning_Transfer_Trigger" type="IdentityAttributeChange">
  <Description>Custom trigger to detect department changes to Financial Planning</Description>
  
  <Attributes>
    <Map>
      <!-- ======================================================= -->
      <!-- 1. TRIGGER SOURCE ATTRIBUTE SELECTION                  -->
      <!-- ======================================================= -->
      <!-- attributeName: Kis global identity attribute par radar lagana hai -->

      <!-- ======================================================= -->
      <!-- 2. CORE FILTER LOGIC (Old Value vs New Value Check)    -->
      <!-- ======================================================= -->
      <!-- Hum check kar rahe hain ki user naye department me aaya hai ya nahi -->
      <entry key="attributeValue" value="Financial Planning"/>
      
      <!-- operation: Kya attribute ki sirf value badalni chahiye, ya specific value aani chahiye -->
      <entry key="operation" value="ChangedTo"/>

      <!-- ======================================================= -->
      <!-- 3. DETAILED BEHIND-THE-SCENE BEHNSHELL SCRIPTING        -->

      <!-- Agar direct attribute match se kaam na chale, toh custom selector script -->
      <entry key="selectorScript">
        <value>
          <Script>
            <Source>
              // Engine runtime par 'previousIdentity' (purana data) aur 'newIdentity' (naya data) dono deta hai
              if (previousIdentity != null &amp;&amp; newIdentity != null) {
                  String oldDept = previousIdentity.getAttribute("department");
                  String newDept = newIdentity.getAttribute("department");
                  
                  // Strict Check: Agar banda pehle se HR me tha aur ab Finance me aaya hai, toh hi trigger fire karo
                  if ("Human Resources".equals(oldDept) &amp;&amp; "Financial Planning".equals(newDept)) {
                      return true; // Radar match found! Trigger fired.
                  }
              }
              return false;
            </Source>
          </Script>
        </value>
      </entry>
    </Map>
  </Attributes>

<?xml version='1.0' encoding='UTF-8'?>
<!DOCTYPE Workflow PUBLIC "sailpoint.dtd" "sailpoint.dtd">
<Workflow name="KKR_Custom_Transfer_Provisioning_Workflow" type="IdentityLifecycle">
  
  <Variable name="identity" input="true" required="true">
    <Description>The Identity object undergoing the transfer event.</Description>
  </Variable>

  <!-- START STEP -->
  <Step icon="Start" name="Start" x="50" y="50">
    <Transition to="Execute Custom Provisioning Plan"/>
  </Step>

  <!-- ====== TUMHARA CUSTOM PROVISIONING STEP ====== -->
  <Step icon="Provision" name="Execute Custom Provisioning Plan" x="220" y="50">
    <Script>
      <Source>
        <![CDATA[
          import sailpoint.object.ProvisioningPlan;
          import sailpoint.object.ProvisioningPlan.AccountRequest;
          import sailpoint.object.ProvisioningPlan.AttributeRequest;
          import sailpoint.object.Identity;

          if (identity == null) {
              return;
          }

          // 1. Ek naya custom provisioning plan taiyar karo
          ProvisioningPlan plan = new ProvisioningPlan();
          plan.setIdentity(identity);

          // ==========================================
          // STAGE 3: PURANA ACCESS DEPROVISION (REVOKE)
          // ==========================================
          AccountRequest hrRequest = new AccountRequest();
          hrRequest.setApplication("Active_Directory_App");
          hrRequest.setOperation(AccountRequest.Operation.Modify);
          
          // Remove Request: Purana Active Directory group chheen lo
          hrRequest.add(new AttributeRequest("memberOf", ProvisioningPlan.Operation.Remove, "CN=HR-Admin-Group,OU=Groups,DC=kkr,DC=com"));
          plan.add(hrRequest);

          // ==========================================
          // STAGE 4: NAYA ACCESS PROVISIONING
          // ==========================================
          AccountRequest finRequest = new AccountRequest();
          finRequest.setApplication("Financial_SAP_App");
          finRequest.setOperation(AccountRequest.Operation.Modify);
          
          // Add Request: Naya SAP role automatic de do
          finRequest.add(new AttributeRequest("roles", ProvisioningPlan.Operation.Add, "SAP_FIN_LEDGER_VIEW"));
          plan.add(finRequest);

          // 2. SailPoint ke core Provisioner Engine ko compile karke call maro

Target systems se permissions aur explicit access scopes ko read karne ki rules.
Target / Target Source Object (SailPoint IIQ)
Ye terms workflow, provisioning, mapping, integration me aate hain.
Simple language me:-
Target = Jis object/system par action hona hai
Target Source Object = Jis source object se target ki value/data aa rahi hai.

<?xml version='1.0' encoding='UTF-8'?>
<!DOCTYPE TargetSource PUBLIC "sailpoint.dtd" "sailpoint.dtd">
<!-- name: Is target governance model ka unique identifier -->
<TargetSource name="KKR_ActiveDirectory_Governance_Target">
  <Description>Advanced Target Source for mapping AD Administrative Directives and Explicit Scopes</Description>
  
  <!-- Target Application Reference: Kis core application se rules link karne hain -->
  <ApplicationRef>
    <Reference class="sailpoint.object.Application" name="Active_Directory_App"/>
  </ApplicationRef>

  <Attributes>
    <Map>
      <!-- ======================================================= -->
      <!-- 1. IDENTITY AUTHORIZATION & ACCESS GOVERNANCE DIRECTIVES-->
      <!-- ======================================================= -->
      <!-- authorizationRule: Yeh custom rule user ki identity ko padh kar target authorization level set karega -->
      <entry key="authorizationRule" value="KKR_AD_Target_Authorization_Directive_Rule"/>
      
      <!-- enforceStrictGovernance: True matlab bina complete match ke access pass nahi ho sakta -->
      <entry key="enforceStrictGovernance" value="true"/>

      <!-- ======================================================= -->
      <!-- 2. EXPLICIT ACCESS SCOPES & PERMISSION RECON RULES     -->
      <!-- ======================================================= -->
      <!-- scopeReadRule: Target system se nested/explicit scopes read karne ka dimaag -->
      <entry key="scopeReadRule" value="KKR_AD_Scope_Resolution_Rule"/>

      <!-- targetType: Kis tarah ke explicit structure ko read karna hai -->
      <entry key="targetType" value="Administrative_Domain_Scope"/>
      
      <!-- Max target depth to parse nested group structures -->
      <entry key="maxHierarchyDepth" value="5"/>
    </Map>
  </Attributes>
</TargetSource>

()Capability Object;-
Kisi specific Rights ya Permissions ke administrative scope ko limit karne ki settings.
Naye UI modules ko capabilities ke sath tightly bind karne ke flags.

-->SailPoint ke andar kisi bhi user, developer, ya administrator ka kya power level hoga—
wo kya dekh sakta hai aur kya button daba sakta hai—wo isi object se control hota hai.

<!-- name: Is custom permission level ka unique identifier -->
<Capability name="KKR_Regional_Finance_Auditor_Capability">
  <Description>Custom administrative capability for Finance Auditors with regional boundaries and custom UI module locks.</Description>
  
  <!-- ======================================================= -->
  <!-- 1. RIGHT ASSIGNMENT (Asli Access Power)                 -->
  <!-- ======================================================= -->

    <Reference class="sailpoint.object.Right" name="ViewIdentity"/>
    <Reference class="sailpoint.object.Right" name="ViewReports"/>
    <Reference class="sailpoint.object.Right" name="FullAuditComplianceRight"/>
  </Rights>

  <!-- ======================================================= -->
  <!-- 2. ADMINISTRATIVE SCOPE LIMIT SETTINGS                  -->
  <!-- ======================================================= -->
  <Attributes>
    <Map>
      <!-- scoper: Scope Control Rule Reference -->
      <!-- Yeh user ke hath baandh deta hai taaki wo puri duniya ka data na dekh sake -->
      <entry key="scoper" value="KKR_Finance_Regional_Scope_Control_Rule"/>

      <entry key="defaultScopeRestriction" value="true"/>

      </entry>

      <entry key="uiModuleAssignmentStrict" value="true"/>
    </Map>
  </Attributes>

()<entry key="scoper" value="KKR_Finance_Regional_Scope_Control_Rule"/>:-
Karta kya hai: Dynamic Boundary Rule Trigger. Yeh engine ko ek custom parameter deta hai. 
Jab is capability wala user kisi ka profile dhoondhne jayega, 
toh system is key par mapped Beanshell code (KKR_Finance_Regional_Scope_Control_Rule) ko chala dega. 
Wo code user ke location attribute ko check karega aur strict filter laga dega ki "Bhai,
tum right toh rakhte ho dekhne ka, par tum sirf 'APAC region' ke employees ka data hi dekh sakte ho, 
baki pure global corporate user database ko tum touch nahi kar sakte."

()<entry key="defaultScopeRestriction" value="true"/>:-
Karta kya hai: Strict Isolation Switch. Agar iski value true hai, 
toh engine bina upar wale scope filter pass huye user ko kisi bhi data query ka access nahi phekega (Green light nahi dega).

()<entry key="restrictedUiModules">:-ye dono tab ushi ko dikhenge jisse ye oject capibility(KKR_Regional_Finance_Auditor_Capability) mili hai.
()2. <String>KKR_Custom_Finance_Audit_Dashboard</String>
Karta kya hai: Custom Page Identity. Yeh tumhare us naye full-screen UI module ya component ka exact layout code 
reference ID hai jahan finance audit ki reports padi hain. Yeh page ab strictly lock ho chuka hai.
()3. <String>KKR_SOX_Violation_Analytics_Widget</String>
Karta kya hai: Custom Widget Identity. Home page par jo analytics tiles ya summary cards dikhte hain,
 yeh us specific custom widget ka reserved name hai. Yeh card bhi normal users se ab chup gaya hai.
 
()<entry key="uiModuleAssignmentStrict" value="true"/>:-
Agar koi shatir user (jaise Rahul) direct browser ke URL path me jaakar type kar de:-
https://sailpoint-url/ui/KKR_Custom_Finance_Audit_Dashboard.
If Value is true: SailPoint ka server frontend rendering ko thukra dega.aur rahul ko 403 errorr dega 
kyuki ushke pass main capability object(KKR_Regional_Finance_Auditor_Capability) nahi hai.

()<entry key="autoRefresh" value="true"/>:-
Karta kya hai: Maintenance Scan Engine Switch. Jab har subah global Identity Refresh Task chalega, 
toh engine is factory ko re-evaluate karega. Agar koi employee IT-Mumbai se transfer ho kar HR-Mumbai me gaya, 
toh is auto-refresh switch ke chalte system user ko purane group se fenkega bahar aur naye group me automatic inject kar dega.
Agar tum ise false kar doge, toh group sirf ek baar ban kar freeze ho jayega. 
Naye join karne wale employees ya transfer hone wale log is dynamic group me kabhi automatic add nahi honge 
jab tak tum manually task nahi chalaoge. true hone par ye automatic har scheduling cycle me sync hota hai.

()2. entry key="processingIndexPriority" value="10" (Execution Sequence Controller)
Karta kya hai: Yeh pooray server background tasks ka VIP Pass (Priority Index) hai.
Engine Level Logic: SailPoint me low integer numbers (jaise 5, 10, 20) ko high priority mana jata hai.
value="10" ka matlab hai ki server ke memory threads baki faltu reports aur lower-priority group mappings ko hold par daal denge,
aur is group factory ke processing indexes ko sabse pehle compile karenge taaki access allocation me delay na ho.

()4. <GroupFilter> Tag Block (The Master Gatekeeper)Karta kya hai:-Yeh tumhare automatic combinations par ek strict Global 
Security Filter lagata hai.operation="AND": Iska matlab hai ki is block ke andar likhi saari conditions ka pass hona zaroori hai
,<Filter operation="EQ" property="identityStatus" value="Active"/>: EQ ka matlab hota hai Equal To ($=$). 
Yeh line strictly engine ko bolti hai ki tum combination chahe jitne bana lo (IT-Mumbai, HR-Bangalore, etc.), 
lekin un groups ke andar sirf aur sirf wahi log jayenge jinka identityStatus ekdum strictly "Active" chal raha hai. 
Suspended, Terminated ya Leave-of-Absence wale employees ko is membership se door rakha jaye.
"AND" ka matlab upar conditions locaion wale aur ye filter ka condition status wale dono match tabhi user ko group me inject karo.

()<entry key="optimizedIndexQuery" value="true"/>:-
Maan lo tumhari company me 50,000 employees hain aur unka data ek badi register (database table spt_identity) me likha hai. 
Tumne system ko bola: "Mujhe un sab logo ko dhoondh kar do jo Mumbai location ke hain."

Agar yeh FALSE (ya missing) hai: SailPoint ka engine register ka pehla panna kholega aur aakhri panne tak ek-of-a-kind line ko
padhega (Isse technical language me Full Table Scan kehte hain). Agar 50,000 rows hain, toh engine 50,000 baar database par 
search karega. Isse tumhara database server bohot slow ho jayega.
Agar yeh TRUE hai: SailPoint ka engine database ko bolta hai: "Mujhe poori table line-by-line nahi padhni.
 Humne jo upar se 'Location' aur 'Department' ke indexes bana rakhe hain, tum seedhe un par jump karo" (Isse Index Scan kehte hain).
toh engine bina baki 49,000 rows ko chhue, seedhe un 1,000 Mumbai wale employees par hit karega.

()Provisioning Project Object (Runtime Object):-
Compilation state flags (Workflow jab provisioning plan execute karta hai, toh runtime par conflicts, 
evaluation steps aur targeting rules isme temporary save hoti hain).

<?xml version='1.0' encoding='UTF-8'?>
<!DOCTYPE ProvisioningProject PUBLIC "sailpoint.dtd" "sailpoint.dtd">
<!-- identityName: Kis employee ke liye ye provisioning runtime engine chal raha hai -->
<ProvisioningProject identityName="shiva.kumar">
  
  <!-- ======================================================= -->
  <!-- 1. COMPILATION STATE FLAGS & TARGETING RULES            -->
  <!-- ======================================================= -->
  <Attributes>
    <Map>
      <!-- compiled: true batata hai ki provisioning plans successfully compile ho chuke hain -->
      <entry key="compiled" value="true"/>
      
      <!-- optimizable: Engine check karta hai ki kya requests ko batch kiya ja sakta hai -->
      <entry key="optimizable" value="true"/>

      <!-- conflictDetected: Runtime evaluation step me agar koi Access Policy break ho rahi ho -->
      <entry key="conflictDetected" value="false"/>

      <!-- targetSystemStatus: Backend validation state tracker before push -->
      <entry key="targetSystemStatus" value="VERIFIED_ONLINE"/>
    </Map>
  </Attributes>

  <!-- ======================================================= -->
  <!-- 2. EVALUATION STEPS & PROVISIONING PLANS                -->
  <!-- ======================================================= -->
  <!-- ProvisioningPlan: Asli physical changes jo target system par bhejne hain -->
  <ProvisioningPlan minimal="true">
    <AccountRequest application="Active_Directory_App" op="Modify" nativeIdentity="CN=Shiva Kumar,OU=Users,DC=kkr,DC=com">
      <!-- Evaluation Step: Directives mapping rules on attribute level -->
      <AttributeRequest name="memberOf" op="Add" value="CN=Enterprise-Core-Admins,OU=Groups,DC=kkr,DC=com">
        <Attributes>
          <Map>
            <!-- Tracking evaluation assignment origin -->
            <entry key="assignmentId" value="0a0a0a0a-7c8b-11e9-b27a-1234567890ab"/>
          </Map>
        </Attributes>
      </AttributeRequest>
    </AccountRequest>
  </ProvisioningPlan>

</ProvisioningProject>

()<entry key="compiled" value="true"/>:-
Engine ko green signal deta hai ki plan compile ho chuka hai, koi structural code syntax error nahi hai.

4. <entry key="optimizable" value="true"/>:-
Provisioner ko bolta hai ki agar is user ke aur bhi tickets hain toh unhe merge karke ek single payload bana lo.

5. <entry key="conflictDetected" value="false"/>:-
Radar toggle confirmation deta hai ki background evaluation me koi SoD (Separation of Duties) policy break nahi hui hai.

()1. entry key="targetSystemStatus" value="VERIFIED_ONLINE" (The Gateway Guard)
Is single line ka backend database aur implementation me sabse critical role hai. Jab hum SailPoint se kisi target application (jaise Active Directory, SAP, ya Workday) par koi provisioning request push karne wale hote hain, toh system blind hit (andhadhundh transaction) nahi marta.
The Connection Verification (Handshake): SailPoint ka core Provisioner Engine plan execute karne se just pehle target application connector ke internal testConnection() method ko trigger karta hai.
Engine Level State Machine:-
Jaise hi remote Active Directory domain controller respond karta hai, engine is parameter ki value ko temporary memory me VERIFIED_ONLINE mark kar deta hai.
Iska matlab hai ki rasta ekdum saaf hai, connection pipelines responsive hain, aur payload push kiya ja sakta hai.

()<ProvisioningPlan minimal="true">
Engine Level Operation: Jab user request generate karta hai, 
toh Java code internally ProvisioningPlan plan = new ProvisioningPlan(); call karta hai.
minimal="true" Ka Khel: Yeh attribute batata hai ki is plan me user ka poora data 
(jaise uski reporting hierarchy, manager identity, manager name etc.) nahi bhejna hai. 
Sirf aur sirf wahi data bhejenge jo badalna hai, jisse dynamic payload ka size chhota rehta hai aur execution speed fast ho jaati hai.

()<AccountRequest application="Active_Directory_App" op="Modify" nativeIdentity="CN=Shiva Kumar,OU=Users,DC=kkr,DC=com">
Yeh line target application connector ke liye "Search and Lock Warrant" hai:
application: System directly target directory context ko check karta hai ki mujhe is process ke liye kis application ke driver (connector class) ko call karna hai.
op="Modify": Iski value Modify, Create, Delete, Enable, ya Disable ho sakti hai. Modify batata hai ki target user account pahle se hi Active Directory tree par zinda hai, hamein naya account nahi banana hai, bas usi me badlaav karna hai.
nativeIdentity: Yeh target system par us user ka unique identity address (Distinguished Name - DN) hai. AD Connector is pure string ko target LDAP filter me inject karta hai taaki direct usi employee ke object boundary ko fetch kiya ja sake.

()<AttributeRequest name="memberOf" op="Add" value="CN=Enterprise-Core-Admins,OU=Groups,DC=kkr,DC=com">
Yeh line target account ke properties page par chalne wala Asli Action Operation hai:
name="memberOf": Connector framework ko target column ya field specific structure lock milta hai. AD me group membership ko memberOf kehte hain.
op="Add": Iski value Add, Remove, ya Set ho sakti hai. Add batata hai ki user ke paas jo purani groups hain unhe touch nahi karna hai, bas unke sath is naye administrative group ko parallel me append (jod) dena hai.
value: Jo group user ko access me milna hai, uska exact deployment value string.

()<entry key="assignmentId" value="0a0a0a0a-7c8b-11e9-b27a-1234567890ab"/>
Engine Level Operation: Jab ye provisioning transaction successfully target AD par complete ho jata hai, toh SailPoint ka Identity Refresh engine database table spt_identity_entitlement me entry save karta hai.
Asli Kaam: Yeh unique assignment token (GUID) us record ke sath permanent link ho jata hai. Kal ko agar audit team poochegi ki "Shiva Kumar ke account me yeh group kisne joda?", toh SailPoint is audit key (assignmentId) ko read karega aur batayega ki "Yeh kisi manager ne request nahi kiya tha, yeh piche chal rahe ek LCM automatic lifecycle rule ke zariye system me provision hua tha."
