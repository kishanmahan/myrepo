()IdentityIQ ki Main Capabilities (Firse, easy points mein):
Identity Cube: Har employee ka ek "Digital Folder" banta hai. Isme uske saare accounts (AD, SAP, Gmail) ki jankari ek jagah hoti hai.

Joiner, Mover, Leaver (JML):-
Joiner: Naya banda aaya -> Auto access mil gaya.
Mover: Dept change hua -> Purana access gaya, naya mil gaya.
Leaver: Job chhodi -> Saare access 5 minute mein band.

Certification: Managers ko har 3-6 mahine mein "Yes/No" click karna padta hai ki unki team ko wo access chahiye ya nahi.

Separation of Duties (SoD): Aise rules set karna ki koi fraud na ho sake. (Jaise jo bill banaye, wo khud pay na kar sake).

Workflows: Har request ke peeche ek logic hota hai (Pehle Manager approve karega, fir Owner, fir account banega).

1. SP ka matlab "SailPoint":-
Agar aap overall SailPoint ki capabilities puch rahe hain, 
toh iska kaam hai "Who has access to what, and who SHOULD have access to what?" (Kiske paas kya access hai, aur kya hona chahiye?).
Iska main focus in 3 cheezon par hota hai:-
Visibility: Saare accounts aur permissions ko ek jagah dekhna.
Control: Access dena (Provisioning) aur hatana (De-provisioning).
Compliance: Ye prove karna ki audit ke hisaab se sab sahi hai.

2. Service Provider (SAML / SSO ke context mein - Technical Meaning):-
Agar aap kisi integration ya Single Sign-On (SSO) ki baat kar rahe hain, toh SP ka matlab hota hai Service Provider.
Ye kya hota hai? Ye wo application (ya service) hai jise user use karna chahta hai. Is case mein, khud IdentityIQ ek Service Provider ban jata hai.
Kaise kaam karta hai? Jab koi user IIQ kholta hai, toh IIQ (SP) user se khud password check nahi karta.
Wo user ko ek doosre system ke paas bhejta hai jise IdP (Identity Provider) kehte hain (jaise Okta, PingIdentity, ya Azure AD).
IdP check karta hai ki user asli hai ya nahi, aur fir SP (IIQ) ko secure token bhejta hai ki "Haan, identity verify ho gayi hai, isko andar aane do".

Short Summary:-
Agar login, authentication, ya SAML connection ki baat ho rahi hai -> SP = Service Provider

()Identity Provider (IdP):- Reception/Security: SP aapko reception (IdP - jaise Okta, Microsoft Entra ID) bhejta hai. 
Receptionist aapse aapka ID card mangti hai, verify karti hai, aur aapko ek "Entry Pass" (Token) de deti hai.

🧠 Difference samajh (important 🔥)
1️⃣ SP Right (Low Level)
👉 Ye actual permission / right hota hai system ka
👉 Example:
Identity read
Role edit
Task run
2️⃣ Capability (High Level)
👉 Ye SP Rights ka group / bundle hota hai
👉 Example:
"Helpdesk Admin" capability
→ multiple SP Rights ka combination
🔁 Relation
👉 Pehle SP Rights hote hain
👉 Fir unko combine karke Capability banayi jaati hai
Id="flow1"
Copy code
SP Rights → Combine → Capability → Assign to User
🟣 Example
👉 SP Rights:
Id="flow2"
Copy code
IdentityView
IdentityEdit
PasswordReset
👉 Capability:
Id="flow3"
Copy code
HelpdeskCapability
👉 Is capability ke andar upar wale sab rights honge
🔴 Important Point
👉 Capability tabhi useful hai jab usme SP Rights ho
👉 Without SP Rights → Capability ka koi kaam nahi
🔥 Final Answer
👉 "Capability SP Rights ka group hoti hai, isliye SP Rights pehle define hote hain, fir unko combine karke capability banayi jaati hai."
